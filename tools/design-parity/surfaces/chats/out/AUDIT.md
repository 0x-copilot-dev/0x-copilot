# Chats destination — design-parity audit

**Baseline:** `tools/design-parity/design-kit/app-v3/copilot-app.jsx:287-330` (`ChatsSurface`) + `copilot.css` + `copilot-data.jsx`
**Live:** `packages/chat-surface/src/destinations/chats/ChatsArchive.tsx` (mounted directly by both hosts — `ChatsDestination.tsx` is a dead forwarder)
**Measured states:** 1 — `default`. That is a property of the design, not a skipped step: the baseline renders three `.rowlist` cards unconditionally with no loading/error/empty branch.
**Raw comparator totals:** 🔴 17 HIGH · 🟠 59 MED · 🟡 64 LOW · ⚪ 10 INFO (`out/report-default.md`) → **15 root causes**.

## Part 1 — UI fidelity, grouped by root cause

Ordered by (HIGH count × anchors affected). Every row cites a file:line I opened.

### 🔴 RC-1 — `StatusPill` never migrated onto the design-system `.ui-badge` chip recipe

**12 HIGH + 12 MED + 12 LOW — the single largest cluster on the surface.**

The design `.chip` (`design-kit/app-v3/copilot.css:575-586`) is a **transparent, mono, lowercase, 25 %-alpha-bordered tag**: `font-family: mono`, `background: transparent`, `1px var(--line2)`, `10.5px`, `gap 5`, `padding 1px 8px`, no tracking, no uppercase; tone variants recolour **text + border only**, at 25 % alpha (`copilot.css:591-605`).

`packages/chat-surface/src/shell/StatusPill.tsx:66-84` (`pillStyle`) diverges on **eight properties at once**:

| Property          | Design                 | Live                                                                                          | Site                   |
| ----------------- | ---------------------- | --------------------------------------------------------------------------------------------- | ---------------------- |
| fontFamily        | mono                   | _unset_ → inherits `--font-sans` (typeface-class change)                                      | `StatusPill.tsx:66-84` |
| backgroundColor   | transparent            | `--color-success-bg` #1a2f23 / `--color-warning-bg` #322615 / `--color-surface-muted` #16161a | `:75`                  |
| borderColor       | `rgba(87,199,133,.25)` | full-opacity `--color-success`                                                                | `:77`                  |
| muted borderColor | `--line2` (.10)        | `--color-border` (.06)                                                                        | `:62`                  |
| fontSize          | 10.5px                 | `--font-size-2xs` = 11.2px                                                                    | `:78`                  |
| fontWeight        | 500                    | 600                                                                                           | `:79`                  |
| letterSpacing     | normal                 | 0.3                                                                                           | `:80`                  |
| textTransform     | none                   | uppercase                                                                                     | `:81`                  |
| padding / gap     | 1px 8px / 5            | 0 8px / 6                                                                                     | `:71,:73`              |

The design-system **already ships this recipe**: `.ui-badge` at `packages/design-system/src/styles.css:555-568`, whose comment literally reads _"design `.chip` — mono, bordered, NO fill"_, with `--success/--warning/--danger/--accent` at `:570-590`. This is exactly the migration PR #221 performed elsewhere in the kit; `StatusPill` was missed. Because `StatusPill.tsx:1-10` declares itself the SSOT for every destination's status chip, fixing it corrects **Chats, Activity, Projects and Connectors at once**.

Two recipe defects to fix _while_ migrating (verified in source): `.ui-badge` itself is `font-weight: var(--font-weight-semibold)` (`styles.css:564`) and `font-size: var(--font-size-2xs)` = 11.2px (`:563`), and there is **no `.ui-badge--muted`** for the archived tone.

### 🔴 RC-2 — the 28×28 row icon lost its tile fill (all 8 rows)

Design `.lrow__ic` (`copilot.css:1617-1625`) = 28×28, radius 7px, `background: var(--panel3)` (#1d1d23). `packages/chat-surface/src/destinations/_shared/Row.tsx:70-79` (`iconSlotStyle`) sets width/height/radius/color but **no `backgroundColor`** → computes `rgba(0,0,0,0)`. The token exists byte-identical and unused: `--color-surface-elevated: #1d1d23` (`packages/design-system/src/styles.css:201`). Same site: `borderRadius` uses `--radius-md` = 8px against a literal 7px (`--radius-sm` is 6px — neither ladder step lands on it; 8px is acceptable).

### 🔴 RC-3 — Chats is full-bleed, so the destination title, subtitle and the only search affordance are all absent

`packages/chat-surface/src/shell/ChatShell.tsx:43-46` declares `FULL_BLEED_DESTINATIONS = {"chats","run"}` (read and confirmed); `:236-237` derives `fullBleed`; `:296`, `:304`, `:322` then render `null` in place of Topbar / ContextPanel / right rail. The design **does** render the Topbar on Chats — `TITLES.chats = ["Chats", "every conversation with the agent"]` (`copilot-app.jsx:599`), styled `.tb-title h1` 13.5px/600 `--disp` `-.01em` + `.sub` 11.5px `--mut2` (`copilot.css:80-82`). The Topbar also carries the ⌘K "Search & commands" button — **the only search entry point on this surface**. Verified as structural, not harness artifact: no host substitutes a title (`apps/frontend/src/app/App.tsx:1054`, `apps/desktop/renderer/DestinationOutlet.tsx:190`). `run` is legitimately full-bleed (a cockpit owning full height); `chats` is a padded, max-width-960 scrolling list page.

### 🔴 RC-4 — the mono model marker is one tone _brighter_ than the sub-line containing it (all 8 rows)

Design: `.lrow__sub` is `--mut2` #64646d (`copilot.css:1643-1648`) and `.mono` sets **font-family only**, no colour (`copilot.css:138-140`) — the model name recedes with its preview. `ChatsArchive.tsx:427` hard-sets `color: var(--color-text-muted)` = `--mut` #98989f, **inverting** the design's read ("preview first, model as a quiet mono tail"). `Row.tsx:108` `subStyle` is already `--color-text-subtle`, so deleting the colour key fixes it.

### 🟠 RC-5 — section heads picked `--font-size-2xs` (11.2px) where the ladder's `--font-size-3xs` (9px) is the design step

Design `.sect-h` = 9.5px mono / `.12em` / uppercase / `--mut2` (`copilot.css:1563-1570`). `packages/chat-surface/src/destinations/_shared/SectionHeader.tsx:40` uses `--font-size-2xs` = 11.2px — **+1.7px, the largest type divergence on the surface**, on all three heads. The proximate cause is documented in the file itself: `SectionHeader.tsx:4` claims _"Mono, ~9.5px (--font-size-2xs)"_, which is **factually wrong** — `--font-size-2xs` is 11.2px (`styles.css:63`, comment "badge, kbd") and the 9px step is `--font-size-3xs` (`styles.css:62`), documented _there_ as "mono micro-labels: settings group header (.set-sec\_\_head) + FTUE fr-\* meta" — i.e. exactly this register, already used for the same element class elsewhere.

### 🟠 RC-6 — `--font-size-2xs` is standing in for the design's 10.5px mono-metadata register

Design `.lrow__time` (`copilot.css:1655-1660`) and `.chip` (`copilot.css:575-586`) are both 10.5px. The main rem ladder has no 10.5px step, but `styles.css:71` carries `--font-size-mono-10: 0.625rem` (10px), documented as _"canonical small-mono pill metadata … deliberately off the main ladder"_ — the intended home. 10px is 0.5px under; the chosen 11.2px is 0.7px over and shares its step with badges/kbd. Sites: `Row.tsx:117` (`metaStyle`), `StatusPill.tsx:78`. For the record, `Row.tsx:107` `subStyle` also uses `--font-size-2xs` against the design's 11px `.lrow__sub` — a 0.2px delta, within tolerance and correctly **not** flagged.

### 🟠 RC-7 — the app-wide body size is 13.6px; the design's is 13px

`styles.css:377` sets `body { font-size: var(--font-size-sm) }` = 0.85rem = 13.6px, and the comment at `styles.css:365-368` is explicit that this is an **approximation** ("the closest token on the existing scale to the design's 13px"). Design `body { font-size:13px }` (`copilot.css:36`). **MED ×5**, every purely-inherited anchor: `page.container`, `header.row`, `list.pinned`, `row.running`, `row.*.ic(.svg)`. Cosmetically small alone, but nothing that inherits will ever match until a 13px step (`0.8125rem`) exists.

### 🟠 RC-8 — font-weight inflation: three components default to semibold where the design is 400/500

(a) `.sect-h` sets no weight → inherits 400 (`copilot.css:1563-1570`); `SectionHeader.tsx:41` uses `--font-weight-semibold`. (b) `.lrow__name` is 500 (`copilot.css:1635-1642`); `Row.tsx:98` uses semibold. (c) `.chip` inherits 500; `StatusPill.tsx:79` hard-codes 600. Both `--font-weight-regular: 400` and `--font-weight-medium: 500` exist (`styles.css:73-74`) — nothing forced the semibold step. This is why the live list reads heavier than the mock even where sizes match.

### 🟠 RC-9 — `.ui-button--sm` sets weight, `.ui-button--primary` does not, so the CTA computes 500

`styles.css:443-449` `.ui-button--sm { font-weight: var(--font-weight-medium) }`; `styles.css:462-466` `.ui-button--primary` sets only background + colour → primary+sm computes **500**. The design is explicitly the other way round: `.cbtn` is 500 (`copilot.css:465-481`), `.cbtn--sm` changes **only** padding + font-size (`copilot.css:567-570`), and `.cbtn--pri` **re-asserts** `font-weight: 600` (`copilot.css:491-496`) so tone wins over size tier. The one CTA on Chats ("New chat") renders lighter than designed. Immaterial and deliberately uncounted: padding `4px 8.8px` vs `4px 9px`, font-size 11.2 vs 11.5 (under the 0.4px threshold). **This is a design-system change touching every primary/sm button in both apps — its own PR.**

### 🟠 RC-10 — row glyphs render 18×18; the design's CSS override forces 15×15

The design passes `<Mark size={18}/>` / `<Icon.chats/>`, but `copilot.css:1627-1630` declares `.lrow__ic svg { width:15px; height:15px }`, which **overrides the attributes**. The extractor confirms it: `row.running.ic.svg` = 15×15 design vs 18×18 live. Live has no such rule and `ChatsArchive.tsx:465` (`BrandMark size={18}`) / `:471` (`Icon size={18}`) stand — a 20 % oversized glyph inside a 28px tile, compounding with RC-2's missing fill to make the icon column read heavy and unframed.

### 🟠 RC-11 — `formatRelativeTime` emits prose, and that widens the time column 2.8×

`packages/chat-surface/src/util/time.ts:33` returns "just now"; `:35-48` defers to `Intl.RelativeTimeFormat(locale,{numeric:"always",style:"narrow"})` → "2 hr. ago" / "1 day ago". The design's vocabulary is compact and coarse — `now / 2h / 3h / 1d / Mon` (`copilot-data.jsx:722-760`), never prose, never absolute. **Measured consequence, not a stylistic quibble:** the flex-none time column is 18.9px design vs 53.8px live, taking that width off the title on every row and destroying the right-edge mono-tick alignment. `time.ts:4-5` declares itself the sole definition, read by Chats, Activity, Projects, Inbox and Library.

### 🟠 RC-12 — page/row box metrics are hardcoded px that miss the design, and the 960 column is centred

Three literals, one habit. (a) Page padding `ChatsArchive.tsx:161` = `24px 28px 96px` vs `.pg` `20px 24px 40px` (`copilot.css:1552-1555`) → content starts 4px lower, 4px narrower per side (912px → 904px usable), 56px dead space under the last card. (b) Row padding `Row.tsx:62` = `10px 12px` vs `.lrow` `11px 14px` (`copilot.css:1582-1596`) → rows 2px shorter, 4px tighter (`list.pinned` 63.25 → 58px). (c) **Centring:** `ChatsArchive.tsx:160` sets `margin:"0 auto"` while `.pg` declares no auto margin anywhere (`copilot.css:1552-1555` is the only `.pg` rule in the sheet) — design left-aligns, live centres (margin `0px` → `0px 110px`). The comment at `ChatsArchive.tsx:157` claims _"centred (FR-4.1)"_, so **decide this deliberately**: if that PRD specifies centring it is a design-side question, not a code bug.

### 🟡 RC-13 — the live-run jade tint sits on a nested wrapper, not the tile (near-zero pixel effect today)

The design puts `color: var(--jade)` on `.lrow__ic` **itself** (`copilot-app.jsx:265-268`); live puts it on a nested `<span style={liveIconStyle}>` (`ChatsArchive.tsx:416-422`, applied `:463-470`) _inside_ Row's icon slot, leaving `[data-testid="row-icon"]` at `--mut`. The comparator ranks it HIGH; **I am downgrading it deliberately** — both sides render the brand mark with a hard-coded sky gradient #9bd4ff→#4593d8 (`copilot-data.jsx:7-32`, and the live `BrandMark` emits the identical gradient — verified in `surfaces/chats/live/default.html`), so `currentColor` tints nothing visible in either. It becomes visible the moment the tile regains its fill (RC-2), so **fix it with RC-2, not before**.

### 🟡 RC-14 — status labels are Title Case, then uppercased

`ChatsArchive.tsx:80-91` `statusLabel` returns "Running"/"Paused"/"Done"/"Archived"; the design reads `running / paused / archived` (`copilot-app.jsx:256-261`). Compounded with `StatusPill.tsx:81 textTransform: uppercase`, the pill renders **"RUNNING"** — two transformations from the design's quiet lowercase.

### 🟡 RC-15 — three design class names survive with zero CSS behind them

`PageLead.tsx:37` emits `pg-lead`, `SectionHeader.tsx:64` emits `sect-h`, `RowList.tsx:56` emits `rowlist` — the design's three load-bearing hooks. Grepping `packages/design-system/src/styles.css` and every `.css` under `packages/` + `apps/` for `.pg-lead` / `.sect-h` / `.rowlist` returns **zero rules**; all styling is inline `CSSProperties` reading `var(--*)`. A reader grepping `.sect-h` to change section-head type finds a class with no rule. It is the single reason this surface needed `data-testid` parity selectors rather than class ones.

### ⚪ Not defects — do not "fix"

- **Rhythm re-expressed as flex gaps.** The design carries rhythm on margins (`.sect-h { margin:22px 0 10px }` `copilot.css:1563-1570`; header wrapper inline `margin-bottom:14px` `copilot-app.jsx:299`; `.pg-lead { margin:-2px 0 18px }` `copilot.css:1556-1562`). Live zeroes margins and uses gaps: 22→24px between sections, 14→10px header-to-list, 18→20px after the lead (`ChatsArchive.tsx:164,167-171,344-348`). **~10 MED/LOW rows in `report-default.md` trace to this one substitution** and should not be triaged individually. Only RC-12's padding values are real.
- **Semantic upgrades.** `<div>`→`<h2>` (`SectionHeader.tsx:69`), `<div>`→`<ul>` (`RowList.tsx:55`), `<button>`→`role="button"` `<div>` (documented at `Row.tsx:12-18` so nested links compose). Reported LOW; accessibility wins.
- **Live-only section count pill** (`ChatsArchive.tsx:369-375`) — an addition, not drift. It does, however, inherit RC-1's filled/uppercase styling and currently reads as loudly as a real status.

## Part 3 — Remediation (architectural, no bandaids)

### The one fix that matters

**Delete the two hand-duplicated `Conversation → ChatArchiveRow` projections and replace them with one seam in `packages/chat-surface`.**

Four of the five surviving feature gaps (`chats.pinned`, `chats.preview`, `chats.model`, and the sub-line collapse behind them) are **the same defect**: `apps/desktop/renderer/destinationBinders.tsx:163-181` reads `conversation.metadata.{preview,model,pinned}` through `metaString`, a path nothing on the server writes — `metadata` is caller-supplied free-form (`services/ai-backend/src/runtime_api/schemas/conversations.py:126,:244`) while `preview`/`model`/`pinned` are **top-level projected fields** (`:246-256`, `packages/api-types/src/index.ts:561-577`). Web reads them correctly (`apps/frontend/src/features/chats/api/chatsApi.ts:150-178`).

The bandaid is a three-line swap in the desktop binder. **The architectural fix is that the projection should never have been written twice.** `packages/chat-surface` already depends on `@0x-copilot/api-types` (`packages/chat-surface/package.json:19`), and `ChatArchiveRow` is already the shared contract (`packages/api-types/src/chats.ts:62-73`). So:

> **New file `packages/chat-surface/src/destinations/chats/projectChatsArchive.ts`** exporting pure `toChatArchiveRow(conversation)`, `chatStatus(conversation)` and `bucketConversations(rows)`, exported from `destinations/chats/index.ts` and the package barrel. Both hosts then supply only _transport_ (`chatsApi.ts:56-70` fetch, `destinationBinders.tsx:201-214` fetch) and call the shared projector.

Evidence this is the right altitude, not speculation: the same two copies already fixed the baseline's pinned/archived double-render bug **identically** (`chatsApi.ts:85-96` ≡ `destinationBinders.tsx:191-197`) _and_ drifted on three fields — duplication that both converges and diverges is duplication the type system cannot police. One projector makes the desktop defect unrepresentable and kills the class.

### Do NOT change

- **Margin→flex-gap rhythm** (~10 report rows). Different mechanism, near-identical result; churning it re-opens a solved layout.
- **Semantic upgrades** `<h2>`/`<ul>`/`role="button"` — accessibility wins, documented at `Row.tsx:12-18`.
- **The four extra states** (loading/error/unavailable/empty). The design has no baseline for them; they exceed it. Judge them on tokens only.
- **RC-13's jade tint, in isolation.** Zero pixel effect until RC-2 lands.
- **The 960px centring** without a product decision — `ChatsArchive.tsx:157` cites FR-4.1.
- **`chats.noRowActions` / `noModal` / `noProjectAttribution` absences.** Parity, and in the last case enforced by the shared row type.
