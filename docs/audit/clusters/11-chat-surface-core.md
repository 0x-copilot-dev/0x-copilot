# Cluster: Chat Surface — Core (shell, canvas, composer, ports)

## Purpose

`@0x-copilot/chat-surface` is the single-source-of-truth interaction layer for the 0xCopilot product. Both deployable app substrates — `apps/frontend` (web, Vite+React) and `apps/desktop` (Electron renderer) — mount the SAME React components from this package and bind data through their own host adapters. This core cluster owns everything except the per-destination surfaces (`src/destinations/*`, audited as `chat-surface-destinations`): the app shell (`ChatShell` + rail/topbar/panels), the Run cockpit canvas (`ThreadCanvas` + swimlanes + chat column + timeline + surface mount), the composer family, the ports contract, the React providers that expose ports, the citation/subagent/approval/workspace presentation families, the surface-renderer adapter registry, routing primitives, the Settings surface, and the package-level barrel/lint/config files (claimed as catch-all).

The package's one hard rule is substrate-agnosticism: no bare `window`/`document`/`fetch`/`localStorage`/`EventSource` — all substrate access flows through **ports** (interfaces the host implements) or through the sanctioned `globalThis.X` member-access convention in explicitly-marked web reference implementations. The ports layer (`src/ports/`) is the load-bearing seam of the whole frontend architecture: `Transport` (HTTP+SSE to the facade), `Router`, `KeyValueStore`, `SecretStorage`, `PresenceSignal`, `PaletteSearchPort`, `SurfaceHost`, plus the Phase-0.5 quartet (`BadgePort`/`NotificationPort`/`FilePickerPort`/`ClipboardPort`). `DeploymentProfile` (`single_user_desktop` | `team`) is supplied via a provider and gates which destinations/settings sections render.

Architecturally the cluster encodes three SSOT registries — `shell/destinations.ts` (slug↔label + profile-gated rail order), `shell/shortcuts.ts` (the 12-chord keyboard table), and `settings/settingsNav.ts` (settings section slugs + profile gate) — and one central event-reduction story: run events (`RuntimeEventEnvelope[]`) stream through the `Transport` port once (`useRunSession`) and are projected client-side into canvas/chat/timeline/approval/subagent views. The audit found that this "one projection" story is partially aspirational in the code (see Findings F1–F3).

**Architecture.** The package is rigorously presentational: components take normalized data + callbacks, or read a port via a hook. Network I/O appears in exactly five places, all via the Transport port: `useRunSession`, `TcChat` (messages fetch), `TcSwimlanes` (own SSE), `TcSurfaceMount` (surface reads), and the two settings data adapters. Registries come in two shapes: module singletons for cross-tree resolution (surfaces, refs) and pure-data SSOTs (destinations, shortcuts, settingsNav). The `globalThis.` member-access convention marks the ~9 deliberate substrate touchpoints (HashRouter, storage, presence, scrollChatToCitation, keyboard hooks) — all with `undefined` guards except `Composer.tsx:660`.

## Data Owned

- No server-side data. Client-side state only:

## Key Flows

1. **Shell mount + profile-gated rail.** Host constructs Transport/Router/KV/Presence port impls → `<ChatShell>` stacks the four providers (shell/ChatShell.tsx:146-167) → rail list resolves once: explicit `destinations` prop → else `destinationsForProfile(useOptionalDeploymentProfile())` → else frozen `SHELL_DESTINATIONS` (ChatShell.tsx:142-144). `chats`/`run` and Settings render full-bleed (topbar/context/right-rail suppressed, ChatShell.tsx:199-217). Rail clicks call host `onNavigate(slug)`; the host owns slug→route mapping.
2. **Run streaming (flow-run-streaming).** `RunDestination` → `useRunSession`: `GET /v1/agent/runs?conversation_id=` picks the live-est run (useRunSession.ts:357-375) → SSE `GET /v1/agent/runs/{id}/stream?after_sequence=N` with `event: runtime_event` frames, dedupe by `sequence_no`, append-only array (useRunSession.ts:221-256) → `ThreadCanvas` projects once via `useEventProjector` (ThreadCanvas.tsx:157) for timeline beads + surface payloads → `projectSubagents`/`projectApprovals` run as memoized pure selectors over the same array for fleet cards, agents-tab counts, in-chat ApprovalCards and the approvals queue (destinations/run/RunDestination.tsx:406-440). Reconnect resumes from the cursor; `retry()` re-subscribes without discarding events. **In parallel** TcSwimlanes opens its own uncursored SSE to the same endpoint (TcSwimlanes.tsx:157-184 — F1) and TcChat fetches persisted messages over HTTP (TcChat.tsx:210-227).
3. **Approval decision round-trip.** `approval_requested` event → `projectApprovals` opens a pending `RunApproval` (approvalProjection.ts:196-241) → TcChat renders `ApprovalCard` (⌘↵/⌘⌫ chords run-scoped per shortcuts SSOT) → host POSTs the decision → optimistic overlay flips the card to a receipt (`overlayApprovalDecisions`, approvalProjection.ts:161-179) → trailing `approval_resolved` frame settles it server-side.
4. **⌘K palette.** `useCommandPaletteHotkey` (document-level listener, `globalThis` convention) flips host-owned open state → `CommandPalette` debounces 150ms and calls `PaletteSearchPort.search()` (~200ms p95 budget, ports/PaletteSearchPort.ts:24-35) → hits grouped by kind; entity hits navigate through the `ItemLink` resolver registry so palette navigation and inline refs share one resolution path.
5. **Surface rendering (tiered).** Events carrying `surface_uri` update per-URI payloads in the projector → `TcSurfaceMount` resolves an adapter via `resolveAdapter(uri)` (version-sorted, broken-skipping — SurfaceRegistry.ts:80+) with a 100ms render budget and tier-3 `GenericStructuredDiff` fallback; time-travel scrubbing re-projects via `projectAt(events, seq)` (TcSurfaceMount.tsx:74). Tier-1 adapters come from `surface-renderers` `registerAll()`; tier-2 from the desktop `Tier2Bridge`.

## Test Posture

Colocated `*.test.ts(x)` per module; 97 test files in this cluster's scope (186 package-wide), run via `vitest` (jsdom, testing-library, setup at src/test/setup.ts). Coverage is genuinely strong where it matters: the ports facade has re-export identity + type round-trip tests (ports/index.test.ts), `destinations.test.ts` freezes the legacy 12-slug contract (FR-2.7), `shortcuts.test.ts` locks the chord table, the projector/approval/subagent reducers have replay/idempotence tests, `ThreadCanvas.test.tsx` asserts the single-projection/mode-no-remount invariants, and the Settings pages each have behavioural tests.

Gaps:

- **Nothing runs any of it in CI.** No workflow executes `npm run test`/`lint`/`typecheck` for `@0x-copilot/chat-surface`; `ci-frontend.yml` does not even trigger on `packages/chat-surface/**` changes (.github/workflows/ci-frontend.yml:4-23), and `ci-desktop.yml` only typechecks what desktop imports (.github/workflows/ci-desktop.yml:63-74). Pre-commit runs prettier only (.pre-commit-config.yaml:43-59). See F0.
- The package's own eslint (the substrate-boundary enforcement) currently **fails** with 11 errors — all bare `document` in test files (useShellShortcuts.test.ts, useCommandPaletteHotkey.test.ts, two destination tests) — proving it is not being run anywhere automatically.
- The dead-in-production projector slices (chat/activity/approvals — F2) are exactly the best-tested code paths, which masks their unreachability: tests exercise `project()` directly, so no test fails when no component consumes a slice.
- No integration test covers the double-SSE behaviour (F1) or the `approval_resolved` decision-value mismatch (F11).

## Health Assessment

**Strengths.** This is one of the most deliberately-architected front-end packages I have audited: the ports contract is small, typed, contract-tested, and genuinely honored (production code has zero bare browser globals; the `globalThis.` convention makes the ~9 sanctioned touchpoints greppable); the three SSOT registries (destinations, shortcuts, settings nav) are real single sources with derived views and freeze tests; both hosts consume through the barrel; boundary types are re-exported from api-types rather than redeclared; the error-envelope parser and notification center give a single user-facing failure path. Test discipline is excellent at the unit level.

**Weaknesses.** (1) The enforcement story is hollow at the CI layer: the package's lint (currently red) and its 186-file test suite run nowhere automatically, and chat-surface changes don't even trigger the web-frontend build in CI. For the declared SSOT layer of the product, this is the highest-leverage fix available. (2) The "ONE projector" run-cockpit architecture has quietly forked: TcSwimlanes opens a second SSE stream, TcChat does its own message fetch, `projectApprovals`/`projectSubagents` are parallel reducers, and most of `eventProjector`'s output (chat, activity, approvals slices) has no production consumer — the invariant comments and the code now disagree, and a latent decision-value bug (F11) hides in the unconsumed path. (3) A layer of acknowledged "deferred convergence" duplication is accumulating (depth models, subagent status helpers, aui-CSS in two places, web legacy Settings vs package SettingsSurface, workspace boundary types) — each is documented, but collectively they erode the package's core promise. (4) Stale VS-Code-fork commentary misdescribes the desktop substrate in the most-read contract files.

**Risks.** A regression in chat-surface can land on main without any CI signal and break both hosts simultaneously (single point of failure by design, unguarded). The dual-SSE cockpit doubles backend stream load per open run and can visually desync lanes vs timeline. The dead projector slices invite a future contributor to wire a consumer onto the buggy approval-state path.

## Findings
