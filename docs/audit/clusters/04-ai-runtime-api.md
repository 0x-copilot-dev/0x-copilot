# Cluster: AI Runtime — HTTP/SSE API layer

## Purpose

This cluster is the ingress surface of the `ai-backend` service: the FastAPI application (`runtime_api/`) that exposes conversations, runs, event replay, SSE streaming, cancel, approvals, usage/budget/retention admin, sharing/forking, drafts, todo-extractions, local (Ollama) model management, and internal service-token routes; plus the service layer behind those routes (`agent_runtime/api/` — coordinators, query services, resolvers, the event producer) and the cross-cutting observability toolkit (`agent_runtime/observability/` — PII-safe logging, OTel bootstrap, token-usage normalization, usage recording, redaction).

Its role in the system: the backend-facade (`services/backend-facade`, `AI_BACKEND_URL=http://127.0.0.1:8000`) proxies all app-facing `/v1/*` agent traffic here; browsers never hit it directly. The API persists a run + queues a `RuntimeRunCommand`; the separate worker process (ai-runtime-worker cluster) claims it and appends `RuntimeEventEnvelope` records; this cluster streams those envelopes back to clients over SSE with a monotonic `sequence_no` reconnect contract (`GET /v1/agent/runs/{run_id}/stream?after_sequence=N`).

Architecturally the cluster is built around five constructor-injected coordinators (`RunCoordinator`, `ApprovalCoordinator`, `ConversationCoordinator`, `ConversationQueryService`, `WorkspaceCoordinator`) wired by `RuntimeApiAppFactory.create_app()` onto `app.state` — no module-level singletons for the core path (the SSE buses are the exception). Persistence is abstracted behind async `Protocol` ports (`agent_runtime/api/ports.py`) implemented by the runtime_adapters (persistence cluster). Everything that talks to the core `backend` service (policies, membership, connector suggestions, projects, routines, todos, email notifications) goes through small HTTP resolver classes with Null fallbacks so the runtime degrades rather than blocks when the trusted lane (`BACKEND_BASE_URL` + `ENTERPRISE_SERVICE_TOKEN`) is unconfigured.

**Catch-all scope**: per the audit assignment this cluster also documents the `services/ai-backend/src/agent_runtime` files not owned by the execution/capabilities/persistence/worker clusters — `settings.py`, `validation.py`, `deployment/`, `pricing/`, `budgets/`, `retention/`, `prompts/` — plus service-level files (`Dockerfile`, `pyproject.toml`, `requirements.txt`, `langgraph.json`, `docker-compose.yml`, `config/`, `scripts/`, `skills/`). Migrations (`services/ai-backend/migrations/`) are treated as owned by ai-runtime-persistence.

Key abstractions: (1) **coordinators over ports** — all writes flow through `PersistencePort`/`EventStorePort`/`RuntimeQueuePort` protocols so in-memory and Postgres adapters are swappable; (2) **event envelope pipeline** — `RuntimeEventProducer` is the single chokepoint that sanitises payloads (`RuntimeEventPresentationProjector.payload_for_event`), projects timeline fields, attaches deterministic presentation, tracks the lifecycle ledger, appends, and notifies the SSE bus; (3) **Null-object resolvers** — every backend dependency has a Null fallback selected by env-driven factories, making the service degrade instead of fail when the trusted lane is absent; (4) **redaction-at-the-log-boundary** — `MetadataRedactor`/`SafeLogDumper`/`SafeAttributeSpanProcessor` enforce that credentials and bodies cannot reach logs or spans.

The notable structural defect is that the layering is circular: `agent_runtime.api` (and even `agent_runtime.capabilities`/`persistence`/`context`) imports `runtime_api.schemas` and `runtime_api.http.errors`, while `runtime_api` imports `agent_runtime.api.*`; `runtime_api.app` also imports `runtime_worker` (see F1).

## Data Owned

This cluster owns no tables directly (tables belong to ai-runtime-persistence), but it is the sole write path for: `runtime_audit_log` rows authored at the API edge (run_created, run_cancel_requested, approval decisions, share lifecycle, connector-scope PATCH audits), runtime event envelopes appended via `RuntimeEventProducer`, and `runtime_run_usage`/`runtime_model_call_usage` rows written through `UsageRecorder` (observability/usage_recorder.py — "single boundary for persisting LLM token usage and cost"). In-process state it owns: the `InMemoryEventBus`/`PostgresEventBus` listener maps, the `InMemoryInboxBus` per-user deques (256-event buffer, process-local), the `_MembershipCache` TTL cache, the MCP discovery cache instance, and the lifecycle ledger per run. Filesystem: `services/ai-backend/skills/` (system skills projected read-only), `config/pricing_overrides.yaml` + `agent_runtime/pricing/litellm_data/model_prices.json` + `pricing/seeds/*.yaml` (pricing catalog inputs). Env contract listed under Public Interface.

## Key Flows

**1. Run create → stream** — facade POSTs `/v1/agent/runs` → `RuntimeApiRoutes.create_run` overrides identity from trusted headers and forbids caller-supplied `runtime_context` (routes.py:227-243) → `RunCoordinator.create_run` prefetches the conversation, merges per-chat connector scopes (run_coordinator.py:103-109), applies workspace default model, then `asyncio.gather`s workspace overrides + user-policy snapshot + connector suggestions (run_coordinator.py:113-125) → `ProviderKeysParser.split` strips BYOK plaintext from the persisted snapshot (run_coordinator.py:129-131) → seals `AgentRuntimeContext`, persists run+message idempotently, writes audit row, emits `RUN_QUEUED`, enqueues `RuntimeRunCommand` (run_coordinator.py:178-217). Client then opens `GET /runs/{id}/stream` → `RuntimeSseAdapter.stream` loops replay-from-`after_sequence` via `ConversationQueryService.replay_events`, waking on `event_bus.wait(run_id, timeout=2.0)` (sse/adapter.py:41-70); worker appends fire Postgres `NOTIFY runtime_events_v1 '<run_id>:<seq>'` → `PostgresEventBus._listen_loop` dispatches to the per-run asyncio.Event (postgres_event_bus.py:168-213).

**2. Approval decision → resume** — `POST /v1/agent/approvals/{id}/decision` → `ApprovalCoordinator.record_approval_decision` validates state, persists the decision, emits `APPROVAL_RESOLVED` via the event producer, enqueues `RuntimeApprovalResolvedCommand` for the worker (never resumes inline — approval_coordinator.py:1-7), fans out notifications via `InboxAndEmailNotificationDispatcher` → `InboxEventBus.publish(user_id=...)` wakes `GET /me/inbox/stream` subscribers (app.py:288-306, routes.py:363-390). Forwarding is depth-capped at 3 and restricted to action/mcp_tool kinds (approval_coordinator.py:66-75); undo honours `UNDO_WINDOW_SECONDS`.

**3. App boot** — `RuntimeApiAppFactory.create_app()` (module-level at app.py:842) → logging + OTel configure → deployment profile `resolve_or_exit()` → `_build_coordinators` (env path: `RuntimeSettings.load`, event-bus selection via `resolved_event_bus_backend`, inbox bus, HTTP membership resolver, BYOK policies resolver over the pooled `BackendHttpPool`, `RuntimeAdapterFactory.from_settings`) → optional services (drafts/shares/forks/feeds) → routers + exception handlers. Lifespan: open/migrate store → build MCP discovery cache → start event bus → optionally start in-process worker; teardown in reverse, closing `BackendHttpPool` last (app.py:112-144).

**4. Usage read path** — `GET /v1/usage/me` reads `runtime_usage_daily_user` rollups; when empty, falls back to a 30-day-capped scan of `runtime_run_usage` synthesized through `UsageQueryService.rollup_user_rows`, plus a live per-call scan for the connector axis (routes.py:689-734, 1066-1096). Org/subagent/purpose/agent variants mirror the rollup-first/cold-start pattern (routes.py:902-946, 1206-1287; http/agent_usage.py:49-104).

**5. Local model pull (desktop/BYOK)** — `GET /v1/local-models/pull?repo=&quant=` (feature-gated) → `LocalModelService.pull_events` streams Ollama `POST /api/pull` NDJSON frames for `hf.co/{repo}:{quant}`, annotating speed/ETA from byte deltas, re-framed as SSE by `LocalModelsApiRouter._sse_frame` (local_models/service.py:75-105, http/local_models_routes.py:63,148).

## Test Posture

Tests live in `services/ai-backend/tests/unit/{runtime_api,agent_runtime/api,agent_runtime/observability}` — 36 + 19 + 22 files respectively out of 319 total, plus cross-cutting suites (`tests/test_run_inbox_routing.py`, `tests/integration/*`). Well-covered: route surface behavior via FastAPI test apps (`test_fastapi_runtime_api.py`, tenant isolation in `test_tenant_isolation_runtime_api.py`, identity strictness `test_get_identity.py`, RBAC `test_rbac.py`), approval forwarding/undo hardening (4 files), usage routes + rollup math, budget routes, retention routes, local models (routes + service), event-bus backends, BYOK split/hydration (`test_provider_keys_byok.py`, `test_run_coordinator_byok.py`), presentation projection, and the observability redaction/deny-key/OTel-safe-attribute suites (which act as compliance tests). Gaps: no test exercises `RuntimeSseAdapter.stream` end-to-end against the `PostgresEventBus` (only the bus units are tested — cross-process wakeup is untested in-repo); no load/soak coverage of the 2 s replay polling; `inbox_producer`/`inbox_fallback` are tested but have no production wiring to verify (see F2); `/readyz` checker wiring is untested because no checkers exist (F8); nothing asserts that `RBAC_MODE=enforce` is actually set anywhere (F6).

## Health Assessment

**Strengths.** The API layer is genuinely well-engineered for its size: async-native ports with in-memory/Postgres parity, a single event-producer chokepoint that makes redaction and lifecycle bookkeeping unforgettable, Null-object degradation for every cross-service dependency, disciplined BYOK plaintext handling (split-before-persist + worker rehydration is carefully reasoned and tested), a real cross-process SSE wakeup design (LISTEN/NOTIFY with reconnect backoff), and an observability stack (deny-key redaction, safe-attribute span processor, single usage-write boundary) that most teams at this stage don't have. Docstrings consistently encode invariants and PR provenance.

**Weaknesses.** The documented layering (agent_runtime = pure domain, runtime_api = HTTP shell) is fiction at the import graph level: domain services import HTTP schemas and error types, `runtime_api` imports `runtime_worker`, and capabilities/persistence modules reach into `runtime_api.schemas` — everything works because it is one deployable, but the module boundaries the READMEs advertise do not hold (F1). `routes.py` has accreted business logic (usage cold-start orchestration, budget CRUD) that belongs in the service layer, with 4+ copies of the same aggregation loop (F4, F5). Two security-relevant defaults are soft: RBAC runs in log-only mode unless an env var nobody sets flips it (F6), and readiness always reports ready (F8). The per-user inbox bus is process-local while the production image runs 4 gunicorn workers plus a separate runtime worker — cross-process inbox delivery is silently lossy today (F3).

**Overall**: healthy core with strong invariants; the risks are concentrated in "shipped scaffolding" (audit-mode RBAC, in-memory inbox bus, unwired readiness) and in the erosion of the internal package boundary.

## Findings

F1. **[boundary-violation | high | high]** Circular layering between `agent_runtime` and `runtime_api`/`runtime_worker` — the "pure domain" layer imports the HTTP layer pervasively. 17 of 28 `agent_runtime/api` modules import `runtime_api.schemas` and/or `runtime_api.http.errors` (e.g. run_coordinator.py:37-38, ports.py:44-61, approval_coordinator.py:38-55); `agent_runtime/capabilities` (citations.py, tool_budget_guard.py, backends/draft_backend.py, render_adapter_generator/capability.py), `agent_runtime/persistence/message_copy.py`, and `agent_runtime/context/memory/subagent_trace.py` also import `runtime_api.schemas`; meanwhile `runtime_api.app` imports `runtime_worker` (app.py:76,593) and `runtime_api/system_skills.py:30` imports `runtime_worker.dependencies`, and `agent_runtime/api/{self_fork,conversation_fork,mcp_discovery_service}.py` import `runtime_worker.audit`. The de-facto domain records (`ConversationRecord`, `RunRecord`, `MessageRecord`, `RuntimeEventEnvelope`) live in `runtime_api.schemas` yet are consumed by every layer. evidence: services/ai-backend/src/agent_runtime/api/ports.py, services/ai-backend/src/runtime_api/app.py, services/ai-backend/src/runtime_api/system_skills.py, services/ai-backend/CLAUDE.md. suggestion: relocate the record/envelope contracts (and `RuntimeApiError`) into `agent_runtime` (e.g. `agent_runtime/contracts` or merging them into `persistence/records`), keep `runtime_api.schemas` as thin re-exports for wire-only request/response types, and move `WorkerAuditEmitter` + `BUILTIN_SKILLS_ROOT` out of `runtime_worker` into `agent_runtime` so all imports point one direction.

F2. **[dead-code | medium | medium]** `agent_runtime/api/inbox_producer.py` (352 LOC — `HttpInboxProducer`, `InboxProducerFactory`) and `agent_runtime/api/inbox_fallback.py` (316 LOC — `InboxFallbackScheduler`, presence/tenant-settings ports) have no production consumers: the only importers are `inbox_fallback.py` itself, a docstring reference in `project_resolver.py:16`, and `tests/test_run_inbox_routing.py` (which uses `NullInboxProducer`). `InboxProducerFactory.default` / `HttpInboxProducer` are constructed nowhere in `src/`. evidence: services/ai-backend/src/agent_runtime/api/inbox_producer.py, services/ai-backend/src/agent_runtime/api/inbox_fallback.py, services/ai-backend/tests/test_run_inbox_routing.py. suggestion: wire the fallback scheduler into the approval flow it was written for, or delete both modules until the backend inbox-items lane ships.

F3. **[risk | high | high]** Per-user inbox SSE cannot deliver across processes in the shipped production topology. `InboxEventBus` is the in-memory singleton only (`InboxEventBus = InMemoryInboxBus`, sse/inbox_bus.py:195; the docstring at lines 11-18 admits a Postgres-backed bus is a "planned follow-up"), while the Dockerfile runs gunicorn with `WEB_CONCURRENCY=4` uvicorn workers (services/ai-backend/Dockerfile:7,34) and production runs a separate runtime-worker process. An approval decision handled by API worker A (or an expiry emitted by the worker process) never wakes a `/v1/agent/me/inbox/stream` subscriber connected to worker B — events land in a different process's deque. evidence: services/ai-backend/src/runtime_api/sse/inbox_bus.py, services/ai-backend/src/runtime_api/http/routes.py:363-390, services/ai-backend/Dockerfile. suggestion: back the inbox channel with LISTEN/NOTIFY (`runtime_inbox_v1`) like the run bus, or set WEB_CONCURRENCY=1 until it lands and document the constraint.

F4. **[risk | high | high]** ai-backend RBAC never enforces by default and its audit-row hook is dead. `RBAC_MODE` defaults to `audit` (log-and-pass — rbac.py:43-46,166-191) and is set in no deploy config (`deploy/`, `docker-compose.dev.yml`, `tools/desktop-runtime/`, env examples all lack it), so `RequireScopes(ADMIN_BUDGETS)`, `RequireAnyScope(AUDIT_READ, "admin:users")`, and `ADMIN_AUDIT_EXPORT` on the SIEM cursor allow any caller who passes the service-token/identity check; additionally the "best-effort audit row" reads `app.state.runtime_audit_appender` which is assigned nowhere, so denies reach structured logs only. Facade-side checks may mitigate in practice, but this service's own gate is advisory. evidence: services/ai-backend/src/runtime_api/rbac.py:32-46,223-243, services/ai-backend/src/runtime_api/http/routes.py:1398-1424, deploy/self-host/docker-compose.prod.yml. suggestion: default `RBAC_MODE` to `enforce` outside development (mirror the fail-closed posture of the service token), and wire `runtime_audit_appender` or delete the dead branch.

F5. **[duplication | medium | high]** Usage aggregation logic exists in 4+ near-identical copies. `UsageApiRoutes._rows_by_day`, `_rows_by_model`, `_connector_rows_from_calls`, `_connector_rows_from_rollup` (routes.py:990-1204) are the same accumulate-into-defaultdict-bucket loop with different keys; `UsageQueryService.rollup_user/connector/subagent/purpose/org_rows` + `_RollupBucket`/`_ConnectorRollupBucket` (agent_runtime/api/usage_service.py:95-296,445-520) implement the same summation for the rollup job; `AgentUsageRoutes._aggregate` (http/agent_usage.py:106-149) re-implements it again with a comment admitting it ("promote `_aggregate` to `UsageQueryService`" — agent_usage.py:36-41). evidence: services/ai-backend/src/runtime_api/http/routes.py, services/ai-backend/src/agent_runtime/api/usage_service.py, services/ai-backend/src/runtime_api/http/agent_usage.py. suggestion: a single generic `bucket_by(key_fn, rows) -> UsageTotals` utility in `UsageQueryService` would delete ~250 LOC and make the token-kind column list one edit site.

F6. **[inconsistency | medium | high]** SSE poll-backstop contract diverges from code, and the bus attribute it depends on is dead. `EventBusBackend.fallback_poll_seconds` (event_bus.py:34; 2.0 on InMemory, 10.0 on Postgres — postgres_event_bus.py:65, whose module docstring promises "the SSE poll fallback becomes a backstop (10s)") is never read by `RuntimeSseAdapter`, which always uses its own `FALLBACK_POLL_SECONDS = 2.0` (sse/adapter.py:24,68). Every connected client therefore re-runs the replay query every 2 s even on the Postgres bus, and the per-backend tuning knob only exists in tests (tests/unit/runtime_api/sse/test_event_bus_backends.py:40-44). Relatedly, the run-stream emits no keepalive frames in follow mode while the inbox adapter sends `: keepalive` every 25 s with a comment claiming it "matches the runtime SSE" (inbox_adapter.py:26-30) — it doesn't; an idle run stream (e.g. paused on approval) emits nothing between events. evidence: services/ai-backend/src/runtime_api/sse/adapter.py, services/ai-backend/src/runtime_api/sse/postgres_event_bus.py, services/ai-backend/src/runtime_api/sse/inbox_adapter.py. suggestion: have the adapter use `event_bus.fallback_poll_seconds` and add the same keepalive comment frame to the run stream.

F7. **[complexity | medium | high]** `runtime_api/http/routes.py` (1,777 LOC) mixes five router factories with real business logic. `UsageApiRoutes` performs persistence-port orchestration, cold-start fallback synthesis, and multi-query joins inline in handlers (routes.py:689-946), and `BudgetApiRoutes` does record construction/merge logic against the port directly (routes.py:1481-1608) — bypassing the coordinator/service pattern every other surface uses (own module + `app.state` service). evidence: services/ai-backend/src/runtime_api/http/routes.py. suggestion: split usage and budget routes into their own modules (matching `retention_routes.py` et al.) and push the fallback/rollup orchestration into `UsageQueryService`; `routes.py` shrinks to the `/v1/agent` router plus internal routes.

F8. **[risk | low | high]** `/readyz` always reports ready: `register_health_routes(app)` is called with no `readiness_checkers` (app.py:247), and `CheckResult`/`Checker` have no other producer — the docstring itself says checkers can be passed "once those wirings exist" (routes/health.py:27-38). Orchestrators pointing readiness probes here will route traffic to an instance whose store/queue/event-bus is down. evidence: services/ai-backend/src/runtime_api/routes/health.py, services/ai-backend/src/runtime_api/app.py:247. suggestion: register store (`SELECT 1` via lifecycle port) and event-bus-listener checkers at boot.

F9. **[inconsistency | low | high]** Python version signals disagree: monorepo rule and pyproject say 3.13 (`requires-python = ">=3.13"`, pyproject.toml:9; CLAUDE.md "Python 3.13 everywhere"), the production image is `python:3.14-slim-bookworm` (Dockerfile:1), and pyright targets `pythonVersion = "3.11"` (pyproject.toml:74). evidence: services/ai-backend/Dockerfile, services/ai-backend/pyproject.toml. suggestion: align all three (likely 3.13 image + pyright 3.13).

F10. **[dead-code | low | high]** Tracked junk in the service root: `Oops.rej` (a leftover patch-reject hunk for `stream_subagents` keys) and `.coverage` (53 KB binary coverage DB) are committed; `env_example` and `.env.example` are duplicate tracked copies of the same template. evidence: services/ai-backend/Oops.rej, services/ai-backend/.coverage, services/ai-backend/env_example, services/ai-backend/.env.example. suggestion: delete the first two, keep one env template, add both patterns to .gitignore.

F11. **[ssot-violation | low | medium]** Accepted-by-design duplication worth tracking: the deployment-profile loader is hand-duplicated across the three Python services (admitted at agent_runtime/deployment/profile.py:4-7 — "duplicated rather than shared because monorepo rules forbid cross-service Python imports"), and the RBAC dependency helpers "mirror the RBAC contract from the backend service (no cross-import)" (runtime_api/rbac.py:3). Both are candidates for `copilot_service_contracts` (constants + pure logic, no service deps) if the constants-only rule is ever relaxed to pure-stdlib logic. evidence: services/ai-backend/src/agent_runtime/deployment/profile.py, services/ai-backend/src/runtime_api/rbac.py. suggestion: none urgent; record as deliberate.

F12. **[risk | low | high]** Dev-open identity fallback rests entirely on env discipline: with `ENTERPRISE_SERVICE_TOKEN` unset and `RUNTIME_ENVIRONMENT != production`, tenant routes accept caller-supplied `org_id`/`user_id` query params (`scoped_identity` fallback, routes.py:476-493; lenient mode auth.py:58-66). Fail-closed depends on deployments actually setting `RUNTIME_ENVIRONMENT=production` — the self-host compose and desktop supervisor do (deploy/self-host/docker-compose.prod.yml:157, tools/desktop-runtime/run-local.mjs:408), but any deployment that forgets both vars is silently multi-tenant-spoofable. evidence: services/ai-backend/src/runtime_api/auth.py, services/ai-backend/src/runtime_api/http/routes.py:476-493. suggestion: log a prominent boot-time warning (or refuse non-loopback binds) when running in identity-open mode.

F13. **[ssot-violation | low | high]** `RequireAnyScope(AUDIT_READ, "admin:users")` hardcodes the `admin:users` scope as a string literal in three route registrations while the `ADMIN_USERS` constant is imported in the same file (routes.py:12,1406,1416,1424). evidence: services/ai-backend/src/runtime_api/http/routes.py. suggestion: use the constant.
