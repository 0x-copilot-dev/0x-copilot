# Cluster: Backend Facade — Product-facing API gateway

## Purpose

`services/backend-facade` is the single HTTP surface (`:8200`) that every app (web frontend, Electron desktop, curl/Postman, CI bots) is allowed to call. It authenticates the caller's bearer, stamps a verified identity, and forwards `/v1/*` requests over HTTP to two upstreams: `services/backend` (`BACKEND_URL`, identity/MCP/skills/product destinations) and `services/ai-backend` (`AI_BACKEND_URL`, conversations/runs/events/approvals/usage/budgets/local-models). It deliberately owns almost no business logic — the exceptions are the cross-service audit-log merge (`audit_routes.py`), the merged `/v1/skills` list, invitation `accept_url` decoration, and the composite bearer-verification machinery itself.

Its second job is being the trust boundary. Inbound requests carry an HMAC-signed bearer (or SCIM token, or routine-webhook HMAC secret, or nothing on the public login ramps); outbound requests carry `x-enterprise-service-token` plus identity headers derived from the verified token, never from caller-supplied query/body values (`src/backend_facade/auth.py:367-379`). Newer route modules additionally do a per-request "session touch" against the backend's sessions table (cached 30s LRU) so revocation and canonical roles propagate (`auth.py:215-301`).

Beyond proxying, the facade hosts three infrastructure concerns: a deployment-profile resolver that fail-closes bad env combinations at boot (`deployment_profile.py`), a redacting observability stack (structured JSON logs, OTEL bootstrap with attribute-denylist span processor, pure-ASGI request-context middleware — `observability/`), and an env-gated static file server for the desktop SIWE wallet page (`wallet_page_routes.py`, fixing the packaged-desktop wallet-404 by serving `wallet.html` + `/assets` same-origin with the SIWE API when `FACADE_WEB_DIST_DIR` is set by the desktop supervisor).

At ~9.8k source LOC across 34 modules it is large for a proxy. The size is almost entirely per-route forwarding boilerplate: ~280 route decorators, each handler 8–15 near-identical lines, plus ~20 private copies of the same upstream-error helper. The service is functionally healthy and well tested (207 tests), but it carries one significant security inconsistency (two-tier auth: the highest-value `/v1/agent/*` surface skips session touch and expiry entirely) and a silently-shadowed duplicate admin route family.

Internal architecture: everything hangs off one `FastAPI` instance; route modules are `register_*(app)` functions with closure-captured handlers (no APIRouter). Registration order is load-bearing twice — webhooks before `/{connector_id}` (`app.py:140-144`) and stream-before-item inside each destination module — and silently harmful once (F2). There are three distinct forwarding kernels: `forward_json` in `app.py:1322-1473` (used by app.py + adapter*registry via a deferred circular import at `adapter_registry_routes.py:19`), the per-module `\_forward*_` helper family (me/workspace/settings/local-models/adapter-review), and fully inline handler bodies (tools/connectors/etc.). Identity attachment is uniform: verified identity rides both query params (`org_id`/`user_id`, dev fallback) and `x-enterprise-_` service headers (production path).

## Data Owned

The facade is intentionally stateless — no database, no tables. It owns:

- **Process state**: per-worker `httpx.AsyncClient` pool (`app.state.http_client`, `http_client.py:61-64`); per-process `_TOUCH_CACHE` TTL-LRU of verified identities keyed on token hash (`auth.py:95-172`) — a _security-relevant_ cache (30s revocation lag by design).
- **Config objects**: `app.state.settings` (`FacadeSettings`), `app.state.deployment` (resolved profile + toggles hash surfaced at `/v1/health`).
- **Env vars owned** (documented in `docs/reference/env-vars.md` and `.env.example`): `FACADE_ENVIRONMENT`, `BACKEND_URL`, `AI_BACKEND_URL`, `OTEL_COLLECTOR_HTTP_URL`, `FACADE_WEB_DIST_DIR`, plus shared secrets `ENTERPRISE_AUTH_SECRET` / `ENTERPRISE_SERVICE_TOKEN`. (`.env.example:2-3` also lists `FACADE_DEV_ORG_ID`/`FACADE_DEV_USER_ID`, which no code reads — stale.)
- **Served files** (read-only): built `wallet.html` + `assets/` from `FACADE_WEB_DIST_DIR` (`wallet_page_routes.py:51-70`).

## Key Flows

1. **Authenticated proxy request (product destination)** — e.g. `GET /v1/tools`: `RequestContextMiddleware` assigns `req_*` id (`observability/request_context.py:55-105`) → handler calls `FacadeAuthenticator.verify_with_touch` (`auth.py:215-301`): HMAC-verify bearer locally → extract `sid` → check 30s touch cache → on miss `POST {backend}/internal/v1/auth/sessions/touch` → canonical identity (roles/scopes/mfa from the sessions row) → forward to `{backend}/v1/tools` with identity query params + `x-enterprise-*` headers (`tool_routes.py:82-105`) → coerce object / map upstream error → access log line with route template + status + duration.
2. **Run streaming (SSE)** — `GET /v1/agent/runs/{id}/stream?after_sequence=N`: static bearer verify (`app.py:1032`) → pooled-client `send(stream=True)` to ai-backend with `timeout=None` (`app.py:1040-1050`) → 4xx drained+raised; otherwise chunks yielded byte-for-byte until client disconnect (`app.py:1060-1069`). Same shape ×8 for tools/connectors/inbox/home/memory/team/library-search/local-models-pull, which add `Last-Event-ID` forwarding and `X-Accel-Buffering: no`.
3. **Login ramp (SIWE example)** — `POST /v1/auth/siwe/nonce` then `/verify`: no bearer; facade adds `ip`/`user_agent` + anonymous service headers (`org` = `-`) and forwards to `{backend}/internal/v1/auth/siwe/*` (`auth_routes.py:332-372`); backend mints the session and returns `bearer_token`; on desktop the same-origin `wallet.html` served by `wallet_page_routes.py` drives this handshake.
4. **Logout/revocation** — `POST /v1/auth/logout`: `verify_with_touch(cache_bypass=True)` → derive `sid` from bearer → `POST /internal/v1/auth/sessions/{sid}/revoke` → invalidate local touch cache for that token hash (`auth_routes.py:93-114`). Takes effect immediately on touch-path routes; **never** on the `app.py` static-auth surface (F1).
5. **Boot** — `create_app()`: `configure_logging()` (JSON formatter, uvicorn.access disabled) → `TelemetryBootstrap.configure()` (OTLP or fail-closed in production; `OTEL_SDK_DISABLED` kill switch for desktop) → `resolve_or_exit()` deployment profile (exit 78 on misconfig) → `HttpClientPool.attach` → 30 `register_*` calls in comment-annotated order (`app.py:101-165`).

## Test Posture

Tests live in `services/backend-facade/tests/` (33 files, 207 test functions, ~8.1k LOC) and run via `make test` (curated subset) or the service `.venv`. Per-route-module proxy tests dominate (`test_tool_routes_proxy.py` 714 LOC, `test_settings_routes_proxy.py`, `test_projects_proxy.py`, …) and pin: identity stamping from the verified bearer, `org_id=hacker` smuggling rejection, upstream error passthrough, SSE headers. Cross-cutting suites are genuinely good: `test_session_binding.py` (561 LOC — REQUIRE_SESSION_BINDING gate, touch cache TTL/invalidation), `test_tenant_isolation_facade.py`, `test_deployment_profile.py`, `test_observability.py`/`test_otel.py` (redaction), `test_forwarder.py` (204-body regression), `test_public_route_contract.py` (OpenAPI presence of core paths), and per-IdP suites (SAML/SIWE/Google/SCIM).

Gaps: (a) nothing pins that revocation/expiry applies to the `/v1/agent/*` surface — because it doesn't (F1); no test asserts `exp` handling anywhere; (b) `test_adapter_review_routes.py` passes against the _shadowing_ registry handlers without noticing the module under test is unreachable (F2) — a route-uniqueness assertion over `app.routes` would have caught it; (c) no test covers the `/v1/audit` merge edge cases (cursor tampering is tested for shape only via 400, per-stream degradation partially); (d) no contract test that `docs/reference/api-surface.md` tracks the OpenAPI surface (F5).

## Health Assessment

**Strengths.** The trust-boundary discipline is genuinely good: caller-supplied identity is dropped everywhere, allowlisted query forwarding is the norm, webhook bodies are forwarded byte-exact for HMAC fidelity (`routines_webhook_routes.py:57`), SCIM/SAML/SIWE ramps keep validation upstream so the wire contract has one owner. The shared HTTP pool with lifespan teardown, the TTL-bucketed touch cache, the pure-ASGI (non-buffering) middleware, and the redacting log/OTEL stack are all thoughtfully engineered with the "why" documented inline. Test volume is proportionate and behavior-pinning.

**Weaknesses.** The service has grown by accretion — one route module per product phase, each copy-pasting the previous module's helpers — leaving three forwarding kernels, ~20 copies of `_raise_for_upstream`, ~25 copies of `settings_for`, and 9 copies of the SSE proxy closure. `app.py` at 1,505 lines mixes bootstrap with the entire agent surface. The accretion has now produced a real defect class: the _oldest_ surface (app.py) still uses the _oldest_ auth (static HMAC, no touch, no expiry), so security posture varies by route family in a way no one route's reader can see. The duplicate admin adapter registration is the same failure mode: two modules own one path, and registration order silently picks the weaker-auth one.

**Risks.** (1) Session revocation/expiry not enforced on conversations/runs/MCP/skills/usage — the highest-value data behind the facade (F1). (2) Docs-as-source-of-truth is violated by a stale `api-surface.md`, which will mislead exactly the compliance-review audience this repo optimizes for (F5). (3) The boilerplate mass means every cross-cutting change (timeout policy, retry, header) is a 30-file diff with omission risk.

## Findings

F1. **[risk | high | high]** Core `/v1/agent/*` + MCP + skills + usage + budgets surface skips session touch and token expiry — revoked/logged-out bearers keep working there. All ~65 `app.py` handlers call the static `FacadeAuthenticator.authenticate_request` (65 hits in `app.py`; zero `verify_with_touch`), which HMAC-verifies and never consults the sessions store; `verify_identity_token` also never checks the `exp` claim (`auth.py:382-412`) even though `CLAIM_EXPIRES_AT` exists in the shared contract (`packages/service-contracts/src/copilot_service_contracts/auth_claims.py:17`) — no `exp`/`expires` reference exists anywhere in the facade src. Logout/revoke (`auth_routes.py:93-114`) therefore does not invalidate access to conversations, runs, run streams, MCP servers, skills, usage, budgets until the HMAC secret rotates. The `auth.py:199-200` docstring claim "in practice every facade route uses the async `verify_with_touch` path" is false. evidence: `services/backend-facade/src/backend_facade/app.py:394,890,1032`, `services/backend-facade/src/backend_facade/auth.py:187-212,382-412`. Suggestion: route all `app.py` handlers through `verify_with_touch` (they already have the app-scoped client), and enforce `exp` in `verify_identity_token`; add a regression test that a revoked session 401s on `/v1/agent/runs`.

F2. **[dead-code | medium | high]** `/v1/admin/adapter_registry/candidates{,/{id},/{id}/decisions}` is registered by BOTH `adapter_registry_routes.py:79-131` and `adapter_review_routes.py:56-93`. `register_adapter_registry_routes` runs first (`app.py:132-133`), so FastAPI's registration-order matcher makes the review module's three handlers unreachable — and the dead pair is the one with the stronger `verify_with_touch` auth, while the live pair uses static auth. `tests/test_adapter_review_routes.py` passes against the wrong handlers because both produce identical upstream calls. evidence: `services/backend-facade/src/backend_facade/app.py:132-133`, `services/backend-facade/src/backend_facade/adapter_registry_routes.py:79-131`, `services/backend-facade/src/backend_facade/adapter_review_routes.py:56-93`. Suggestion: delete the admin trio from `adapter_registry_routes.py` (keep its user-facing routes), keep the touch-authenticated review module, and add a duplicate-path guard test over `app.routes`.

F3. **[duplication | medium | high]** Massive per-module helper copy-paste: `_raise_for_upstream` ×20 files, `_upstream_error_detail` ×12, `settings_for`/`_settings_for` in ~25 modules, `_safe_json` + `_coerce_object_or_raise` ×~8, the SSE pass-through closure ×9 (`app.py:1060-1069`, `tool_routes.py:157-173`, `connector_routes.py:136-149`, `inbox_stream_routes.py:110-126`, `home_routes.py`, `memory_routes.py`, `team_routes.py`, `library_routes.py`, `local_models_routes.py:65-78`), and three parallel forwarding kernels (`app.py:1435-1473`, `me_routes.py:296-424` ×3 near-identical, per-module inline). Several copies self-justify as "avoid circular import" (`me_routes.py:428-429`) — the actual fix is a `backend_facade/proxy.py` leaf module. evidence: `services/backend-facade/src/backend_facade/*.py` (grep `def _raise_for_upstream`). Suggestion: one shared `proxy.py` (forward_json, forward_sse, error mapping, settings accessor) would remove an estimated 2.5–4k LOC and make timeout/retry policy single-source.

F4. **[risk | medium | medium]** `atlas_pk_*` API-key bearers are rejected (401) on the entire `app.py` surface: `authenticate_request` explicitly raises for API-key bearers (`auth.py:207-211`) and only `verify_with_touch` handles them (`auth.py:243-250`), which `app.py` never calls. So personal/workspace API keys minted via `/v1/me/api-keys` (whose stated use case is CI bots, `auth.py:238-240`) cannot create conversations or runs — the primary automation surface. evidence: `services/backend-facade/src/backend_facade/auth.py:207-211,304-352`, `services/backend-facade/src/backend_facade/app.py:886-900`. Suggestion: same fix as F1 (migrate app.py to `verify_with_touch`); add an API-key smoke test against `/v1/agent/runs`.

F5. **[ssot-violation | medium | high]** `docs/reference/api-surface.md` diverges badly from the code while the service CLAUDE.md declares docs "the source of truth": it documents non-existent routes (`/v1/api-keys*` — real path `/v1/me/api-keys*`; `/v1/me/sessions*` — real path `/v1/auth/sessions*`; `/v1/workspace/providers`, `/v1/workspace/privacy`, `/v1/auth/password-reset/*` vs real `/v1/auth/password/reset/*`) and omits entire live families (tools, connectors+webhooks, inbox, home, library, memory, palette, projects, routines, team, todos, agents, adapter-registry, local-models, liveness, wallet page, SIWE, MFA) — ~280 registered decorators vs ~90 documented rows. evidence: `services/backend-facade/docs/reference/api-surface.md:56-66,92-97`, `services/backend-facade/CLAUDE.md:8-10`. Suggestion: regenerate from `app.openapi()` (a 20-line script) and add a CI check.

F6. **[inconsistency | low | high]** Root CLAUDE.md states "Backend's `/internal/v1/*` is consumed only by `ai-backend`", but the facade consumes ~50 backend internal endpoints (auth/sessions/OIDC/SAML/SIWE/MFA/SCIM, me, workspace, policies, liveness, adapter_registry, audit) plus ai-backend's `/internal/v1/skills/system` and `/internal/v1/audit/list`. The boundary rule as documented no longer matches the architecture. evidence: `CLAUDE.md` (root, Request path section), `services/backend-facade/src/backend_facade/auth_routes.py:59`, `services/backend-facade/src/backend_facade/app.py:938`, `services/backend-facade/src/backend_facade/audit_routes.py:126-135`. Suggestion: update the doc to "backend's internal plane is consumed by ai-backend and backend-facade only; never exposed".

F7. **[duplication | low | medium]** MFA _management_ exists on two facade surfaces backed by two backend wrappers: `/v1/auth/mfa/{factors,factors/totp/*,factors/webauthn/register/*,factors/{id}}` (`auth_routes.py:620-803`) and `/v1/me/mfa/*` (`me_routes.py:109-210`). The frontend now manages factors exclusively via `/v1/me/mfa/*` and uses `/v1/auth/mfa/*` only for challenge/verify/recovery (`apps/frontend/src/api/authApi.ts:126-147`, `apps/frontend/src/api/mfaApi.ts:15-46`), leaving the auth_routes factor-management routes without a known consumer. evidence: `services/backend-facade/src/backend_facade/auth_routes.py:620-803`, `services/backend-facade/src/backend_facade/me_routes.py:105-210`. Suggestion: deprecate the `/v1/auth/mfa/factors*` management routes (keep challenge/verify/recovery), or document the split in api-surface.md.

F8. **[dead-code | low | medium]** Legacy `GET /v1/session` (`app.py:224-234`) has no consumer anywhere in apps/packages/tools (frontend uses `/v1/auth/session`); the auth_routes comment says it "stays as-is for one release". Also `.env.example:2-3` documents `FACADE_DEV_ORG_ID`/`FACADE_DEV_USER_ID`, which nothing in src/tests reads (leftover from the removed DEV_AUTH_BYPASS era). evidence: `services/backend-facade/src/backend_facade/app.py:224-234`, `services/backend-facade/src/backend_facade/auth_routes.py:46-48`, `services/backend-facade/.env.example:2-3`. Suggestion: remove both.

F9. **[inconsistency | low | high]** Dockerfile builds on `python:3.14-slim-bookworm` while the repo convention (root CLAUDE.md) is "Python 3.13 everywhere" and `pyproject.toml` pins `requires-python >=3.13` with local `.venv`s on 3.13 — the shipped container runs a different interpreter major than everything tested locally/CI. evidence: `services/backend-facade/Dockerfile:1`, `services/backend-facade/pyproject.toml:9`. Suggestion: pin the image to 3.13 (or bump the convention everywhere deliberately).

F10. **[refactor | medium | high]** `app.py` is a 1,505-line monolith: bootstrap + 85 handlers, each 8–15 lines of identical `authenticate → forward_json(target=…, params=identity.scoped_params(…))` boilerplate; `adapter_registry_routes.py:19` even does a function-local `from backend_facade.app import forward_json` to dodge the resulting circular import. A declarative route table (method, path, target, param policy, body policy) driving one generic handler — or at minimum splitting agent/MCP/usage groups into modules sharing the F3 kernel — would collapse most of the file and make the auth-tier fix (F1) a one-line policy. evidence: `services/backend-facade/src/backend_facade/app.py:236-1315`, `services/backend-facade/src/backend_facade/adapter_registry_routes.py:16-19`. Suggestion: as stated; keep genuinely special handlers (skills merge, stream, telemetry relay, workspace-data confirm_slug) hand-written.

F11. **[inconsistency | low | medium]** `_anonymous_service_headers` sends `ENTERPRISE_SERVICE_TOKEN` with a silent `""` fallback (`auth_routes.py:921`) while the authenticated path fails closed with 503 in production (`auth.py:473-482`) and the SCIM proxy fails closed too (`scim_routes.py:116-125`). In a production misconfig, login ramps would emit unauthenticated internal calls (backend will reject them, but with a confusing error) instead of a clear 503. evidence: `services/backend-facade/src/backend_facade/auth_routes.py:911-924`, `services/backend-facade/src/backend_facade/auth.py:473-482`. Suggestion: reuse `FacadeAuthenticator._service_token()` in `_anonymous_service_headers`.

F12. **[dead-code | low | low]** `routes/health.py` supports injectable `readiness_checkers`, but `create_app` registers none (`app.py:1317`), so `/readyz` always reports ready — readiness is currently equivalent to liveness. evidence: `services/backend-facade/src/backend_facade/routes/health.py:29-52`, `services/backend-facade/src/backend_facade/app.py:1317`. Suggestion: either add backend/ai-backend reachability checkers or drop the unused parameter.
