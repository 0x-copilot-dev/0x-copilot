# Cluster: Shared Packages — contracts, transport, tokens, renderers

## Purpose

Six workspace packages that carry everything the monorepo's hard service-boundary rule allows to cross a deployable boundary: wire contracts, constants, a transport port, a crypto primitive, design tokens, and pluggable surface renderers. They are the "neutral zone" between `apps/*`, `packages/chat-surface`, and the three Python services — no package here owns routes, persistence, or business logic.

**api-types** (`@0x-copilot/api-types`, ~9.6k LOC) is the hand-maintained TypeScript mirror of the public HTTP surface served by `backend-facade`: MCP registry, conversations/runs/events/approvals, settings, auth/MFA, audit, usage, plus one file per desktop-redesign destination (home, inbox, library, todos, team, routines, memory, agents, projects, activity, chats, palette, tools, connectors). The server-side Pydantic models are the source of truth (`README.md:18-25`); this package dual-writes them in TS. **service-contracts** (`copilot_service_contracts`) is the Python constants twin — internal header names, bearer-claim names, RBAC scope catalog, deployment-profile enums, and the tier-2 adapter allowlist JSON — shared into all three services via `PYTHONPATH` in dev and `pip install` in Docker.

**chat-transport** defines the substrate-agnostic `Transport` port (typed request + raw SSE subscribe + session + capabilities) and its three implementations: `WebTransport` (fetch + streaming-fetch SSE, used by the web app and the desktop *main* process), `IpcTransport` (Electron renderer → main RPC, plus the zod-validated IPC channel protocol both preload and main consume), and `MockTransport` (scripted email-draft fixture for the unwired dev shell). `withBearerRefresh` decorates any Transport with a single 401-retry-after-refresh. **audit-chain** (`copilot_audit_chain`) is the HMAC-SHA256 hash-chain signer/verifier both backends use for tamper-evident audit rows — extracted from two in-tree duplicates, with byte-compat fixtures pinning the legacy canonical form.

**design-system** is the v2 "quiet" token SoT (`src/styles.css` `:root`) plus a small set of React primitives (Button/Card/Badge/Switch/Menu/Popover/AppIcon/HarnessRow…). **surface-renderers** holds the four tier-1 SaaS surface adapters (email, Salesforce opportunity, sheet, slide) that register into chat-surface's `SurfaceRegistry`, with an eslint boundary config + lint-negatives harness enforcing the D28 "pure render, no substrate" contract at developer time.

Architecturally the cluster has three distinct patterns. (1) **Dual-written mirrors**: api-types and the services' Pydantic models are maintained by hand in parallel; the newer destination work went file-per-domain with runtime `as const` tuples so unions stay runtime-checkable, while older contracts live in the 3,978-line barrel. (2) **Physical-file SSOT**: the adapter allowlist is one JSON read by both runtimes (TS via relative import, Python via `importlib.resources`), with snapshot canary tests on each side. (3) **Port + implementations**: chat-transport's `Transport` interface keeps chat-surface substrate-free; the three implementations differ only in how bytes move (fetch, Electron IPC, timers). All packages ship source (`"main": "./src/index.ts"`) — consumers compile them; there is no build artifact drift, but every consumer Dockerfile must copy the source trees it transitively needs.

## Key Flows

1. **Web request + stream** — host builds `WebTransport({baseUrl:"", bearerProvider})` (`apps/frontend/src/api/transport.ts:16`) → `request()` adds `authorization` + `x-request-id`, JSON-encodes, parses FastAPI `detail`, throws `UnauthorizedError` on 401 (`WebTransport.ts:52-173`) → SSE: `subscribeServerSentEvents` → `runSseStream` streaming fetch with `accept: text/event-stream`, frame-splits on `\n\n`, dispatches matching `event:` frames; reconnect cursoring (`?after_sequence=N`) is deliberately the caller's job (`sse.ts:28-113`).
2. **Desktop IPC transport** — renderer `IpcTransport.request` → `bridge.ipc.invoke(CHANNELS.transportRequest)` (`IpcTransport.ts:82-89`) → preload allowlist check (`apps/desktop/preload/bridge.ts:16-22`) → main validates with `TransportRequestParamsSchema`, executes over its own `WebTransport` wrapped in `withBearerRefresh` (bearer never crosses IPC, `rpc-protocol.ts:15-18`) → stream events return on `CHANNELS.streamEvent`, matched to subscriptions with a capped microtask buffer for re-mount races (`IpcTransport.ts:148-202`).
3. **Contract drift guard** — a server payload changes → engineer hand-edits the Pydantic model and the api-types mirror in one PR (SPEC.md:27-33) → `test_api_type_contracts.py` regex-extracts the 4 runtime tuples from `index.ts` and set-compares them to Python enums; everything else relies on review discipline plus TS-side tuple lock-in tests.
4. **Tier-2 allowlist consistency** — edit `adapter_allowlist.json` once → TS side re-exports it (`adapterAllowlist.ts:10→20`) for the desktop AST scanner (`apps/desktop/main/adapters/ast-allowlist.ts:3`); Python side loads it at import (`adapter_allowlist.py:30`) for the 6B generator auditor; snapshot canaries on both sides (`adapterAllowlist.test.ts`, ai-backend `test_adapter_allowlist_loader.py`) force acknowledgment of changes.
5. **Audit chain** — service store computes payload dict → `signer.sign(prev_hash, payload)` canonicalizes `{prev_hash, key_version, payload}` with sorted keys/tight separators/typed stringify and HMACs it (`signer.py:187-205, 253-280`) → rows persist `(seq, prev_hash, signature, key_version)` → `verify_chain` walks linkage + recomputes (`signer.py:224-251`); byte-compat with the deleted in-tree implementations is pinned by captured hex fixtures (`services/backend/tests/test_audit_chain_compat.py:1-17`).
6. **Renderer registration (currently dormant)** — desktop bootstrap calls `registerAll()` (`bootstrap.tsx:45`) → each adapter lands in chat-surface's `SurfaceRegistry` (`email/index.ts`) → the only reader is `TcSurfaceMount.resolveAdapter` (`packages/chat-surface/src/thread-canvas/TcSurfaceMount.tsx:240`) inside the thread-canvas subtree, which no app currently mounts (see F2).

## Health Assessment

**Strengths.** This cluster is disciplined for its role: no business logic has leaked in, every package ships a small, documented public surface, and the three genuinely shared cross-language facts (adapter allowlist, internal headers/scopes, audit-chain canonical form) each have a real single source with canary/compat tests. The audit-chain extraction is exemplary — one implementation, env-driven keys, fail-closed production, and byte-compat fixtures in both consumers. chat-transport's port design keeps chat-surface substrate-free and the IPC protocol is zod-validated at the boundary with a channel allowlist enforced in preload. The surface-renderers lint-negatives harness actively proves its own boundary rules still fire.

**Weaknesses.** The cluster's central risk is api-types itself: a ~9.6k-LOC hand-written mirror of Pydantic contracts spread across three services, with automated drift detection for only four enum tuples. The package's own SPEC acknowledges generation is the end-state; meanwhile 54 backend files carry "mirror of api-types" comments as the only linkage. Secondary issues are reachability and hygiene: the entire tier-1 renderer package registers into a registry that nothing mounted reads yet (thread-canvas pending), design-system carries ~123KB of admitted-dead font assets, and a few small hand-maintained duplicates (accent/theme unions, IpcTransport's hand validator beside its own zod schema) will drift silently. The api-types → service-contracts relative-path JSON import works but is an invisible build contract every consumer Dockerfile must know about.

**Risks.** Contract drift on the un-tested 95% of api-types is the highest-likelihood, highest-blast-radius failure (frontend renders wrong/missing fields with no compile-time signal). The LF-only SSE frame parser is spec-incomplete and would fail silently behind a CRLF-normalizing proxy. Docs drift (root CLAUDE.md's "Python + TS" audit-chain claim) can mislead a compliance reviewer into believing client-side chain verification exists.

## Findings

F1. **[ssot-violation | high | high]** api-types is a ~9.6k-LOC hand-maintained dual-write of server Pydantic contracts with drift tests covering only 4 enum tuples — `README.md:6-8` ("currently hand-maintained") and `SPEC.md:25-33` say every contract change must dual-write TS + Pydantic; 54 files in `services/backend/src` alone declare themselves "mirrors of api-types" (e.g. `backend_app/home/routes.py:49`, `backend_app/connectors/routes.py:49`); the only cross-language check is `services/ai-backend/tests/unit/runtime_api/test_api_type_contracts.py:11-37` (regex-parses `index.ts`, set-compares `RUNTIME_API_EVENT_TYPES`/`RUNTIME_EVENT_SOURCES`/`RUNTIME_ACTIVITY_KINDS`/`AGENT_RUN_STATUSES`).
