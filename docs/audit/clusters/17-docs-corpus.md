# Cluster: Docs Corpus & Prior Art

## Purpose

This cluster is the repository's entire non-code knowledge base: 325 tracked files, ~89k lines, spanning root governance docs (`CLAUDE.md`, `README.md`, `SECURITY.md`, `LICENSE`, `CODEOWNERS`), the Cursor rule mirrors (`.cursor/rules/*.mdc`), and the `docs/` tree. There are no tracked files under `.claude/` (`git ls-files .claude` is empty), and `docs/audit/` (this audit's own output) is untracked and excluded per instructions.

The corpus divides cleanly into three strata. **(1) Living engineering contracts** — `CLAUDE.md` + the `.cursor` mirrors, `docs/architecture/`, `docs/decisions/` (2 ADRs), `docs/dev-testing.md`, `docs/security/` (9 runbooks/mappings), `docs/ci-cd/` (assurance spec + supply-chain + runbooks), `docs/deployment/` (profiles, DB tuning/replicas/observability, Google OAuth, SIWE, desktop release), `docs/operations/` (file-store migration). These are current, implementation-tied (most cite exact files, env vars, and workflow names), and are consumed by humans and agents as behavior contracts. **(2) Planning corpora** — three generations of redesign/PRD sets totaling ~70k LOC: `docs/new-design/` (65 files, the "Atlas — 0xCopilot" web-frontend wave plan), `docs/atlas-new-design/` (16 files, the May-2026 "Atlas Workspace" destinations PRDs, status "binding"/"draft, in flight"), and `docs/plan/` (65 files: the original custom-Electron desktop plan, the superseding `desktop-redesign/` PRD set, the current `agent-capabilities/` track last updated 2026-07-18, and the `account-linking/` PRD approved 2026-07-19). Plus `docs/roadmap/` (32 files, the 30-PR bank/gov hardening roadmap A1–A10/B1–B8/C1–C12). **(3) Prior-art audits** — `docs/unused-code/` (61 files, a 2026-05-06 dead-code audit across ai-backend, backend-facade, and frontend, with vulture/knip artifacts), `docs/decomp/` (22 files, a per-file structural inventory of ai-backend's ~27k bespoke LOC), and `docs/use-cases/` (14 files, end-to-end flow traces).

The cluster exists because this repo is developed largely by orchestrated agents: the docs are the mechanism by which decisions, contracts, and supersession chains persist across sessions. Its health therefore matters more than in a typical repo — a stale index or a mislabeled audit folder directly misroutes future agents. This audit found the living-contract stratum in good shape, the planning stratum huge but with a documented supersession chain (only in the newest layer), and the prior-art stratum valuable but with a mislabeled folder, one internal contradiction, and several broken indexes.

A load-bearing fact for downstream consumers: `docs/unused-code/backend/` is **not** an audit of `services/backend` — its own README scopes it to `services/ai-backend` ("Out of scope: services/backend, services/backend-facade") — so `services/backend` (~120k LOC) has never had a dead-code pass at all.

## Test Posture

Docs are exercised only indirectly: pre-commit prettier formats markdown; CI's `tenants-lint`/manifest checks validate artifacts the docs describe, not the docs themselves. There is no link checker — which is precisely the class of rot found here (13/14 broken links in one README, ~20 missing decomp docs, two dangling companion references, a link into a contributor's machine-local Claude-memory path). The unused-code corpus ships two `vulture_whitelist.py` stubs meant to be passed to vulture as a second path; nothing in CI runs them.

## Health Assessment

**Strengths.** The living-contract stratum is genuinely strong: `CLAUDE.md` is dense and accurate; deployment/security/ci-cd docs are implementation-tied with exact env vars, secret names, and file paths; the CI assurance spec's gating/evidence-only/known-gap taxonomy and the trust matrix are exemplary. Supersession is mostly *documented* rather than silent (desktop-app.md names what it supersedes; DEV-GUIDE records the shell supersession; citations 03 carries a shipped-but-replaced banner; roadmap 00-overview honestly describes the pre-W0 state as its context). The binding cross-audit pattern in atlas-new-design is a best-practice for parallel-agent PRD reconciliation, and desktop-redesign/README's DAG + consistency report is the same pattern matured.

**Weaknesses.** (1) The prior-art stratum has a landmine: `docs/unused-code/backend/` audits ai-backend, not backend — any agent sent to "check the backend dead-code audit" gets the wrong service, and `services/backend` has never been audited. (2) Index rot: use-cases README indexes 14 docs that don't exist under those names; decomp's own indexes promise ~20 missing docs plus two nonexistent companion deliverables (`docs/refactor/`, `replacement-analysis.md`) and a nonexistent `data-flow.md` upstream doc. (3) Mirror drift: `.cursor/rules/{service-boundaries,workspace-architecture}.mdc` and `docs/architecture/workspace-topology.md` still describe the apps/mac + apps/windows era. (4) `multi-tenant-deployment.md` lists RLS/KMS-vault/SIEM as "planned" while sibling security runbooks document them as implemented — a stale claim in either direction is a compliance hazard. (5) ~70k LOC of planning docs carry point-in-time "Draft" statuses that no longer reflect shipped reality; only the AC TODO.md and a handful of new-design headers track actual status.

**Risks.** Doc-driven agent workflows amplify every one of these: a stale mirror or wrong-named audit folder is executed, not just read. The highest-leverage fixes are cheap: rename/merge the unused-code trees, regenerate two READMEs' indexes, refresh two architecture files, and add archived/status banners to the superseded planning corpora.

## Findings

F1 **[duplication | high | high]** Two parallel dead-code audits of the same service; folder named "backend" audits ai-backend — `docs/unused-code/backend/README.md` titles itself "Unused code and smells — `services/ai-backend`" and scopes out `services/backend`, duplicating `docs/unused-code/ai-backend/` with a different clustering (12 vs 10 cluster docs, separate vulture artifacts, same date/revision). Consequences: contradictory claims between the passes (see F6), and **no dead-code audit of `services/backend` exists anywhere** despite the folder name implying it. evidence: docs/unused-code/backend/README.md, docs/unused-code/ai-backend/README.md. Suggestion: merge both passes into `docs/unused-code/ai-backend/`, delete or repurpose the `backend/` folder name, and record explicitly that services/backend is unaudited.

F2 **[inconsistency | high | high]** use-cases README index is desynced from its own folder — 13 of the 14 scenario links in `docs/use-cases/README.md` point at files that don't exist (`01-cold-start-first-message.md`, `08-user-cancels-mid-stream.md`, …); the actual files are 13 differently-named docs (`01-new-conversation-simple.md`, `05-cancel-run-mid-stream.md`, …). Only `06-mcp-installed-not-authenticated.md` matches. The README also points at `services/ai-backend/docs/architecture/data-flow.md`, which is not tracked. evidence: docs/use-cases/README.md, git ls-files docs/use-cases. Suggestion: regenerate the index from the directory listing and fix or drop the data-flow link.

F3 **[dead-code | medium | high]** decomp corpus is half-written and its indexes overstate coverage — `docs/decomp/README.md` claims a 92-file inventory and names two companion deliverables that don't exist (`docs/refactor/`, `docs/decomp/replacement-analysis.md`); the 8 `_index.md` files link ~20 per-file docs that were never written (e.g. `capabilities/mcp-loader.md`, `runtime-worker/handlers-approval.md`, `persistence/persistence-bundle.md`). What exists (14 standalone docs) is good, but a consumer following the index hits missing files more often than not. evidence: docs/decomp/README.md, docs/decomp/*/_index.md. Suggestion: mark missing docs as "not yet written" in each index (or write them), and delete the refactor/replacement-analysis references until those artifacts exist.
