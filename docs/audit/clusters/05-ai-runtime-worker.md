# Cluster: AI Runtime — Worker & Streaming Executor

## Purpose

`runtime_worker` is the execution half of the ai-backend split: a separate process (`python -m runtime_worker`, `deploy/self-host/docker-compose.prod.yml:208`) that claims durable commands off the runtime queue and drives them to completion. Three command types exist (`loop.py:206-238`): `RUN_REQUESTED` (build a LangGraph Deep-Agents harness and stream it), `RUN_CANCEL_REQUESTED` (flip run status to CANCELLED), and `APPROVAL_RESOLVED` (resume a paused graph after a human decision). The same `RuntimeWorker` class also runs in-process inside the API when `RUNTIME_START_IN_PROCESS_WORKER=true` (`runtime_api/app.py:791-829`) — which is the desktop production posture (`tools/desktop-runtime/run-local.mjs:410`).

Its core value-add is the **stream projection pipeline**: raw LangGraph v2 stream chunks (messages / updates / custom / values, with subagent namespaces) are normalized into the typed `RuntimeEventEnvelope` records the product consumes — `model_delta`, `tool_call_started/delta/result/completed`, `subagent_started/progress/paused/resumed/completed`, fleet bookends, reasoning summaries, approval/consent cards, `final_response`, `run_completed` — each persisted with a monotonic `sequence_no` so SSE clients can resume. The worker also owns per-run accounting (token usage per LLM call with attribution, budget preflight/charge, audit emission) and the per-run capability wiring (citations, ordinals, tool budgets, MCP discovery, BYOK key re-hydration, desktop file-store/workspace/broker backends).

A second, looser population lives under `jobs/`: periodic maintenance loops (usage rollup, retention sweeper/backfill, pricing refresh via `agent_runtime.pricing`) that the entrypoint actually starts, plus a set of product background jobs (routine scheduler, todo/proposal extractors, todo recurrence materializer, approval expiry sweeper, field-encryption backfill) that — as verified below — no production entrypoint starts at all.

Architecture: the split between the _executor_ (`streaming_executor`) and the _projector_ (`stream_events` orchestrator delegating to message/update/custom processors) is sound and composition-based, with `stream_parts`/`stream_messages` as shared pure parsers. `StreamTextHelper` is a genuine single canonical helper. The two handlers, however, are near-siblings that share logic by ad-hoc reach-ins (`approval.py:311,335,343` call `RuntimeRunHandler._build_tool_display_lookup` / `_is_action_interrupt` / `_extract_final_text` as private cross-class utilities) rather than via a shared base or module. The three desktop wiring modules are a well-executed pattern (env-gate, lazy import, `None` = byte-identical off-path), created explicitly to stop run/resume drift (`file_store_wiring.py:1-18` documents "Bug R1"). Per-run capability state is passed via ContextVar bind/unbind tokens (7 in the run handler's try/finally, `run.py:284-543`), which is functional but fragile — every new capability adds another token pair in two handlers.

## Data Owned

No tables are owned directly — all writes go through `PersistencePort`/`EventStorePort`. The cluster is however the **sole writer** of: `runtime_events` run-stream rows (via `RuntimeEventProducer`, incl. the `sequence_no` ordering invariant), assistant `agent_messages` rows for final responses (`run.py:411-428`, `approval.py:588-602`), `runtime_run_usage` + `runtime_model_call_usage` (`run.py:701-732`), the five `*_daily_usage` rollup tables (`usage_rollup_loop.py:138-267`), worker-side `runtime_audit_log` entries (`audit.py`), approval request/batch rows projected from interrupts (`stream_events.py:371-477`), and retention tombstones/evidence (`jobs/retention_sweeper.py`). In-memory state: per-run tool-call ledgers, subagent linkage maps, reasoning buffers, MCP discovery cache (process-wide). Env-var namespace listed under Public Interface.

## Key Flows

**1. Queued run execution** — API enqueues `RUN_REQUESTED` → `RuntimeWorker.run_forever` → `run_once` claims (`loop.py:118-147`) → `_dispatch` re-parents OTel context (`loop.py:212-238`) → `RuntimeRunHandler.handle` (`run.py:229`): budget preflight (`run.py:574-613`) → status RUNNING → bind ledger/allocator/resolver/budget-guard/discovery/display registries (`run.py:284-355`) → build dependencies (default factory + per-run desktop backends, `run.py:902-965`) → hydrate BYOK keys (`run.py:799-811`) → build harness → `_stream_runtime` under `asyncio.timeout` (`run.py:1293-1325`) → `StreamingExecutor.run` drains chunks: usage attribution (`streaming_executor.py:277-313`), delta coalescing (`delta_coalescer.py`), activity projection (`stream_events.py:146-317`), interrupt flagging without abandoning parallel branches (`streaming_executor.py:342-354`) → final message + `FINAL_RESPONSE` with citations (`run.py:404-455`) → COMPLETED + `run_completed` + audit + usage record + budget charge (`run.py:545-572`).

**2. Interrupt → approval → resume** — a LangGraph interrupt in the stream produces N consent-card payloads (`stream_events.py:632-859`; vendor recognisers `approval_recognisers.py`), one `ApprovalBatch` + N items inserted atomically before per-item events (`stream_events.py:408-477`); run flips to WAITING_FOR_APPROVAL (`run.py:396-403`). Human decision → API enqueues `APPROVAL_RESOLVED` → `RuntimeApprovalHandler.handle` (`approval.py:179`): batch fan-in via `record_item_decision_and_maybe_lock_batch` — exactly one caller wins READY_TO_RESUME (`approval.py:255-269`) → aligned `decisions[]` resume payload (`approval.py:630-694`) → rebind allocator/resolver/display, rebuild dependencies, resume stream (`approval.py:292-334`) → complete or re-pause; batch stamped RESOLVED in `finally` (`approval.py:369-376`).

**3. Tool-call projection & reconciliation** — `messages`-mode chunks accumulate `ToolCallStreamState` keyed by index/call_id (`stream_tools.py:396-466`), emit `TOOL_CALL_STARTED`/`DELTA`, record in `ToolCallLedger`; tool results emit `TOOL_RESULT`+`TOOL_CALL_COMPLETED` with duration, offloading oversized output on the desktop file store (`stream_tools.py:204-279`); `task` tool calls are re-routed to subagent lifecycle events with supervisor-call-id linkage and fleet bookends (`stream_subagents.py:248-404`). On run failure/timeout, unsettled ledger entries get synthetic terminal events so no "Running" card is orphaned (`run.py:1339-1403`).

**4. Worker boot & maintenance loops** — `__main__.amain`: adapter factory (`role="worker"`) → open/migrate → per-process MCP discovery cache + BYOK resolver → `RuntimeWorker` → opt-in loops: usage rollup (default on, `__main__.py:91-100`), retention sweeper (default off), db statement metrics, pricing refresh, one-shot retention backfill (`__main__.py:101-178`).

**5. Cross-turn tool context** — at run start, prior runs' `TOOL_RESULT`/`SUBAGENT_COMPLETED` events are projected into a bounded prompt context with `[[N]]` ordinal cite hints joined from the persistent binding store (`tool_observations.py:79-124,306-346`) and inserted as a SYSTEM message before the last user message (`run.py:1043-1061`).

## Test Posture

Tests live in `services/ai-backend/tests/unit/runtime_worker/` — 43 files, ~15,832 LOC (more test code than source). Well covered: worker claim/retry/dead-letter (`test_runtime_worker.py`), trace propagation, approval batch fan-out/fan-in (`test_approval_batch_fanout.py`, `test_approval_batch_fanin.py`), draft-send resolution, subagent resume, streaming isolation (`test_streaming_executor_isolation.py`), attribution (`test_attribution_builder.py`, `test_per_call_usage.py`), delta coalescer, ledger, tool-observation builder, usage rollup, retention phases, BYOK hydration, desktop wiring gates (`test_capability_tool_wiring.py`, `test_workspace_backend_wiring.py`, `test_browser_provider_wiring.py`). The unwired jobs are thoroughly unit-tested (8 files) — tests are their only consumers. Gaps: no test asserting the approval-resume path records usage/audit/budget (it doesn't — see F2); no cancellation-during-stream test (cancellation mid-run is in fact not implemented, F3); no test pinning the `run_forever` concurrency contract (serial execution is only implicitly exercised).

## Health Assessment

The hot path — claim → stream → project → persist — is genuinely strong: composition over inheritance in the stream family, typed contracts at every boundary, careful ordering invariants (coalescer flushes before non-delta events), a batch-approval primitive that fixed a real N>1 resume crash, and disciplined desktop gating that keeps non-desktop images byte-identical. Test volume matches the criticality. Documentation-in-code is exemplary.

Three structural problems drag it down. First, roughly a quarter of the cluster (~3.2k LOC of `jobs/`) is dormant: built, documented against PRDs, unit-tested, and started by nothing — while the API already ships the read side (todo-extraction accept/reject routes at `runtime_api/app.py:222-223`) over a table no producer fills. Second, the run and approval handlers have drifted: the resume path is missing usage recording, audit emission, budget charging, tool-call reconciliation, and a timeout — precisely the class of drift `FileStoreWorkerWiring` was created to prevent for file wiring. Third, the concurrency story is misleading: the production loop is strictly serial, `max_parallel_runs` machinery is test-only, and cancellation cannot preempt an in-flight run — acceptable for single-user desktop, but a scaling and UX trap for the web deployment. Layering inversions (`agent_runtime` importing `runtime_worker.audit`) and nine copies of env-parsing helpers are smaller but symptomatic.

## Findings

F1. **[dead-code | high | high]** ~3.2k LOC of `jobs/` background workers are wired to no entrypoint — `RoutineSchedulerLoop` (+`routine_pre_fire_gate`), `TodoExtractor`, `ProposalExtractor`, `TodoRecurrenceMaterializerLoop`, `ApprovalExpirySweeper` are instantiated only by tests; `__main__.py` starts only rollup/retention/db-metrics/pricing loops. Meanwhile `runtime_api/app.py:222-223` serves the todo-extraction accept/reject API over a table nothing populates, and approvals never expire because the sweeper never runs. Evidence: `services/ai-backend/src/runtime_worker/__main__.py:91-178`, `services/ai-backend/src/runtime_worker/jobs/routine_scheduler.py:483`, `services/ai-backend/src/runtime_worker/jobs/approval_expiry_sweeper.py:69`, `services/ai-backend/src/runtime_api/app.py:222`. Suggestion: either wire them into `__main__.amain` behind their existing `*_ENABLED` env gates (they were designed for exactly that) or delete/park them out of the shipping image; add a boot-time log of started/skipped loops.

F2. **[inconsistency | high | high]** Approval-resume completion path skips the run path's terminal side effects: no `usage_recorder.record_run/record_call`, no budget `charge_run`, no `emit_run_completed`/`emit_run_failed` audit, no `_reconcile_inflight_tool_calls`, and `_stream_resume` has no `asyncio.timeout` (run path wraps at `handlers/run.py:1302`). Runs that pause for approval and then finish are unmetered, unbilled, unaudited, and can hang the worker forever. Evidence: `services/ai-backend/src/runtime_worker/handlers/approval.py:552-628` vs `services/ai-backend/src/runtime_worker/handlers/run.py:456-572`. Suggestion: extract a shared `RunCompletionFinalizer` (usage + audit + budget + reconciliation + termination) used by both handlers, and wrap resume streaming in the same timeout.

F3. **[risk | high | medium]** Production worker executes strictly serially and cannot preempt: `run_forever` → `run_once` awaits each claim inline (`loop.py:118-170`), so `max_parallel_runs`/`_claim_batch`/semaphore concurrency is exercised only by tests via `run_until_idle`; a queued `RUN_CANCEL_REQUESTED` for the currently-streaming run is not claimed until that run finishes, and neither `StreamingExecutor.run` nor the run handler ever checks the CANCELLING status mid-stream — cancel is cosmetic while a run streams (worst on desktop, which is single-process in-process worker). Evidence: `services/ai-backend/src/runtime_worker/loop.py:118-170`, `services/ai-backend/src/runtime_worker/streaming_executor.py:269-426`, `services/ai-backend/src/agent_runtime/api/run_coordinator.py:230-292`. Suggestion: make `run_forever` dispatch claims as tasks under the existing semaphore, and poll run status (or a cancellation flag) inside the streaming loop to abort promptly.

F4. **[boundary-violation | medium | high]** Layering inversion against the documented "agent_runtime = pure domain" split: `agent_runtime/api/self_fork.py:27` and `agent_runtime/api/conversation_fork.py:25` runtime-import `runtime_worker.audit.WorkerAuditEmitter`; `runtime_api/system_skills.py:30` imports `BUILTIN_SKILLS_ROOT` from `runtime_worker.dependencies`; `agent_runtime/capabilities/tool_budget_guard.py:28` type-imports `ToolCallLedger`. Package dependency graph is circular (worker → agent_runtime/runtime_api → worker). Evidence: paths above; `services/ai-backend/CLAUDE.md` module split. Suggestion: move `WorkerAuditEmitter` (and `BUILTIN_SKILLS_ROOT`, `ToolCallLedger` protocol) into `agent_runtime` (audit is not worker-specific — the API already uses it for forks) and have the worker import it.

F5. **[duplication | medium | high]** Run vs approval handler duplicated flows held in sync by hand: `_workspace_snapshot_emitter` is byte-identical (`run.py:889-900` vs `approval.py:471-482`); `_dependencies_for_run`/`_dependencies_for_resume` re-implement the same augmentation incl. duplicated `DraftBackend` construction (`run.py:902-965` vs `approval.py:484-535`); the DRAFT_UPDATED payload shape is built independently twice (`run.py:981-1008` vs `approval.py:790-809`); final-response persistence duplicated (`run.py:404-455` vs `approval.py:577-628`); and `approval.py:311,335,343` reach into `RuntimeRunHandler` private classmethods. This is the exact drift class that already produced Bug R1 (`file_store_wiring.py:1-18`) and F2. Suggestion: a shared per-run wiring/completion module (same pattern as `FileStoreWorkerWiring`) consumed by both handlers.

F6. **[duplication | medium | high]** Nine near-identical copies of `env_float`/`env_int`/`env_bool` class helpers across the service (5 in this cluster: `usage_rollup_loop.py:34-61`, `jobs/approval_expiry_sweeper.py:39-66`, `jobs/retention_backfill.py:35`, `jobs/retention_sweeper.py:45-64`, `jobs/routine_scheduler.py:444-461`, `jobs/todo_recurrence_materializer.py:389-399`; plus `agent_runtime/observability/db_statement_metrics.py:54`, `agent_runtime/pricing/refresh_loop.py:54`, `runtime_adapters/postgres/runtime_api_store.py:247`, and a tenth variant `_positive_float/_positive_int` in `dependencies.py:182-200`), and seven hand-rolled `start/stop/_run` asyncio periodic-loop lifecycles (`UsageRollupLoop`, `RetentionSweeperLoop`, `RoutineSchedulerLoop`, `TodoRecurrenceMaterializerLoop`, `ApprovalExpirySweeper`, plus pricing/db-metrics loops). Suggestion: one `EnvReader` and one `PeriodicLoop` base in `agent_runtime`; each loop shrinks to a `tick()`.
