# Cluster: AI Runtime — Capabilities (tools/MCP/skills/sandbox)

## Purpose

This is the capability layer of the agent runtime: everything the model can _reach_ — built-in tools, dynamically loaded MCP servers, skills, code execution (three generations of it), the desktop filesystem/browser bridges — plus the cross-cutting per-call middleware that governs every invocation (budget accounting, error-policy classification, retry, citation capture, display-metadata synthesis). It sits inside `agent_runtime/` (documented as "pure domain") and is consumed by the execution factory (`agent_runtime/execution/factory.py`), the runtime worker (`runtime_worker/dependencies.py`, `capability_tool_wiring.py`, `handlers/*`), and the API presentation layer (`agent_runtime/api/presentation.py`, `runtime_api/app.py`).

Architecturally it follows a consistent pattern: Pydantic contracts at every untrusted boundary (`*Card`, `*LoadRequest`, `*LoadResult` with typed error codes and safe messages), Protocol-based ports for providers (MCP client factory, sandbox provider, snapshot store, broker client), and ContextVar-bound per-run singletons (`ToolBudgetGuard`, `CitationLedger`, `CitationResolver`, `ConversationOrdinalAllocator`, `McpDisplayRegistryContext`) so tools deep in the LangChain call stack reach run-scoped services without threading context through signatures. Permission checks are defense-in-depth: the MCP loader re-checks card authorization on every load (`mcp/loader.py:104`), and the dispatcher re-checks it again per call (`mcp/middleware/call_tool.py:77`) so a stale tool reference from an earlier turn cannot bypass per-chat pausing.

The cluster is also where the desktop product's Wave-1 capabilities live: the AC5 brokered `/workspace/` backend (read + approval-gated snapshot-before-write over the Electron capability broker), the AC6 Monty embedded interpreter (pure-compute or HITL-gated external calls), the AC7 remote sandbox (LangSmith provider, gated off by default), and the AC8 desktop browser MCP card. All are env-gated fail-closed seams (`build_*` returning `None`) so non-desktop images are byte-identical without them.

The notable pathology is accumulation: three code-execution subsystems coexist (`tools/code_sandbox.py` in-process AST-gated exec, `interpreter/` Monty, `sandbox/` remote), two citation token systems coexist (`[cN]` ledger tokens for provider-native grounding, `[[N]]` conversation ordinals for tool calls), and roughly 3,100 LOC of the cluster (render_adapter_generator, dynamic tool registry/loader, code_sandbox+adapter, mcp/files, runtime_gate) has no production consumer at all — only tests.

## Key Flows

**1. MCP dynamic load → call.** Model calls `load_mcp_server` → `LoadMcpServerTool.ainvoke` (`mcp/middleware/dynamic_loader.py:33`) → `McpLoader.load_server` (`mcp/loader.py:73`): registry resolve → transport + permission checks (always live, never cached) → `McpDiscoveryCache.get_or_load` keyed `(server, org, user)` → on miss `BackendMcpProvider.create_client` → `connect`/`list_tools`/`list_resources` over backend's `/internal/v1` RPC proxy (`mcp/backend_provider.py:303`) → descriptor validation, budget caps, duplicate/collision checks (`loader.py:226-304`) → display templates synthesised and registered per-run (`backend_provider.py:352-369`). Model then calls `call_mcp_tool` → `CallMcpTool.ainvoke` (`call_tool.py:52`): re-resolve + re-authorize (`:77`) → `client.call_tool` under the loader timeout → `isError` protocol classification (`:145`) → citation projection + `[[N]]` ordinal hint append (`:135-198`) → typed result dict.

**2. Per-call middleware stack.** At run start the worker composes `ToolErrorPolicyRegistry(ToolBudgetGuardedRegistry(CitationCapturingRegistry(WebSearchToolRegistry)))` (`runtime_worker/dependencies.py:129-131`); `deep_agent_builder` then applies the display-args wrap. A single tool invocation therefore traverses: display strip (`display_metadata.py:593-635`) → error policy classify (`tool_error_policy_tool.py:83-99`) → budget admit/warn/reject against the ContextVar guard (`tool_budget_guard.py:219-242`) → citation capture + ordinal hint (`citation_capturing_tool.py:123-181`) → optional `RetryingTool` backoff (`retrying_tool.py:82-99`) → inner tool.

**3. Citation pipeline (two lanes).** Tool lane: `CitationCapturingTool`/`CallMcpTool` allocate a conversation ordinal bound to `tool_call_id` (persisted, conflict-retried — `conversation_ordinals.py:79-161`), append a `[[N]]` hint to the result; `CitationResolver.observe_delta` watches streamed MODEL_DELTAs, regex-matches `[[N]]`, and emits `citation_made` events with `source_tool_call_id` (`citation_resolver.py:104-185`). Provider lane: the Anthropic/Gemini/OpenAI stream adapters call `CitationLedger.cite(SourceRef)` which dedups on `(connector, doc_id)`, bulk-persists via `CitationStorePort`, emits `source_ingested`/`sources_ingested`, and returns `[cN]` base36 tokens (`citations.py:103-229`).

**4. Desktop `/workspace/` read/write.** Worker wiring reads `DESKTOP_BROKER_URL/TOKEN`, fetches grants, builds a slug-deduped mount table (`workspace_backend.py:293-355`) and hands the config to `build_workspace_backend` (`:1009`). Reads translate `ls/read/glob/grep` into broker `/v1/fs/*` calls carrying `grant_id` + root-relative paths only (never host-absolute). Writes require writable grant + per-run `run_capability_context` + durable snapshot store; before mutating existing bytes the backend `put`s the pre-image into the content-addressed object store and emits a typed reference event, failing closed on snapshot failure (`workspace_backend.py:27-49,185-197`); the HITL approval itself lives in the factory's `FilesystemPermission` interrupt, not here.

**5. Gated code execution.** `CapabilityToolWiring` (worker) builds `run_code_mode` only when the Monty gates hold and a file object store exists; external calls from interpreted code route through `HitlPolicyToolInvoker`: budget charge (bridged to the same per-run `ToolBudgetGuard`) → `InterruptApprovalGate` LangGraph interrupt → `LangChainToolDispatcher` to a scope-filtered, model-visible tool (`interpreter/policy_invoker.py:145-199`); `AuthorizedToolResolver` fails unknown aliases closed (`:327-346`). `run_in_sandbox` similarly goes `seam.build_sandbox_backend` → `RemoteExecutionService` (TTL sessions, leak detection) → `PolicyEnforcedSandboxBackend` (command budget) → LangSmith provider (`sandbox/remote_execution_service.py:90-190`).

## Test Posture

Tests live under `services/ai-backend/tests/unit/agent_runtime/capabilities/` (plus `tools/`, `mcp/`, `skills/` siblings) — 59 test files covering this cluster. Coverage is broad and generally deep: dedicated suites for the discovery cache, loader-cache integration, MCP protocol-error classification, dispatcher unwrap, paused-connector enforcement, budget guard/middleware, error-policy tool, retrying tool, citations (ledger, resolver, hint, capturing tool), draft backend, auth gate, desktop workspace + broker client, browser provider, interpreter (adapter/service/policy invoker), and sandbox (transfer, lifecycle, policy backend).

Notable observations: the dead subsystems are _fully tested_ (`test_dynamic_tool_loading.py`, `test_code_sandbox.py`, `test_code_tool_adapter.py`, `test_files.py`, `test_tool_use_runtime_gate.py`, `render_adapter_generator/*`), which keeps them looking alive and spends CI time on unreachable code. Everything here is unit-level with fakes — consistent with the repo's live-smoke memory that unit fakes have previously hidden real-run breakage (e.g. the composed five-layer wrapper stack + LangChain injection behavior is only exercised piecewise, and `BackendMcpClient` against a real backend RPC proxy is not integration-tested inside this service).

## Health Assessment

**Strengths.** The security posture is genuinely good: every untrusted boundary (model input, MCP descriptors, broker responses) is Pydantic-validated with typed error codes and safe messages; permission checks are re-run at load _and_ call time; all desktop capabilities are env-gated fail-closed seams; the Monty external-call path deliberately reuses the same budget guard and interrupt approval as ordinary tools so interpreted code has no back door; the workspace write path is snapshot-before-write fail-closed. The discovery cache design (permission checks never cached, failures never memoized, per-key locks) is thoughtful. Test coverage is extensive.

**Weaknesses.** The cluster is carrying ~3,100 LOC of production-dead code across four subsystems (F1–F4) — about 16% of its mass — all of it tested, which actively disguises the deadness. The cross-cutting middleware is implemented as four near-identical BaseTool wrapper classes plus three near-identical registry decorators with the composition order defined only in worker prose (F8). The layering contract ("agent_runtime is pure domain") is violated by runtime imports of `runtime_api.schemas` in five capability modules, with comments acknowledging the resulting import cycles (F9). The citation feature is spread over five modules with two token grammars and two event families, and the MCP dispatcher re-implements the capturing-tool's hint logic inline (F5). Small duplications (service-auth headers, token estimator, resume-decision parsing) show copy-paste drift risk (F6, F7, F10).

**Risks.** Low correctness risk in the live paths — the code is careful. The main risks are evolutionary: dead subsystems invite accidental re-entrancy or misleading "capability exists" conclusions in compliance reviews; the inverted domain→API dependency will make any future extraction of `agent_runtime` painful; and the five-layer per-tool wrapping means schema-propagation bugs (args_schema, injected tool_call_id) surface only in fully-composed runs, which unit tests don't exercise.

## Findings

F1. **[dead-code | high | high]** `render_adapter_generator/` (~1,443 LOC) has no production consumer — `capability.py`, `models.py`, `templates.py` are imported only by their own tests; the only non-test reference elsewhere is a comment in `runtime_api/schemas/common.py:181`. Its sole contract dependency `copilot_service_contracts.adapter_allowlist` is likewise consumed only here + tests. Evidence: `services/ai-backend/src/agent_runtime/capabilities/render_adapter_generator/`, `services/ai-backend/tests/unit/agent_runtime/capabilities/render_adapter_generator/`, `packages/service-contracts/src/copilot_service_contracts/adapter_allowlist.py`. Suggestion: delete (or move to a spec branch) the package, its tests, and the allowlist contract; if the surface-renderers roadmap still needs it, record that in a spec, not in shipped code.

F2. **[dead-code | high | high]** The dynamic tool-loading subsystem — `tools/registry.py` (`DynamicToolRegistry`), `tools/loader.py` (`ToolLoader`), `tools/builtin/load_tool.py` (`LoadToolSpecTool`), `tools/runtime_gate.py` (`ToolUsePolicyGate`) (~520 LOC) — has zero production consumers; only `tests/unit/agent_runtime/tools/test_dynamic_tool_loading.py` and `test_tool_use_runtime_gate.py` reach it. The worker's real registry is `WebSearchToolRegistry` (`runtime_worker/dependencies.py:41`); `handlers/run.py:784` probes `display_for` via `getattr`, but no production registry in the stack implements it, so even that probe never hits `DynamicToolRegistry.display_for`. Evidence: `services/ai-backend/src/agent_runtime/capabilities/tools/registry.py`, `tools/loader.py`, `tools/builtin/load_tool.py`, `tools/runtime_gate.py`, `src/runtime_worker/dependencies.py:41-131`. Suggestion: either wire it (it mirrors the live MCP dynamic-load design and `ToolCard` is already consumed by `auth_gate`/presentation) or delete registry/loader/load_tool/runtime_gate and keep only `cards.py`/`permissions.py`/`privacy.py`, which are live contracts.

F3. **[dead-code | high | high]** `tools/code_sandbox.py` (452) + `tools/code_tool_adapter.py` (238) — an in-process AST-gated `exec` sandbox — have no consumer outside their own tests, and the cluster now contains two _live_ code-execution subsystems (`interpreter/` Monty, `sandbox/` remote) that supersede it. Three parallel "run code" designs in one package is pure drag. Evidence: `services/ai-backend/src/agent_runtime/capabilities/tools/code_sandbox.py`, `tools/code_tool_adapter.py`, consumers grep (tests only). Suggestion: delete both plus their tests; keep `interpreter/` + `sandbox/` as the two sanctioned execution paths.

F4. **[dead-code | medium | high]** `mcp/files.py` (459 LOC, file-based MCP config layout + secret scanner; `McpFileLayout` has no consumer anywhere) is exported via `mcp/__init__` but never used in production. Evidence: `services/ai-backend/src/agent_runtime/capabilities/mcp/files.py`, `tests/unit/agent_runtime/capabilities/mcp/test_files.py`. F6. **[duplication | low | high]** `BackendMcpServiceAuth.headers` (`mcp/backend_provider.py:427-440`) and `BackendSkillServiceAuth.headers` (`skills/virtual.py:215-228`) are byte-identical `ENTERPRISE_SERVICE_TOKEN` header builders. F7. **[duplication | low | high]** The token estimator is implemented twice with the same constants (4 chars/token, 100k cap): `_Estimator` (`tool_budget_guard.py:162-180`) and `_estimate_input_tokens` (`runtime_worker/capability_tool_wiring.py:72-81`). F8. **[refactor | medium | high]** Overlapping tool-wrapping machinery: four wrapper classes (`ToolBudgetGuardedTool`, `ToolErrorPolicyTool`, `CitationCapturingTool`, `RetryingTool`) each re-implement the "propagate name/description/args_schema, override `_run`/`_arun`" decorator, and three registry decorators (`ToolBudgetGuardedRegistry`, `ToolErrorPolicyRegistry`, `CitationCapturingRegistry`) each re-implement `list_available_tools` + idempotent `_wrap`; `display_metadata.py` adds a fifth, structurally different wrap (StructuredTool copy / delegation). A generic `wrap_registry(inner, *wrappers)` + one `DelegatingTool` base would remove ~150–250 LOC, make ordering declarative, and give one place to fix schema-propagation issues (each wrapper currently handles `args_schema=None`, injected `tool_call_id`, and idempotency slightly differently). Evidence: `tool_budget_guard.py:183-289`, `tool_error_policy_tool.py:54-158`, `citation_capturing_tool.py:106-254`, `retrying_tool.py:43-130`, `middleware/display_metadata.py:516-643`, `runtime_worker/dependencies.py:124-132`. Suggestion: introduce the shared decorator base; keep each concern as a small hook.

F9. **[boundary-violation | medium | high]** `agent_runtime` is documented as "pure domain" (root and service CLAUDE.md), yet capability modules import the API layer at runtime: `from runtime_api.schemas import RuntimeApiEventType, RunRecord` in `tool_budget_guard.py:25`, `citations.py:22`, `backends/draft_backend.py:32`, `render_adapter_generator/capability.py:23`, and a late import in `citation_resolver.py:108` whose comment admits it dodges a circular import; `tool_budget_guard.py:28`/`tool_budget_middleware.py:17` also TYPE_CHECK against `runtime_worker.tool_call_ledger`. The dependency direction is inverted (domain → HTTP layer) even though it stays inside one deployable. Evidence: files above; `services/ai-backend/CLAUDE.md` module-split section. Suggestion: move `RuntimeApiEventType`/`RunRecord` (or a minimal event-envelope protocol) into `agent_runtime.execution.contracts` or `agent_runtime.api`, and have `runtime_api.schemas` re-export.

F10. **[inconsistency | low | high]** Three independent interrupt-resume decision parsers with diverging accepted vocabularies: `AuthMcpTool._resume_result` (`mcp/middleware/auth_mcp.py:116-138` — `{"decision": approved|approve}` or `resume is True`), `InterruptApprovalGate._approved` (`interpreter/policy_invoker.py:259-287` — also accepts `"accept"` and the batch `{"decisions":[{"type": ...}]}` shape), `AskAQuestionTool._resume_result` (`tools/builtin/ask_a_question.py:154-196`). Only one of the three understands the batch shape, so identical UI resume payloads can approve one interrupt kind and silently reject another. Evidence: the three files. Suggestion: one `ResumeDecision.parse(resume)` helper shared by all interrupt-based tools.

F11. **[complexity | low | medium]** Two parallel citation token systems — `CitationLedger` `[c<base36>]` tokens + `source_ingested` events (provider-grounding lane) and `[[N]]` conversation ordinals + `citation_made` events (tool lane) — spread across five modules with two wire grammars and duplicated per-run ContextVar plumbing. Both lanes are live, but the split doubles the concepts every consumer (frontend chips, replay, presentation) must understand. Evidence: `citations.py:1-8,265-285`, `citation_capturing_tool.py:20-46`, `citation_resolver.py:40-45`, `conversation_ordinals.py`. Suggestion: document the two-lane contract in one place and consider converging on the ordinal grammar once provider-grounding chips render through the same path.

F12. **[ssot-violation | low | high]** `CapabilityAuthGate` compares MCP auth state via a string literal — `AUTHENTICATED_AUTH_STATE = "authenticated"` and `str(server.auth_state) == ...` (`auth_gate.py:70-72,114`) — to avoid importing `McpAuthState`, restating the enum value by hand. Evidence: `auth_gate.py`, `mcp/cards.py:62` (`McpAuthState`). Suggestion: import the enum (no real cycle exists — `auth_gate` already imports `tools.cards`) or move `McpAuthState` into a leaf constants module.

F13. **[risk | low | medium]** `CallMcpTool.ainvoke`'s final handler `except (McpClientError, Exception):` (`mcp/middleware/call_tool.py:123`) swallows every unexpected exception into a generic retryable "load failed" result with no logging, unlike the loader's equivalent path which logs at warning with `exc_info` (`mcp/loader.py:212-217`). A systematic bug in a client would be invisible. Evidence: both files. Suggestion: log unexpected exceptions before returning the typed failure; drop the redundant union.
