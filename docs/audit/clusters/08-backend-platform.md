# Cluster: Backend — Platform, Audit, MCP registry & Internal API

## Purpose

This cluster is the platform substrate of `services/backend` (:8100): the app composition roots, the MCP registry + OAuth orchestration + skill registry that CLAUDE.md names as the service's core ownership, the token vault (encryption-at-rest for every secret the product holds), the tamper-evident audit chains and their read/export surfaces, the SIEM export pipeline, DB pooling + versioned migrations, structured observability (JSON logs, OTel, request context), background jobs (library indexer), and the project-liveness aggregator. It also owns the entire `/internal/v1/*` surface consumed by `ai-backend` (MCP cards, client sessions, RPC proxy, skill bundles, suggestible connectors, audit deploy) and by `backend-facade` (audit list, liveness).

**Catch-all enumeration.** As the catch-all auditor for `services/backend`, this cluster claims everything not owned by `backend-identity` (identity/, dev_idp/, provider_keys/, api_keys/, and the auth/me/member/session/mfa/oidc/saml/scim/siwe/password/lockout/invitation/workspace route files) or `backend-product` (the destination packages: home, inbox, todos, routines, connectors, webhooks, projects, library CRUD, memory, agents, tools, palette, team, settings, notifications, privacy, policies, adapter_registry, prompts, and their routes). Concretely claimed here: `backend_app/{app,service,store,contracts,auth,audit_reader,mcp_catalog,mcp_oauth,token_vault,token_vault_metrics,migrations,deployment_profile,desktop_app,__init__}.py`, `routes/{audit_export,audit_list,health,siem,runtime_policies}.py`, the five assigned directories (`db/`, `observability/`, `siem_export/`, `jobs/`, `liveness/`), `migrations/*.sql` + `MANIFEST.lock`, `scripts/*.py`, `Dockerfile`, `pyproject.toml`, `requirements.{in,txt}`, and the tests-layout overview. `contracts.py` also carries identity record types (lines 948+) that backend-identity consumes; it lives here because it is the single wire-contract module for the whole service.

Architecturally the cluster is where the monorepo's hard service boundary is enforced in code: `BackendServiceAuthenticator` verifies `ENTERPRISE_SERVICE_TOKEN` + identity headers on internal calls, the `DeploymentProfile` loader fails the process closed on misconfiguration, and the `TokenVaultFactory` refuses a Fernet vault under KMS-required profiles. Audit rows are HMAC hash-chained via the shared `copilot_audit_chain` package (the sanctioned cross-language shared-primitive precedent).

## Key Flows

1. **MCP OAuth connect (flow-mcp):** frontend → facade → `POST /v1/mcp/servers/{id}/auth/start` (app.py:890) → `McpRegistryService.start_auth` creates PKCE session (service.py:450) → `RemoteMcpOAuthClient.discover` fetches `/.well-known/oauth-protected-resource` + auth-server metadata, dynamically registers a client or applies the pre-registered one (mcp_oauth.py:193–267) → provider redirects to public `GET /v1/mcp/oauth/callback` (app.py:932) → `complete_auth` pops the state-keyed session, exchanges the code, encrypts tokens via `TokenVault`, upserts `mcp_auth_connections` with a cross-tenant `WHERE` guard, appends a chained audit row in one transaction (service.py:511–558, store.py:572–623).
2. **Agent tool loading (flow-run-streaming prereq):** ai-backend worker → `GET /internal/v1/mcp/cards` (RUNTIME_USE scope, header identity; app.py:957) → `POST .../client-session` returns URL + `credential_ref` → `POST .../rpc` proxies JSON-RPC with the decrypted bearer, transparently refreshing expired tokens (service.py:607–759).
3. **Audit write→read (flow-data):** every registry mutation appends to a per-(table,org) HMAC chain under an advisory lock (store.py:658–730) → facade `GET /v1/audit` → `/internal/v1/audit/list` → `AuditReader` fans out over 4 streams, merges by `created_at`, returns opaque cursor (audit_reader.py:162–200). SIEM path: `SiemExportPump.tick()` reads cursors, fetches batches, normalizes, ships via exporter, advances cursor or dead-letters (pump.py:158–235) — **currently unreachable, see F1**.
4. **Boot (flow-desktop-boot / prod):** `resolve_or_exit()` loads the deployment profile or exits 78 (deployment_profile.py:182) → `configure_logging()` + `TelemetryBootstrap.configure()` (production requires OTLP endpoint unless `OTEL_SDK_DISABLED`) → store/service wiring → lifespan starts `SessionSweeper`, closes the shared pool on shutdown (app.py:299–309). Desktop: `DesktopComposer.validate_env` (5 required secrets incl. `AUDIT_HMAC_KEY`) → forces `single_user_desktop` profile → wires every existing Postgres adapter explicitly (desktop_app.py:88–249); migrations applied beforehand by `scripts/migrate.py`.
5. **Migrations (flow-contracts):** `MigrationRunner.apply` reads `migrations/*.sql` via yoyo under a DB lock; `MANIFEST.lock` sha256 (forward + rollback pair) is CI-enforced; production sets `BACKEND_MIGRATIONS_AUTO_APPLY=false` and runs the CLI as a deploy step (db/migrate.py:35–125).

## Health Assessment

**Strengths.** The security-critical primitives are genuinely good: the token vault is a clean adapter hierarchy with envelope-encoded key ids, profile-enforced fail-closed, and metered decrypt caching; the audit chain uses the shared package with per-org advisory-lock serialization and cross-tenant write guards; observability enforces "no free-form messages, no PII" structurally (typed `LogEvent`, metadata redactor, span-attribute denylist) rather than by convention; migrations are checksummed and manifest-locked; the internal API consistently rebinds caller-supplied org/user to verified identity. Boundary discipline is real — cross-service calls are HTTP with service-token auth, constants ride `copilot_service_contracts`.

**Weaknesses.** The compliance story is thinner than the code volume suggests: the SIEM pump — the thing that makes `siem_export_required=True` true — is never started by any process and its SQL doesn't match the audit tables it claims to read; the unified audit list and NDJSON export silently return nothing under the Postgres stores that production and desktop actually use; deploy audit has no persistent adapter anywhere. Per the repo's own compliance rules ("never mark audit logging complete if the adapter is no-op, in-memory only, ... or not exportable to customer SIEM"), audit export must be marked **not implemented** for production-style deployments. Separately, `create_app` has grown into a 1,600-line 60-parameter function that reaches into other services' private attributes, and the same platform code (deployment profile, observability stack, pool metrics, env-parse helpers) is hand-mirrored across two or three services.

**Verdict.** Core registry/vault/chain machinery: healthy and well-tested. Audit egress pipeline: façade-only. Composition root: overdue for decomposition.

## Findings

1. **F1 [risk | high | high] SIEM export pump is unwired and schema-broken** — `SiemExportPump` is instantiated nowhere (no composition root, script, compose file, or test; grep over the whole repo finds only its own module + `__init__` re-export). If it were started, `_fetch_local` would fail: it queries `WHERE ... id > %(after_id)s` against `mcp_audit_events` / `identity_audit_events`, whose PK is `audit_id` (no `id` column), and `EventNormalizer` reads `row["id"]` / `row.get("event_type")` where the real column is `action`. The admin routes' pause/resume/replay rows (`siem_exporter_controls`) are written but never read by the pump, contradicting routes/siem.py's own docstring ("the pump consults at the start of each tick"). Three deployment profiles set `siem_export_required=True`; the control cannot function. evidence: services/backend/src/backend_app/siem_export/pump.py:158–320, siem_export/normalizer.py:20–61, migrations/0001_initial_mcp_skills.sql:55–63, migrations/0004_identity_foundation.sql:120–130, routes/siem.py:1–17, deployment_profile.py:71–93. suggestion: wire the pump into a composition root (or a dedicated worker entrypoint), fix the cursor SQL to `audit_id`/`created_at` keys, make the pump honor `siem_exporter_controls`, and add an integration test that runs one tick against a real schema.
2. **F2 [risk | high | high] Audit list/export return empty under Postgres-backed deployments** — `AuditReader` guards each stream with `hasattr(store, "list_audit_events")` etc.; only the **in-memory** stores implement those readers — `PostgresMcpStore`/`PostgresSkillStore` have `append_audit` but no list method, so under `DATABASE_URL` (production, desktop) the mcp/skill/deploy streams silently vanish from `/internal/v1/audit/list` (not even reported in `degraded_streams`, since the hasattr path returns `()` rather than raising). `/internal/v1/audit/export` reads the `audit_events` **attribute** ("Postgres-backed export is a follow-up") and streams nothing. Deploy audit is in-memory only everywhere, including desktop (explicit accepted gap). evidence: audit_reader.py:220–276, routes/audit_export.py:101–123, store.py:405 (no list_audit_events on PostgresMcpStore), store.py:1315–1322, desktop_app.py:157. suggestion: implement `list_audit_events`/`list_skill_audit_events` on the Postgres stores and a `PostgresDeployAuditStore`; make missing-reader a `degraded_streams` entry, not silence.
3. **F3 [dead-code | medium | high] LibraryIndexerLoop is never started by any composition root** — `LibraryService` enqueues `library_index_jobs` rows (library/service.py:447) and memory rides the same queue, but the only construction sites of `LibraryIndexerLoop` are its own module and `tests/test_library_indexer.py`; no process drains the queue, so library/memory embeddings never materialize in any real deployment (search silently degrades to BM25-only). evidence: jobs/library_indexer.py:262–368; repo-wide grep shows no non-test instantiation. suggestion: start the loop from a composition root (or worker entrypoint) gated on `LIBRARY_INDEXER_ENABLED`, and add a boot test asserting it runs.
4. **F4 [complexity | high | high] `create_app` is a 1,620-line, ~60-parameter mega-function** — all route registration, ~30 store/service wirings, mid-function imports, an inline bridge class (`_RoutineProjectAllowlistBridge`), monkey-patching a sibling service's private attribute (`routines_service._project_allowlist_lookup = ...`), and repeated `noqa: SLF001` reaches into `projects_service._membership_port` and `session_service._auth_secret`. Every new feature edits this one file (2,048 lines), and the ordering constraints (SSE-before-service, webhooks-before-connector-routes) are load-bearing comments. evidence: app.py:409–2030, app.py:679, app.py:1768–1813, app.py:1822. suggestion: split into per-domain wiring modules (`wiring/identity.py`, `wiring/destinations.py`, `wiring/mcp.py`, ...) composed by a thin `create_app`, and promote the private-attr contact points to explicit constructor ports.
5. **F5 [risk | medium | high] `_default_token_vault` swallows the factory's fail-closed errors** — `TokenVaultFactory.create` deliberately raises when a KMS-required profile resolves a local vault (or KMS env is missing), but app.py wraps it in `except Exception: return None`, so a misconfigured production deploy boots **without** OIDC/MFA/provider-keys/webhooks routes instead of refusing to start — degrading a fail-closed control into a silent feature outage. evidence: app.py:312–327, token_vault.py:461–482. suggestion: only catch the specific "no secret configured in dev" case; re-raise under production/managed profiles.
6. **F6 [ssot-violation | medium | high] SIEM exporter env-var name split** — the exporter factory and `docs/security/siem-export.md` use `SIEM_EXPORTER_BACKEND`; the admin routes and `docs/roadmap/00-overview.md` use `SIEM_EXPORT_BACKEND` / `SIEM_EXPORT_BACKEND_NAMES`. An operator following the security doc gets an exporter the admin UI can't see (and vice versa). evidence: siem_export/exporters.py:274, routes/siem.py:250–256, docs/security/siem-export.md:46, docs/roadmap/00-overview.md:253. suggestion: pick one name in `copilot_service_contracts`, alias the other for one release.
7. **F7 [inconsistency | medium | high] Two DB-URL contracts inside one process** — runtime stores resolve `DATABASE_URL` (service.py:806, desktop_app.py:92) while the SIEM admin routes 503 unless `BACKEND_DATABASE_URL` is set (routes/siem.py:259–268) and open raw per-request psycopg connections instead of the shared pool; scripts also use `BACKEND_DATABASE_URL`. A deploy configured only with `DATABASE_URL` has working stores but dead `/v1/siem/*`. suggestion: one env name (or explicit fallback), route SIEM queries through `PostgresConnectionPool`.
