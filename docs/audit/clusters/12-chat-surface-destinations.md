# Cluster: Chat Surface — Destinations & Settings

## Purpose

This cluster is the destination-surface and settings layer of `@0x-copilot/chat-surface`, the monorepo's declared single-source-of-truth interaction layer. Every component here is presentational-plus-ports: it takes normalized `@0x-copilot/api-types` shapes and callbacks as props (or reads the `Transport`/`KeyValueStore` ports via chat-surface providers) and is mounted by two host substrates — `apps/frontend` (web, via `features/*/Route.tsx` binders) and `apps/desktop` (Electron, via `renderer/destinationBinders.tsx`). The package is substrate-agnostic: bare `window`/`fetch`/`localStorage` are eslint-banned; all I/O flows through the Transport port to `backend-facade` `/v1/*` routes.

The `destinations/` tree holds 15 destination directories mapping onto the shell's 16-slug registry (`shell/destinations.ts`, in the sibling chat-surface-core cluster). Post-IA-fold (PR-4.11), only **six** destinations render on the solo rail — `run`, `chats`, `projects`, `activity`, `connectors` (labeled "Tools"), `tools` (labeled "Skills", implemented by `destinations/skills/`) — plus `team`/`members`/`billing` under the `team` profile. The flagship is the **Run cockpit** (`destinations/run/`): `RunDestination` composes `useRunSession` (Transport-port SSE run stream with `?after_sequence` resume), `useRunMode` (KeyValueStore-persisted Studio/Focus + ⌘M), pure event-stream selectors (`approvalProjection`, `projectSubagents`), and `ThreadCanvas` — with a hard "one event projection" invariant (FR-3.3).

The `settings/` tree is the ONE adaptive Settings UI: `SettingsSurface` (216px nav + content router + SaveBar/Toast controller) driven by `settingsNav.ts` (the canonical `SettingsSectionSlug` union + profile gate — team-only sections `workspace`/`members`/`billing`/`audit` render only under `DeploymentProfile === "team"`), plus ten section bodies (Profile, Appearance, Shortcuts, Provider keys/BYOK, Local models, Model & behavior, Privacy, Notifications, App lock, Developer tokens), shared modal/save-bar primitives, and two Transport-backed data ports (`createProviderKeysPort`, `createDeveloperTokensPort`).

The gap between intent and reality is the cluster's defining feature: the IA-fold left nine full destination families (home, inbox, todos, agents, library, memory, routines, team, and the old MCP `tools` catalog) with **no live mount on either host** — roughly 32k source LOC reachable only through the barrel and their own tests — and the web host never adopted the Settings surface, several detail views, or the Run cockpit, keeping parallel implementations in `apps/frontend`.

**Internal architecture.** All destination shells follow one shared pattern: a `SectionResult<T[]> | null` prop drives a 4-state machine (loading skeleton / error + Retry / unavailable / ready-or-empty), rendered with the chat-surface-core shell primitives (`PageHeader`, `CardGrid`, `DocList`, `FilterTabs`, `EmptyState`, `StatusPill`) and inline token-only `CSSProperties` styles (no CSS files). Copy strings are exported constants so hosts/tests assert exact text. Cross-destination navigation goes through the `ItemLink`/`registerItemRefResolver` registry (chat-surface-core `refs/`) rather than direct router calls. The Run cockpit is the only stateful/network-active destination: `useRunSession` owns the append-only `RuntimeEventEnvelope[]` (dedupe by `sequence_no`, resume cursor, tolerant run-list parsing), and everything downstream is pure selectors over that one array. Settings sections communicate dirty-state/save/toast upward through the `SettingsSurfaceController` context rather than owning chrome.

## Key Flows

1. **Run cockpit live session (desktop)** — `DestinationOutlet` mounts `RunBinder` → `RunDestination` (`RunDestination.tsx:156`). `useRunSession` resolves the run list (`GET /v1/agent/runs?conversation_id=`, tolerant parse `useRunSession.ts:387-437`), picks the newest non-terminal run (`pickActiveRunId:357`), and subscribes SSE (`:226`) with `sequence_no` dedupe + resume cursor. Events feed `ThreadCanvas` (single projection), plus two pure selectors: `projectSubagents` (fleet cards + Agents-tab count) and `projectApprovals`→`toApprovalsQueue` (in-chat ApprovalCard + Approvals-tab badge). Scrubbing sets `scrubbedSeq` (`:231-286`) which hides approvals, disables the composer, snaps the surface tab, and shows the "Viewing…" banner; ⌘L/Escape snaps back to live.
2. **Empty→live run start** — no active run ⇒ `RunEmptyState` goal composer (`RunDestination.tsx:562-570`); submit fires host `onStartRun` or the fallback `POST /v1/agent/runs` (`:307-315`), parses the run id tolerantly (`runIdFromCreateResponse:660`), and binds it through the `runId` seam so the live layout mounts in place without a shell remount. `modelReady=false` (no BYOK key/local model) gates submission behind a "Set up your model" CTA; failures surface structured `safe_message`/`code`/`correlation_id` via `parseTransportError` (`:329-343`).
3. **Approval optimistic resolve** — card click → `resolveApproval` (`RunDestination.tsx:450-471`) overlays a local decision (Map) so the card flips to a receipt immediately, then fires `POST /v1/agent/approvals/{id}/decision` best-effort (`.catch(() => {})`); the trailing `approval_resolved` SSE frame reconciles server truth.
4. **Settings render + save (desktop)** — `SettingsMount` renders `SettingsSurface` with `renderSection(slug, controller)`; nav/gating derives from `settingsNavForProfile` (`settingsNav.ts:220-240`) so team-admin sections never render on `single_user_desktop`; a dirty section calls `controller.setDirty` docking the shared SaveBar; provider-key writes flow through `createProviderKeysPort` → `PUT /v1/settings/provider-keys/{provider}` with plaintext appearing exactly once in the PUT body (reads carry `key_hint` only).
5. **List destination render (both hosts)** — host binder fetches over its own transport, wraps rows in `SectionResult`, and passes them to e.g. `ActivityDestination`/`SkillsDestination`/`ConnectorsDestination`; the shell runs the 4-state machine and routes actions back through callbacks (`onOpenRun`, `onSetAccessMode`, `onRunSkill`, …) — navigation stays host-owned.

## Test Posture

Tests are co-located Vitest/`@testing-library/react` files — 88 test files, ~22.6k LOC against ~57.3k source (≈0.4:1). The live Run cockpit is the best-covered area (2,647 test LOC vs 3,253 src: `RunDestination.test.tsx` 939, `useRunSession.test.ts` 414, `RunWorkspaceRail.test.tsx` 452, plus mode/projection/empty/multi tests) — covering scrub gating, optimistic approvals, resume cursors, and error banners. Settings is well covered (3,534 test LOC), including nav-gating (`settingsNav.test.ts`) and per-page behavior. Dead families carry their tests along (≈10.5k test LOC exercising unmounted UI — routines 2,069, projects-dead portions, library 1,606, todos 1,638, inbox 1,358, agents 1,502), which inflates CI cost and false confidence. The weakest ratio is the dead `tools/` family: 6,640 src vs 449 test LOC. Notable gap: no test pins the *host-mount contract* (nothing fails when a built component loses its last consumer — exactly how ~68% of the cluster went dark silently).

## Health Assessment

**Strengths.** The live sixth of this cluster is genuinely good staff-level work: the Run cockpit's single-projection invariant is explicit, enforced by pure selectors, and tested; SSE resume/dedupe is correct; the ports discipline is real (no banned globals; all I/O through Transport/KeyValueStore); `settingsNav.ts` and `ShortcutsPage`-derived-from-`SHELL_SHORTCUTS` are model SSOT modules; profile gating fails safe (unknown profile ⇒ solo, team surfaces can't leak); the provider-keys port is honest about secret handling. Documentation-in-code is exceptional.

**Weaknesses.** The cluster is a museum of a fast-moving redesign: roughly 39k of 57k source LOC (~68%) — nine whole destination families plus large sub-families inside live directories — has no mount on either host, yet remains exported, tested, and documented as if live. The "SSOT interaction layer" claim holds for desktop but not web: web keeps parallel implementations of Settings (1,405-LOC `SettingsScreen` + 17 sections), the run surface (`ChatScreen`/chatModel), connector detail, webhooks wizard, team UI, and the template gallery. Transitional `_*-stub.ts` wire types persist in three places after the canonical api-types landed. Tolerant hand-rolled response parsers in the Run session trade contract safety for lenience.

**Risks.** (1) Refactors "to the SSOT" will be misled by the dead exports — a consumer could adopt an unmaintained dead component believing it's the blessed one (e.g. `ToolsDestination` vs `SkillsDestination` on the same slug). (2) Web/desktop behavior divergence in Settings and Run will grow unnoticed because nothing shares code or tests. (3) The optimistic approval path can display a false "approved" receipt if the POST fails silently and no SSE frame arrives. (4) Hardcoded model catalogs will drift from the runtime's actual models (a known product bug family).
