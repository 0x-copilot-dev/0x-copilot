# Cluster: Backend — Identity, Auth & Secrets

## Purpose

This cluster is the identity plane of the core backend (`services/backend`): everything about who a caller is, how they proved it, what they may do, and how secrets tied to them are stored. It spans the full enterprise-auth alphabet the roadmap labels A1–A10: the identity store (orgs/users/roles/members/audit/login-attempts — `identity/store.py`), session mint/verify/revoke (`identity/sessions.py`), OIDC SSO including the deployment-global Google ramp (`identity/oidc.py`, `identity/google.py`, `identity/jwks.py`), SAML SSO (`identity/saml.py`, `identity/_saml_lib.py`), local passwords with Argon2id (`identity/passwords.py`), MFA TOTP/WebAuthn/recovery (`identity/mfa.py`), SCIM 2.0 provisioning (`identity/scim*.py`), account lockout (`identity/lockout.py`), email-first login + magic links (`identity/login_email_first.py`), Sign-In-With-Ethereum (`identity/siwe.py`), workspace invitations (`identity/invitations.py`), and RBAC route enforcement (`identity/rbac.py`).

Around that core sit the auth-adjacent product surfaces: the dev IdP (`dev_idp/` — YAML personas + HMAC bearer mint, registered only under `BACKEND_ENVIRONMENT=development`), personal/workspace API keys (`api_keys/` — `atlas_pk_*` bearers, peppered HMAC at rest), BYOK provider keys (`provider_keys/` — per-user OpenAI/Anthropic/Google/OpenRouter keys encrypted via TokenVault, hint-only responses), tool-use policies (`policies/`), privacy & data settings (`privacy/`), the Team destination (`team/` — a read projection over the identity tables plus invite/role-change/offboard orchestration and an SSE stream), and the Phase-12 Settings namespaces (`settings/`).

Per the task brief this audit also claims `backend_app/token_vault.py` + `token_vault_metrics.py` (top-level files, not in a package dir): the `TokenVault` interface, the Fernet `LocalTokenVault`, the `ManagedSecretTokenVault`/`AwsKmsTokenVault` KMS adapters, the `kms_v1:` envelope format, and the `TokenVaultFactory` fail-closed policy. Every secret-at-rest in the cluster (MCP OAuth tokens, OIDC client secrets, TOTP seeds, BYOK keys) flows through this one abstraction, and `scripts/rotate_token_vault.py` rotates its ciphertexts.

Architecturally the cluster follows one repeated pattern: a `Protocol` store contract with an `InMemory*` adapter (tests/dev) and a `Postgres*` adapter (production), composed by a service class that owns the crypto/policy detail, with HTTP routes living either in-package (`dev_idp`, `provider_keys`, `team`, `settings`) or in the sibling `backend_app/routes/` layer (sessions, oidc, saml, siwe, passwords, mfa, scim, api_keys, privacy, tool_use_policies, runtime_policies). Composition roots are `backend_app/app.py` (in-memory defaults) and `backend_app/desktop_app.py` (the only Postgres composition root, used by both the packaged desktop app and the self-host prod compose).

Internal architecture: the cluster is remarkably uniform — Protocol store / dual adapters / service / thin routes — and services compose each other explicitly through constructor injection wired in `app.py:417-741` (every store and service is overridable for tests). Cross-cutting flows share single seats deliberately: `provision_personal_org` is the one self-signup path (Google + SIWE), `LockoutService` and `MfaService` are injected into every credential verifier, and all secret-at-rest encryption rides `TokenVault`. The main wart is that HTTP routes for the identity core live in `backend_app/routes/` (another cluster's territory) while newer modules (`provider_keys`, `team`, `settings`, `dev_idp`) keep routes in-package — two conventions for the same thing.

## Key Flows

1. **SIWE wallet login** — `POST /v1/auth/siwe/nonce` mints a rate-limited, address+chain-bound single-use nonce (`siwe.py:521-542`); the client signs the byte-frozen EIP-4361 message (frontend `siweMessage.ts:74-87` or desktop `local-login.ts:68-91`); `POST /v1/auth/siwe/verify` strict-parses (`siwe.py:315-436`), checks statement/chain/domain/time, recovers the signer via EIP-191 (`siwe.py:696-718`), consumes the nonce atomically (`siwe.py:731-762`), links or self-provisions a personal org via `provision_personal_org` (`siwe.py:765-857`, `provisioning.py:67-128`), then mints the same `SessionService` bearer as every other ramp (`siwe.py:860-895`).
2. **Bearer verify + touch** — facade verifies HMAC locally (`backend_facade/auth.py:382-413`), extracts `sid` + `sha256(signature)`, and calls the backend session touch; `SessionService.touch_by_components` confirms the session row is live and stamps `last_seen_at` (`sessions.py:296-307`); `RequireScopes` then evaluates scopes/roles against the forwarded envelope, hard-denying `mfa:pending` (`rbac.py:135-180`).
3. **Dev IdP mint** — dev-only route loads `dev_personas.yaml`, idempotently seeds the org/user rows, and signs a 365-day HMAC bearer directly with `sign_identity_token` (`dev_idp/routes.py:111-164`, `_sign.py:26-38`) — note: no session row, no `sid` claim, unlike the parallel `POST /internal/v1/auth/sessions/dev-mint` path (`routes/sessions.py:131-160`).
4. **BYOK key to runtime** — `PUT /v1/settings/provider-keys/{provider}` validates plausibility (`service.py:181-200`), encrypts via TokenVault, writes row + audit in one transaction (`service.py:81-116`); at run start ai-backend hits `GET /internal/v1/policies/runtime`, which composes tool-use policies + privacy + `decrypted_keys` (`routes/runtime_policies.py`, `provider_keys/service.py:156-167`) so plaintext only ever traverses the service-token lane.
5. **Token vault lifecycle** — `TokenVaultFactory.create` resolves `MCP_TOKEN_VAULT_BACKEND`, rejects `local` under KMS-requiring profiles or `BACKEND_ENVIRONMENT=production` (`token_vault.py:403-482`); KMS ciphertexts carry `kms_v1:<b64 key_id>:<b64 blob>` envelopes so `scripts/rotate_token_vault.py` can re-encrypt to a new CMK without schema coupling.

## Test Posture

Tests live in `services/backend/tests/`: a dedicated `identity/` directory (18 files — store, sessions, oidc, oidc_google, saml + routes, scim + filter + routes, passwords, mfa, lockout, login_email_first, siwe (807 lines), rbac, pkce, tenant_isolation) plus root-level route tests (`test_dev_idp.py`, `test_api_keys_routes.py`, `test_provider_keys_routes.py`, `test_tool_use_policy_routes.py`, `test_privacy_settings_routes.py`, `test_runtime_policies_route.py`, `test_token_vault.py`, `test_rls_session_vars.py`, `test_desktop_app.py`) and `unit/{provider_keys,settings,team}/` suites. Frontend pins the SIWE template with `siweMessage.test.ts`; desktop has `local-login.test.ts`.

Well covered: every login ramp end-to-end at the service layer, RBAC deny/enforce behavior, token vault envelopes + factory fail-closed, tenant isolation, provider-key hint discipline, team offboarding. Gaps: no cross-language test asserts the three SIWE template copies stay byte-identical (each side pins only its own fixture); `avatar_store` and `session_sweeper` are only touched incidentally (`test_desktop_app.py`, `test_sessions.py`); Postgres adapters are mostly exercised through in-memory mirrors (integration dir is small — 5 files); nothing tests that `desktop_app` wires _every_ store that has a Postgres adapter, which is exactly the invariant that rotted (see F2).

## Health Assessment

**Strengths.** This is some of the most disciplined code in the repo: uniform Protocol/adapter/service composition, constructor injection everywhere, deliberate single seats for crypto (TokenVault, `_BearerCodec`, `ApiKeyHasher`), a genuinely strict EIP-4361 parser, constant-time comparisons in the right places, append-only audit rows written inside the same transaction as the primary write, and honest docstrings that state the trust model. Self-signup provisioning was correctly extracted (`provisioning.py`) instead of duplicated between Google and SIWE. Test coverage is broad and behavior-pinning.

**Weaknesses / risks.** The two biggest problems are composition-level, not code-level: (1) RBAC ships default-permissive (`RBAC_MODE=audit` passes every deny through) and _no_ deployment artifact in the repo ever sets `enforce`, so the entire `RequireScopes` layer is advisory in every real deployment; (2) the only Postgres composition root (`desktop_app.py`) forgot to wire `PostgresSiweStore`, so wallet identities are in-memory in exactly the deployments that ship wallet login — the migration-0035 tables are dead. Beyond that: the SIWE wire template is hand-mirrored in three languages with docs claiming two; the HMAC bearer codec exists three times; there are two parallel dev-mint paths with different revocability; and the identity package's route layer is split between two conventions. None of the per-module code is over-abstracted; the cluster's debt is concentrated in wiring and cross-copy contracts.

## Findings

F1. **[risk | high | high]** RBAC is default-permissive and nothing ever enables enforcement — `RbacMode.current()` defaults to `audit` (denies are logged, then passed through: `identity/rbac.py:42-50,155-160,175-180`), and a repo-wide search finds no `RBAC_MODE=enforce` in `deploy/self-host/`, `docker-compose.dev.yml`, `Makefile`, `tools/desktop-runtime/`, or the desktop supervisor; `deployment_profile.py` has no RBAC toggle either. Every scope/role annotation in the backend (and the mirrored ai-backend layer) is currently advisory in every shipped configuration; only the `mfa:pending` deny and facade-level authentication actually block. Evidence: services/backend/src/backend_app/identity/rbac.py; deploy/self-host/docker-compose.prod.yml. Suggestion: drive RBAC mode from the deployment profile (enforce for all production-class profiles), or flip the default to `enforce` with an explicit opt-out, and add a boot-time log/health field exposing the active mode.

F2. **[dead-code | high | high]** `PostgresSiweStore` has zero production consumers — `desktop_app.py` (the only Postgres composition root, used by the packaged desktop AND `deploy/self-host/docker-compose.prod.yml:108`) never imports or wires it (no `siwe` match in the file), so `create_app` falls back to `InMemorySiweStore` (`app.py` SIWE block) while migration `0035_siwe.sql` creates tables nothing writes. Consequence: wallet→user links vanish on every backend restart; with `allow_self_signup` on, a returning wallet is re-provisioned into a fresh personal org each restart (org/user pollution + the user "loses" their workspace); with self-signup off, wallet login breaks outright after restart. Evidence: services/backend/src/backend_app/desktop_app.py:175-235; services/backend/src/backend_app/identity/siwe_store.py:145; services/backend/migrations/0035_siwe.sql. Suggestion: wire `siwe_store=PostgresSiweStore(pool)` in `build_create_app_kwargs` and add a test asserting desktop_app passes a Postgres adapter for every store that has one.

F3. **[ssot-violation | medium | high]** The SIWE EIP-4361 template is hand-maintained in **three** places — backend (`identity/siwe.py:290-302`), frontend (`apps/frontend/src/features/auth/siweMessage.ts:30-41`), desktop (`apps/desktop/main/auth/local-login.ts:78-91`) — while the root `CLAUDE.md` documents only two ("duplicated byte-identically in siweMessage.ts and siwe.py"); `CHAIN_NAMES` is additionally mirrored in the frontend LoginScreen (`siwe.py:88-96`). The copies are byte-identical today (verified), but no cross-language fixture test pins all three together. Evidence: the three files above. Suggestion: one shared golden-message fixture consumed by all three test suites (backend already parses strictly, so a single committed fixture file each suite asserts against is enough), and fix the CLAUDE.md count.
