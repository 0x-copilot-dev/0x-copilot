# Cluster: Frontend Web App

## Purpose

`apps/frontend` is the Vite + React 19 web client for 0xCopilot — the browser surface that end users (and the self-host web deployment) interact with. Its single allowed upstream is `backend-facade` (`/v1/*`), reached through the Vite dev proxy (`vite.config.ts:28`) or nginx ingress in production; it never calls `backend:8100` or `ai-backend:8000` directly. The app boots through an `AuthProvider` state machine (`initial → loading → anonymous | mfa_pending | workspace_pick | authenticated`), then mounts the shared `ChatShell` from `@0x-copilot/chat-surface` and dispatches between destinations, settings, share, admin, and project-template screens (`src/app/App.tsx`).

Post-IA-fold (PR-4.11) the web shell renders the six-destination `single_user_desktop` solo rail (Run / Chats / Projects / Activity / Tools / Skills) by default, with a nine-destination team rail behind `VITE_DEPLOYMENT_PROFILE=team`. The flagship Run destination is the working conversation surface, `ChatScreen` — a large host-owned cockpit that renders streaming agent runs by projecting typed `RuntimeEventEnvelope`s through pure reducers in `features/chat/chatModel/` and subscribing to the facade's SSE stream with a `fetch`-based reader (the browser `EventSource` cannot carry an Authorization header).

Architecturally the app is a **host** for the `@0x-copilot/chat-surface` SSOT interaction layer: destination routes are thin data binders that mount chat-surface destination components (`ChatsArchive`, `ActivityDestination`, `SkillsDestination`, `ProjectsDestination`), composer components are re-export shims or thin adapters over chat-surface shells, and substrate concerns (Router, KeyValueStore, SecretStorage, Badge/Notification/FilePicker/Clipboard ports) are injected as web implementations. An eslint boundary (`eslint.config.js`) bans `fetch`/`EventSource`/`chat-transport` imports inside `src/features/**`, forcing all HTTP/SSE through the typed `src/api/*` modules and the shared `WebTransport` singleton.

The cluster also carries two significant legacies: (1) a ~17.8k-LOC subtree of seven folded destinations (home / library / inbox / todos / routines / agents / memory) plus the superseded `tools` feature and their API modules, all unmounted and explicitly parked "for the Phase-6C dead-code sweep" (`App.tsx:26-28`); and (2) the pre-rename "Atlas" identity that survives in CSS class names, runtime adapter names, and the wallet sign-in page title.

## Public Interface

This is a leaf application — nothing in the repo imports it. Its "public" surfaces are user-facing URLs, build artifacts, and env-var contracts:

- **SPA entry** — `index.html` → `src/main.tsx` (installs global error handlers, renders `<App/>`, defers OTel bootstrap to idle; `src/main.tsx:8-35`).
- **Standalone wallet page** — `wallet.html` → `src/walletEntry.tsx` (second Vite input, `vite.config.ts:13-16`): SIWE nonce → sign → verify → relay bearer to the desktop app's loopback listener via `features/auth/WalletHandoffPage.tsx`. Consumed by `apps/desktop`'s wallet sign-in flow (URL contract, not import).
- **URL routes** (parsed by `src/app/HashRouter.ts:35-46`): `/` → Run cockpit; `/<destination>` for any known `ShellDestinationSlug`; `/settings#<section>` (legacy `/settings/<section>` migrated once on mount); `/settings/notification-defaults`, `/settings/security/webhooks` (P12); `/share/<token>`; `/admin/adapter-review[/<id>]`; `/templates` (project-templates gallery); `/auth/magic-link/callback` (`App.tsx:358-360`); `/mcp/oauth/callback` (`App.tsx:534-543`); `/wallet.html?handoff=...` (`walletEntry.tsx:26`).
- **Build-time env contract**: `VITE_DEFAULT_ORG_ID`, `VITE_DEPLOYMENT_PROFILE` (`App.tsx:272-290`), `GIT_SHA`, `DEPLOY_ENVIRONMENT` (`vite.config.ts:20-23`), `BACKEND_FACADE_URL` (dev proxy target, `vite.config.ts:29`).
- **Deployment artifact**: multi-stage `Dockerfile` → nginx:1.31-alpine serving `dist/` on :8080 with SPA `try_files` fallback (`nginx.conf`); no `/v1` proxying in-image (ingress-level concern).
- **localStorage/sessionStorage keys owned** (see Data Owned).

**Key abstractions.** (1) The **Transport singleton** (`api/transport.ts`): one `WebTransport` from `@0x-copilot/chat-transport` owns bearer attachment, 401 notification, and SSE parsing for the whole app; `AuthContext` configures it on mount; the eslint boundary makes `src/api/*` the only place features can reach HTTP. (2) The **pure reducer chatModel**: server events (`RuntimeEventEnvelope`) fold into `ChatItem[]` plus run-scoped registries (citations, citation links, sources, subagents, drafts) — replay-safe and resume-safe against the monotonic `sequence_no` contract. (3) **Host-binder pattern**: destination routes fetch/compose product endpoints host-side and hand read models to substrate-agnostic chat-surface components. (4) **Ports**: routing, KV, secrets, notifications, file picking are injected interfaces so the desktop substrate can swap implementations without consumer changes.

## Key Flows

1. **Boot + auth** — `main.tsx` installs error handlers → `App` constructs KV store + SecretStorage above `AuthProvider` (so dev persona + bearer resolve pre-shell, `App.tsx:294-317`) → `AuthContext.refresh()` probes `/v1/auth/session`; 401 → dev-only silent mint via `/v1/dev/identity/mint` then re-probe (`AuthContext.tsx:225-267`); anonymous → lazy `LoginScreen` (Google redirect | SIWE nonce→`personal_sign`→verify→`adoptSession` | local persona mint); `mfa_pending` → WebAuthn `MfaPrompt` → `authenticated` mounts profile/preferences/appearance providers + `CopilotApp`.

2. **Run streaming** — `ChatScreen` creates a run (`createRun`) → `streamRunEvents` opens fetch-based SSE with bearer + `after_sequence` cursor (`agentApi.ts:607-649`) → each `RuntimeEventEnvelope` folds through `applyRuntimeEvent` + citation/link/source/subagent/draft reducers (`ChatScreen.tsx:74-104`) → `chatRunState.deriveRunUiState` drives phase UI (planning pulse invariant) → disconnects resume from the highest received `sequence_no`; replay via `/v1/agent/runs/{id}/events`. Background conversations keep streaming through `runtime/useBackgroundChatStreams` slots.

3. **Destination dispatch** — `HashRouter.current()` parses the URL into `AppRoute` → `App` dispatch: folded slugs redirect via `FOLDED_DESTINATION_REDIRECTS` (`routes.ts:95-117`), live slugs mount host binders (`ChatsArchiveRoute` → chat-surface `ChatsArchive`; `ActivityRoute` → `activityApi` composition of conversations+audit → `ActivityDestination`; `ProjectsRoute` → controlled `ProjectsDestination`), everything inside one `ChatShell` with Suspense-per-chunk so the rails stay mounted during route-chunk loads (`App.tsx:992-1029`).

4. **MCP connector OAuth round-trip** — connector auth tool stores a pending action (`mcpAuthAction.ts`) → provider redirects to `/mcp/oauth/callback?state&code` → `App.tsx:534-629` completes via `completeMcpOAuthOnce` (StrictMode-safe de-dupe), auto-approves the originating approval (skipping non-persisted `mcp_discovery:` ids), clears pending state, navigates back to chat/settings and refreshes connectors.

5. **Build/deploy** — `tsc && vite build` emits `main` + `wallet` entries with route-level code splitting (lazy screens, OTel SDK as idle-loaded chunk); Docker multi-stage copies the four shared packages' sources, `npm ci` at the workspace root, then nginx serves `dist/` with SPA fallback (`Dockerfile`, `nginx.conf`).

## Test Posture

157 co-located `*.test.ts(x)` files, ~32.3k LOC (≈32% of the cluster), run by Vitest + jsdom + Testing Library (`vitest.config.ts`). Strong: the pure layers — chatModel reducers (`chatModel.test.ts` 2,531 + per-reducer suites incl. an invariant test for the citation store), `chatRunState` (planning-pulse invariants pinned per CLAUDE.md), API modules (each live module has a suite; `agentApi` has dedicated createRun/createConversation suites), `AuthContext` (654-line suite), HashRouter, App-level routing (`App.test.tsx`), ports, and binder routes (Chats/Activity/Skills/Projects/Library each have route tests). Weak: the three biggest interactive surfaces have **zero direct component tests** — `ChatScreen.tsx` (2,698), `SettingsScreen.tsx` (1,405), `LoginScreen.tsx` (1,084) — coverage there is indirect (integration slices like `AssistantMessage.integration.test.tsx`, child-component suites). Also ~7.4k LOC of tests exercise code that is dead in the product (folded features + their API modules), inflating suite runtime and the apparent coverage. No E2E in this cluster (Playwright journeys live in `tools/cli-testing` per repo memory; not evidenced here).

## Health Assessment

**Strengths.** This is a disciplined host application: the substrate-boundary eslint rules are real and enforced; HTTP/SSE is centralized behind one transport singleton with correct bearer/401/resume semantics; the chatModel event-reducer layer is pure, replay-safe, and heavily tested; auth is a clean explicit state machine with dev-IdP recovery that production tree-shakes; code splitting and deferred OTel show genuine performance care; comments consistently cite PRDs/FRs, and the per-app docs KB is unusually good. The migration to chat-surface as SSOT is executed correctly where it has happened — binders mount package components, composer files are shims, citation reducers wrap the package core.

**Weaknesses.** The IA fold left ~17.8k LOC (≈17% of the cluster) of dead destinations plus dead API modules and their tests awaiting the promised Phase-6C sweep — a real tax on build, test time, grep-ability, and agent context. Settings is implemented twice across the repo (web `features/settings` vs chat-surface `settings/` mounted by desktop), which directly contradicts the documented "both hosts mount the same Settings" SSOT and matches the known 7+-divergent-model-list problem. `ChatScreen.tsx` is a 2,698-line god component with no direct test. `package.json` omits `@0x-copilot/chat-surface`/`chat-transport` despite 100+ import sites (phantom workspace deps). Two identity conventions coexist in the API layer (legacy `org_id`/`user_id` query params vs bearer-only). The Atlas→0xCopilot rename is incomplete (user-visible in `wallet.html`).

**Risks.** Mostly hygiene/velocity rather than correctness: the dead subtree and dual Settings invite divergence-style bugs (a fix landing in one copy only); identity-in-query leaks org/user ids into access logs; phantom deps make CI path-filtering and Docker reproducibility reasoning fragile. Auth/token handling itself looks sound (bearer in SecretStorage abstraction, no cookies/URL tokens except the documented magic-link consume-and-strip path).

## Findings

F1. **[dead-code | high | high]** ~17.8k LOC of folded-destination code is dead but still in-tree — features `home/`, `library/`, `inbox/`, `todos/`, `routines/`, `agents/`, `memory/`, `tools/` (9,623 LOC incl. their tests) have zero non-test importers after the PR-4.11 IA fold (`App.tsx:26-28` explicitly defers them "for the Phase-6C dead-code sweep"; `tools` was superseded by the Skills binder on the `tools` slug), plus API modules `homeApi/libraryApi/inboxApi/todosApi/routinesApi/agentsApi/memoryApi/toolsApi` and stubs `_home/_inbox/_library/_routines/_todos/_agents-stub` (4,151 LOC) and ~3,800 LOC of tests for them. `MemoryGateway` is likewise unmounted (memory folds into Settings → Privacy). evidence: apps/frontend/src/app/App.tsx:24-28; apps/frontend/src/app/routes.ts:95-103; apps/frontend/src/features/tools/; apps/frontend/src/api/\_library-stub.ts:18-20 ("TODO(merge): delete this file"). suggestion: execute the Phase-6C sweep now — delete the eight feature dirs, eight API modules, six stubs and their tests; keep `_projects-stub` (still imported by live projects code) until api-types absorbs it.

F2. **[ssot-violation | high | high]** Settings exists as two parallel implementations: web mounts its own `features/settings/SettingsScreen.tsx` (+17 section panels, 9,441 LOC) while `packages/chat-surface/src/settings/` carries a full second Settings surface (ProfilePage, AppearancePage, ProviderKeysPage, LocalModelsPage, ModelBehaviorPage, NotificationsPage, PrivacyPage, ShortcutsPage, WebhookSecurityPage, SettingsSurface) that only `apps/desktop/renderer/SettingsMount.tsx` mounts — the web app imports none of it.
