# Cluster: AI Runtime — Persistence & Store Adapters

## Purpose

This cluster is the durable-state substrate of the agent runtime. `agent_runtime/persistence` defines the contracts: Pydantic records for every persisted aggregate (approvals, batches, budgets, citations, drafts, outbox commands, retention policies, shares, telemetry/usage rows, todo extractions, tool ordinals), the satellite store ports (`DraftStorePort`, `ShareStorePort`, `CitationStorePort`, `SubagentStorePort`, `SourceStorePort`, `ConversationToolOrdinalStorePort`, `TodoExtractionStorePort` in `persistence/ports.py`), typed concurrency errors + a bounded optimistic-retry helper, field-level envelope encryption (AES-256-GCM with KMS-wrapped per-row DEKs), the yoyo-backed `MigrationRunner`, and OTel metric facades for pools and encryption.

`runtime_adapters` supplies three interchangeable implementations of the big runtime port surface (`PersistencePort` + `EventStorePort` + `RuntimeQueuePort` + `RuntimeStoreLifecyclePort`, all defined in `agent_runtime/api/ports.py`): `in_memory` (tests/dev), `postgres` (production-style shared store: psycopg pool, RLS session vars, lock-free event appends, read-replica routing, chunked retention sweeps), and `file` — the desktop `single_user_desktop` backend that persists Claude-Code-style plaintext JSONL session folders plus a content-addressed object store and a disposable SQLite catalog index, with a hash-chained (HMAC) audit log via the shared `copilot_audit_chain` package. `RuntimeAdapterFactory.from_settings` selects the backend from `RUNTIME_STORE_BACKEND` and composes everything into a frozen `RuntimePorts` bundle consumed by `runtime_api` and `runtime_worker`.

`agent_runtime/retention` is a small pure-logic module: `RetentionPolicyResolver` folds per-org `RetentionPolicyRecord` rows plus per-user privacy overrides into a most-specific-wins TTL lookup (`conversation > assistant > user > org > deployment-default`), shared by the retention sweeper/backfill worker jobs, the retention HTTP routes, and the workspace-defaults/conversation services.

The cluster matters because every product guarantee — tenant isolation, run replay via monotonic `sequence_no`, approval durability, deletion/retention compliance, desktop offline-first storage — bottoms out here. It is also where the three backends' behavior quietly diverges, which is the source of the most serious findings below.

**Architecture notes.** The intended shape is clean hexagonal: records + ports in the domain package, three swappable adapter families, one factory. Two structural decisions dominate the actual shape: (1) the _primary_ port (`PersistencePort`, ~90 methods in `agent_runtime/api/ports.py`) lives outside this cluster and is a god-interface every adapter re-implements wholesale; (2) the file store was written as a fork of the in-memory store (same materialized-view dicts, same method bodies) with write-through JSONL bolted on, rather than as a journaling layer over one shared pure-logic core — `base.py` shares only ~400 LOC of the plausible several thousand. The file internals themselves are genuinely well-factored (single-purpose modules, locked design decisions documented in the store docstring, fail-closed corruption handling with a separate repair path).

## Key Flows

1. **Backend selection & boot** — `RuntimeSettings.load()` reads `RUNTIME_STORE_BACKEND` → `RuntimeAdapterFactory.from_settings` (factory.py:160-227) branches: in-memory (fresh store + satellites), postgres (pool store shared by 5 satellite wrappers + NOTIFY-after-append when the SSE bus is postgres, factory.py:176), or `_build_file_ports` (factory.py:48-106) which fail-closes unless profile=`single_user_desktop` and root set, then wires file satellites off `store.layout`. Caller runs `ports.lifecycle.open()` → file store replays JSONL into the materialized view, rebuilds the SQLite catalog, and runs the startup retention sweep (`file/runtime_api_store.py:321-356`); postgres opens the pool and (dev only) auto-applies yoyo migrations.
2. **Run creation → event stream** — `create_run_with_user_message` builds the user message through the shared `RuntimeAdapterHelpers.message_for_run_request` (base.py:198-259, async sibling :348) handling regenerate/synthetic-parent resolution identically across backends; worker claims commands via `claim_next` (outbox pattern; postgres `FOR UPDATE SKIP LOCKED`-style, file/in-memory ordered dicts); events append with per-run monotonic `sequence_no` — postgres is lock-free with UNIQUE-conflict retry surfacing `RuntimeEventSequenceConflict` (ports.py:67-85, postgres:4738), file appends to `events.jsonl` + catalog.
3. **Retention** — worker `RetentionSweeperLoop` lists orgs → loads policies + privacy overrides → `RetentionPolicyResolver.resolve` per kind (policy_resolver.py:105-136) → `sweep_retention_kind`: postgres does per-kind chunked tombstone/hard-delete across 8 kinds (postgres:3649-4192) + `retention_until` backfill/recompute; file maps everything to whole-conversation physical purge on `MESSAGES` (file:2656-2702) via `_purge_conversations` (plan → erase dirs → drop view/index → GC unreferenced objects → signed audit row, file:1820-1906); in-memory is a stub. Evidence rows land in `runtime_deletion_evidence` / `state/deletion_evidence.jsonl`.
4. **Field encryption (postgres only)** — `FieldEncryptionFactory.from_env` picks Null or Envelope (encryption.py:495-513); `FieldCodec` encrypts/decrypts text + JSONB columns at insert/read call sites with AAD = `table|column|org_id` (postgres:386-394, encryption.py:287-290); strict-reads mode turns v0 rows into errors post-backfill.
5. **Desktop store migration** — operator runs `python -m runtime_adapters.migrate` (migrate.py:44-159): opens source (postgres/in-memory) + dest file store through the same ports, `StoreMigrator.migrate` copies conversations/messages/runs/events, `--verify` re-reads both sides for equality; only a clean verify authorizes flipping `RUNTIME_STORE_BACKEND=file`.

## Test Posture

Tests live in `services/ai-backend/tests/unit/{agent_runtime/persistence,agent_runtime/retention,runtime_adapters}` plus `tests/integration/persistence/test_rls_isolation.py`.

**Well covered:** the file adapter is the best-tested code in the cluster (24 test files: adversarial paths, crash consistency, JSONL corruption, physical delete, quota/cleanup, repair, restart/rebuild, export/import, audit manifest, factory gating, telemetry/health). Core persistence contracts (`test_persistence_contracts.py`), encryption (`test_field_codec.py`, `test_field_encryption.py`, postgres projections, worker backfill), migration runner, optimistic retry, message copy, and the retention resolver each have focused suites. A cross-backend conformance suite exists (`tests/unit/runtime_adapters/test_store_conformance.py`, 504 LOC) covering CRUD, ordering, idempotency, tenant isolation, soft delete, queue lifecycle, regenerate.

**Gaps:**

- The conformance suite deliberately excludes Postgres (`test_store_conformance.py:3-4`) and covers only ~10 of the ~90 `PersistencePort` methods; nothing asserts that each adapter structurally satisfies each Protocol — which is exactly how F1 (three methods missing from Postgres) went unnoticed.
- Approval-inbox/expiry methods are tested only against in-memory stores (`test_approval_rescoped_deltas.py`, `test_approval_forwarding_hardening.py`).
- No test wires the worker's citation store against the file or in-memory `RuntimePorts` end-to-end, so the split-brain in F2 is invisible to the suite.
- Postgres unit tests are DB-gated (`tests/unit/runtime_adapters/postgres/conftest.py`) and skip without a live database — consistent with the memory note that unit fakes have hidden real-run breakage before.

## Health Assessment

**Strengths.** The contract layer is genuinely good: typed records everywhere, well-documented ports with explicit conflict semantics, a disciplined migration story (checksummed manifest, rollback files, CI check), a clean pure retention resolver, and a thoughtfully engineered file store (traversal-safe hashed paths, fail-closed corruption handling with a separate salvage/repair path, content-addressed blobs with reachability GC, signed audit chain, physical deletion with legal-hold skip). Env contracts fail closed in the right places (file backend gating, postgres without DATABASE_URL, KMS misconfig).

**Weaknesses.** The cluster's central abstraction — one ~90-method structural Protocol implemented three times — is doing it in. Structural typing means adapter drift is a runtime 500, not a type error, and drift has already happened (F1). The file adapter duplicates the in-memory adapter's business logic nearly verbatim (F3) instead of layering journaling over one core, so every port change now costs three hand-synced edits and the Postgres SQL a fourth. Wiring that bypasses the factory (the worker's isinstance-based citation store fallback) has produced a real cross-backend correctness bug (F2). Retention/deletion semantics differ materially per backend with no capability signaling (F5). Roughly 2,000 LOC of file-store product surface (export/import, FTS search, health/verify) is built and tested but mounted nowhere (F6), consistent with this repo's build-ahead habit.

**Overall: functional but drifting.** The highest-value refactors are mechanical and well-scoped: split the god port, add a per-adapter Protocol-conformance test (including Postgres), move citation wiring into `RuntimePorts`, and collapse the in-memory/file duplication.

## Findings

F1. **[risk | high | high]** PostgresRuntimeApiStore does not implement three `PersistencePort` approval methods — `list_assigned_approvals`, `list_pending_expired_approvals`, `list_pending_approvals_for_membership_audit` are declared on the port (`agent_runtime/api/ports.py:342,357,365`) and implemented by in-memory (`in_memory/runtime_api_store.py:911,949,973`) and file (`file/runtime_api_store.py:1509,1539,1554`) but absent from the entire postgres package (grep confirms zero hits). `list_assigned_approvals` is live: `runtime_api/http/routes.py:354` → `approval_coordinator.py:140` → `persistence.list_assigned_approvals(...)` — on `RUNTIME_STORE_BACKEND=postgres` the assigned-approvals inbox endpoint raises `AttributeError` (HTTP 500). The other two are called only by `ApprovalExpirySweeper` (`runtime_worker/jobs/approval_expiry_sweeper.py:161,178`), which is itself never wired, masking the gap. The conformance suite excludes Postgres by design (`tests/unit/runtime_adapters/test_store_conformance.py:3-4`). evidence: services/ai-backend/src/agent_runtime/api/ports.py, services/ai-backend/src/runtime_adapters/postgres/runtime_api_store.py, services/ai-backend/src/runtime_api/http/routes.py, services/ai-backend/src/agent_runtime/api/approval_coordinator.py. suggestion: implement the three methods in the Postgres adapter; add a static conformance test asserting every adapter satisfies every port Protocol (a simple `isinstance`/`inspect` check catches this without a database).

F2. **[risk | high | high]** Citation store wiring is split-brain on non-postgres backends. `RuntimePorts` has no `citation_store` field, so `RuntimeWorker` improvises: `citation_store = self.persistence if isinstance(self.persistence, CitationStorePort) else InMemoryCitationStore()` (`runtime_worker/loop.py:74-80`). Postgres passes (it implements `insert_many_or_get` at `postgres/runtime_api_store.py:5692`); `FileRuntimeApiStore` and `InMemoryRuntimeApiStore` do not, so the worker writes citations into a **private throwaway** `InMemoryCitationStore` while the API's Sources feed reads a **different** store — file backend: `FileCitationStore(layout)` (factory.py:93,105); in-memory backend: a second fresh `InMemoryCitationStore` (factory.py:191). Consequences: Workspace "Sources" is always empty on the desktop/file and dev/in-memory backends, `FileCitationStore` durability is never exercised by real runs, and citation rebuild-on-resume after worker restart (`ports.py:205-212`) is impossible off-postgres. evidence: services/ai-backend/src/runtime_worker/loop.py, services/ai-backend/src/runtime_adapters/factory.py, services/ai-backend/src/runtime_adapters/file/citation_store.py, services/ai-backend/src/runtime_adapters/in_memory/citation_store.py. suggestion: add `citation_store` to `RuntimePorts`, wire the correct implementation per backend in the factory, pass it explicitly into `RuntimeWorker`/handlers, and delete the isinstance fallback.

F3. **[duplication | high | high]** The file adapter re-implements the in-memory adapter's business logic nearly verbatim across the ~90-method surface (~11.1k LOC across the three `runtime_api_store.py` files). Example: `charge_budget` is byte-identical between `in_memory/runtime_api_store.py:1835-1870` and `file/runtime_api_store.py:2473-2510` except for `async with self._state_lock` and a `_rewrite_budget_states()` ledger call; the same pattern holds for reservations, usage rollups/queries, pricing, approval batches, tool budgets, and queue semantics. `base.py` shares only ~400 LOC. F4. **[complexity | high | high]** `PersistencePort` is a ~90-method god interface (`agent_runtime/api/ports.py:85-927`) spanning conversations, messages, runs, approvals, approval batches, audit, history deletion, workspace defaults, usage recording, pricing, five daily-rollup families, budgets, tool budgets, and retention. F5. **[inconsistency | medium | high]** Retention and deletion semantics diverge materially per backend with no capability signal on the port: postgres implements chunked tombstone/hard-delete sweeps for all 8 `RetentionKind`s plus `retention_until` backfill/recompute (`postgres/runtime_api_store.py:3649-4303`); the file store collapses every kind into whole-conversation physical purge keyed on `MESSAGES` and returns empty tallies for the rest, with `backfill_retention_until`/`recompute_retention_until_for_policy` as hard-coded `return 0` stubs (`file/runtime_api_store.py:2656-2722`); in-memory `sweep_retention_kind` is an explicit no-op (`in_memory/runtime_api_store.py:1027-1053`). `delete_user_history` likewise tombstones on postgres/in-memory but erases bytes on file (`file/runtime_api_store.py:1696-1738`).
