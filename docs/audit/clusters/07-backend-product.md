# Cluster: Backend — Product Domain Modules

## Purpose

This cluster is the product-domain half of `services/backend`: the "destinations" (Todos, Inbox, Routines, Projects, Library, Memory, Agents, Tools, Connectors, Home, Palette) that back the 0xCopilot shell's left-rail surfaces, plus the tier-2 adapter registry, outbound webhooks, notification-preference storage, preloaded skill prompts, and the service's entire hand-written HTTP wire layer in `routes/`. Every destination follows the same phase-built shape: a `store.py` adapter contract + in-memory implementation, a `service.py` that owns ACL ("owner-only writes; project-member reads; admin compliance reads; 404-not-403") and audit-row emission, a `routes.py` presentation layer, a package-local `schema.sql` describing the intended Postgres shape, and (for five destinations) an `sse.py` live-update stream.

The `routes/` directory is a different animal from the destination packages: it is the wire layer for the identity/settings/admin surface — 29 files registering `/internal/v1/auth/*`, `/internal/v1/me/*`, `/internal/v1/workspace/*`, `/internal/v1/policies/*`, `/v1/siem/*`, and `/healthz`+`/readyz`. These delegate to the identity services (`backend_app/identity/*`, the backend-identity cluster) for auth flows, but for the settings domain (preferences, notifications, privacy, tool-use policies, API keys) the business logic lives directly in the route files against bare stores — there is no service layer, unlike the destinations.

The cluster exists because the target architecture makes `backend` the home for "product persistence, admin workflows, and jobs" (CLAUDE.md). Everything here is reachable only through `backend-facade` (public `/v1/*`) or through the service-token internal lane (`/internal/v1/*`) consumed by ai-backend. Cross-destination reuse is deliberate in a few good places — the canonical project ACL (`projects/acl.py`), Memory riding Library's embedding queue and search primitives, webhook HMAC constants in `webhooks/signer.py` — and accidental in many others (six copy-pasted SSE stacks, 14 copies of `_ADMIN_ROLES`, near-identical CRUD service scaffolding per destination).

The single most important health fact discovered: a large fraction of this cluster is _scaffolding that is not yet wired to real production behavior_. All destination stores are in-memory in every shipped composition root; the web Connectors OAuth surface is a stub; three of the five SSE streams have no producers; the palette index has no writers; the agent-installs route module (1,058 LOC) is never registered. The code quality per module is high (typed, documented, tested), but the wiring layer tells a different story than the module docstrings do.

## Key Flows

1. **Run-start policy snapshot (ai-backend → backend):** ai-backend calls `GET /internal/v1/policies/runtime?org_id&user_id` with service token → `internal_scoped_identity` re-binds identity from verified headers (routes/runtime_policies.py:123) → composes workspace+user tool-use rows, merged privacy row, and **decrypted** BYOK provider keys via `ProviderKeysService.decrypted_keys` (routes/runtime_policies.py:182) → single JSON snapshot cached on the run context.
2. **Agent-produced inbox item:** ai-backend posts `POST /internal/v1/inbox/items` (inbox/internal_routes.py) → resolves canonical `InboxService` off app.state → idempotency on (producer_id, external_ref) → store insert + audit row → `activity_bus.publish("item_added")` (inbox/service.py:457) → subscriber's `GET /v1/inbox/stream` SSE loop emits the frame with monotonic `sequence_no` (inbox/sse.py:384-418); FE reconnects with Last-Event-ID.
3. **Desktop connector OAuth (the real path):** desktop calls `POST /v1/connectors/{slug}/desktop/start-oauth` → `DesktopMcpOAuthCoordinator` validates profile + preview gate + loopback port, reconstructs the redirect URI, drives `McpRegistryService.start_auth` (state+PKCE+TokenVault) (connectors/oauth*coordinator.py:1, wired app.py:1683-1694) → callback endpoint enforces caller-owner match → tokens land in the vault; only safe metadata returns. The \_web* equivalent (`/v1/connectors/{slug}/start-oauth`) instead resolves `app.state.connector_oauth_start` which nothing sets, falling back to a hard-coded `https://auth.example/...` stub (connectors/routes.py:557-568), and `/v1/connectors/oauth-callback` 503s (connectors/routes.py:333-339).
4. **Inbound routine webhook fire:** external caller posts `POST /v1/webhook/routines/{trigger_id}` (public) → `routines/webhook.py` decrypts current+grace secrets from TokenVault, constant-time compares `x-atlas-routine-secret`, verifies `hmac-sha256=<hex>` body signature, checks CIDR allowlist → route emits `routine.fire_webhook[_unauthorized]` audit rows → fire enqueued (routines/webhook_routes.py:1).
5. **Library upload + hybrid search:** `POST /v1/library/files/upload-grant` mints a signed PUT URL (bytes never proxy through the API; library/upload_routes.py:1) → finalize HEADs the blob and records size/sha → `LibraryService` enqueues a `library_index_jobs` row (library/service.py:447) → indexer worker (backend_app/jobs/library_indexer.py, outside this cluster) claims → chunk → embed via ai-backend port → `GET /v1/library/search` fuses BM25 + vector with RRF and optional rerank (library/search.py:1). Memory enqueues into the same queue with `target_kind="memory"` (memory/indexer.py:1).

## Test Posture

Tests live in `services/backend/tests/` (flat files) and `services/backend/tests/unit/{adapter_registry,connectors,library,memory,palette,tools,webhooks,...}`. Coverage is broad and behavior-oriented: every destination has service + routes tests (test*todos*_, test*inbox*_ incl. SSE publish, test*routines*_ incl. webhook auth, test*projects*_ incl. ACL/archive/allowlist-inheritance, test*library*\* incl. blob/chunking/indexer, unit/memory, unit/tools, unit/webhooks incl. rotation worker, unit/adapter_registry, unit/palette), plus cross-cutting suites (test_tenant_isolation_skills_mcp.py, test_rls_session_vars.py, test_atomicity.py, test_runtime_policies_route.py, test_suggestible_connectors.py).

Gaps: (1) tests validate the in-memory adapters — for ten destinations there is no Postgres adapter to test, so schema.sql files are untested DDL; (2) dead modules are "covered" only by their own tests (test_agent_installs_routes.py builds its own app at line 154; test_notification_dispatcher_gate.py; unit/palette/test_palette_refresh.py), which masks the fact that nothing in production reaches them — coverage here creates false confidence; (3) no test asserts that the web connectors OAuth path is actually wired (the stub satisfies the route tests); (4) no test asserts a producer publishes to the tools/connectors/home buses, so the silent streams pass CI.

## Health Assessment

**Strengths.** Per-module engineering discipline is genuinely high: consistent store/service/routes layering with ACL and audit centralized at the service layer; the canonical project ACL is a real SSOT with a "second implementation is a bug" contract (projects/acl.py:1); Memory's reuse of Library's queue and fusion primitives is exactly the right call; identity binding is uniformly done via `scoped_identity`/`internal_scoped_identity` from verified bearers, never from bodies; inbound webhook auth (rotating secret, grace window, constant-time compare, CIDR) and the desktop OAuth coordinator (redirect reconstruction, owner-match, preview fail-closed) are careful security work; tests are extensive.

**Weaknesses.** The cluster is phase-built by parallel subagents and it shows at the seams: six copy-pasted SSE stacks, 14 copies of `_ADMIN_ROLES`, per-destination exception/validation/`_safe_dump` scaffolding, and a 2,049-line composition root that reaches into private attributes to share ports. More seriously, a lot of shipped surface is _wiring-incomplete_: the web connectors OAuth alias is a stub in every composition, the connectors read model has no writer, the palette index has no writer, three SSE streams have no producers, agents installs (1,058 LOC) is unregistered, the webhook rotation worker has no launcher. Docstrings say "production wires X" for things no composition root wires.

**Risks.** (1) Durability: every destination's data is process-lifetime-only in every deployment — desktop and self-host users lose todos/inbox/projects/memory/etc. on restart; (2) product truth: FE features (⌘K palette, connectors "Connect", live activity rails) hit endpoints that return empty/stub/503 forever, which will surface as user-facing bugs attributed to the FE; (3) drift: hand-mirrored DDL (todos schema vs migration 0032), hand-mirrored wire types vs api-types, and the duplicated recurrence grammar with ai-backend can all drift silently.

Verdict: well-engineered modules on an unfinished chassis. The highest-leverage work is not new features but closing the wiring gaps and collapsing the copy-paste infrastructure.

## Findings

F1. **[risk | high | high]** Web Connectors OAuth surface is a permanent stub — `POST /v1/connectors/{slug}/start-oauth` resolves `app.state.connector_oauth_start`, which no composition root ever sets, so every deployment returns the hard-coded `https://auth.example/{slug}/authorize?state=stub` URL; `POST /v1/connectors/oauth-callback` always 503s (`connector_oauth_not_configured`); and `ConnectorsService.write_through_from_mcp` — the only writer of connector rows — has no caller, so the destination's installed list is permanently empty. The FE calls these routes (apps/frontend/src/api/connectorsApi.ts). Evidence: services/backend/src/backend_app/connectors/routes.py:281-346,557-568; services/backend/src/backend_app/connectors/service.py:269; services/backend/src/backend_app/app.py:1618-1660. Suggestion: wire `connector_oauth_start`/`connector_oauth_callback` to `McpRegistryService.start_auth`/`complete_auth` in `create_app` and call `write_through_from_mcp` from the callback leg (the desktop coordinator shows the shape); until then, hide the web Connect button behind capability detection.

F2. **[risk | high | high]** All destination stores are in-memory in every shipped composition — no Postgres adapters exist for todos/inbox/routines/connectors/webhooks/projects/templates/library/memory/agents/tools/palette (only identity/settings/MCP/adapter-registry have `Postgres*Store` classes); the SaaS image default CMD boots `backend_app.app:app` with all-in-memory defaults; desktop and self-host boot `desktop_app` which explicitly lists every destination store as an "accepted desktop-v1 gap".
