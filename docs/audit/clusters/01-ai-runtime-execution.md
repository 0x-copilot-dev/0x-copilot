# Cluster: AI Runtime — Execution Core

## Purpose

This cluster is the domain heart of the ai-backend service: it turns a validated request context plus a set of injected capability ports into a runnable LangGraph/Deep Agents graph, and provides the surrounding domain machinery — model selection, reasoning-depth budgets, system-prompt assembly, tool-error policy, in-process subagent delegation, context/memory policy, run budgets, and model pricing.

The central artifact is the **runtime factory** (`execution/factory.py`): `acreate_agent_runtime` fans out five registry listings concurrently (tools, MCP servers, subagents, skill directories, skill cards), assembles the system prompt from `prompts/` fragments + MCP/skill/suggestion cards, composes optional Deep Agents filesystem backends (`/subagents/`, `/drafts/`, `/large_tool_results/`, `/workspace/`, file-native memory routes), computes per-provider model kwargs (training opt-out, region pinning, BYOK keys), and hands a `DeepAgentBuildRequest` to `deep_agent_builder.build_deep_agent`, which is the single funnel for every LangChain chat-model construction (`init_chat_model`) including OpenAI-compatible gateways (OpenRouter, local Ollama). The result is a frozen `RuntimeHarness` that `execution/runtime.py` invokes/streams with traced, error-wrapped helpers.

Everything else in the cluster supports that path: `execution/contracts.py` is the strictly-validated Pydantic contract layer (`AgentRuntimeContext`, `RuntimeDependencies`, `ModelConfig`, stream events, error envelopes) consumed by essentially every other module in the service; `delegation/` supplies the subagent catalog and the `atlas_task_tool` monkey-patch that makes supervisor→subagent call-id linkage deterministic; `context/memory` supplies scoped memory routing + policy + summarization/offload + the read-only `/subagents/` trace backend; `budgets/` + `pricing/` implement pre-run cost preflight (Allow/Warn/Deny + reservations) and post-run micro-USD charging against a LiteLLM-vendored pricing catalog; `deployment/` resolves the process-wide deployment profile (saas / single-tenant / desktop) that gates safety toggles.

The cluster deliberately does not do HTTP, persistence, or event emission itself — those live in `runtime_api`, `runtime_worker`, and `runtime_adapters`, which consume this cluster's contracts. In practice that separation is imperfect (see Findings F3): several domain modules import `runtime_api.schemas` back.

Architecturally the cluster is contract-first: everything flows through frozen Pydantic models with heavy per-field normalization delegating to a shared `ValueNormalizer`. Dependency inversion is done properly at the port layer (`RuntimeDependencies` validates protocol methods rather than importing adapters), and optional desktop-only backends (drafts / workspace / large-results / code-mode / sandbox tools) enter as opaque `object | None` fields so non-desktop images stay byte-identical. Two intentional global mutations exist: `install_atlas_task_tool()` monkey-patches `deepagents.middleware.subagents._build_task_tool` at `factory.py` import time, and `runtime_checkpointer()` maintains a process-wide lazy singleton (InMemorySaver, or AsyncSqliteSaver when the desktop file store env vars are set).

## Key Flows

1. **Run execution (worker path)** — `runtime_worker/handlers/run.py` builds `AgentRuntimeContext` from the claimed run, calls `BudgetEstimator.estimate` → `BudgetEnforcer.preflight` (Allow/Warn/Deny + reservation; `budgets/enforcer.py:82`), then `acreate_agent_runtime` (`factory.py:92`): 5-way `asyncio.gather` listing → `_assemble_harness` composes model tools (+gated MCP/skill/code-mode/sandbox tools, `factory.py:311-413`), prompt blocks (`factory.py:218-236`), CompositeBackend routes (`factory.py:654-720`), workspace-write HITL permission (`factory.py:438-462`), policy model kwargs (`provider_kwargs.py`), and `build_deep_agent` (`deep_agent_builder.py:242`) → worker drives `astream_runtime` (`runtime.py:258`) with `RuntimeStreamOptions.rich()`; chunks pass through `CitationStreamPipeline.for_provider` (`providers/citation_pipeline.py:91`) before `MODEL_DELTA` emission; on completion `CostCalculator.compute` + `BudgetCharger.charge_run` (`budgets/charger.py:40`) settle usage.
2. **Model resolution + depth** — request `ModelSelectionRequest` → `ModelConfigResolver.resolve` (`models.py:46`): provider alias/inference, BYOK-aware credential gate (keyless Ollama bypass), defaults merge, then `DepthBudgetTable.apply` scales timeout/max_output_tokens/tool_call_budget by Fast(0.5/0.6/0.5)/Balanced(1)/Deep(2/1.5/2) multipliers with clamps (`depth.py:92-139`). `build_chat_model` reads `timeout_seconds`/`max_output_tokens` (`deep_agent_builder.py:365-375`).
3. **HITL approval resume** — `call_mcp_tool` (and `/workspace/**` writes) are registered as interrupt tools (`factory.py:416-462`); the graph pauses on a checkpoint (`runtime_checkpointer`, `deep_agent_builder.py:278`); after the user decides, `runtime_worker/handlers/approval.py` calls `astream_runtime_resume(harness, Command(resume=...))` (`runtime.py:287`), resuming from the same `thread_id=run_id` config (`runtime.py:47-73`).
4. **Subagent delegation + trace readback** — the supervisor's `task` tool is the patched `build_atlas_task_tool` (`atlas_task_tool.py:70`), stamping `supervisor_task_call_id` into subagent RunnableConfig metadata so worker stream handlers link subgraph events deterministically; afterwards the model can `ls /subagents/` — `SubagentArtifactsBackend` (`subagent_trace.py:369`) walks conversation run_ids via the persistence port and projects visible `RuntimeEventEnvelope`s into four virtual files on demand.
5. **Pricing refresh** — worker startup seeds via `PricingComposer.load` (LiteLLM primary or YAML for air-gapped, per `DeploymentProfile.pricing_primary_source`) merged with overrides → `plan_actions`/`apply_actions` idempotent upserts (`upsert_planner.py`); `PricingRefreshLoop` re-ingests every 24h with a ±25% sanity bound and `auto_apply` gate (`refresh_loop.py`); `ModelPricingCatalog` LRU-caches lookups for the per-run cost hook (`catalog.py:29`).

## Test Posture

Tests live in `services/ai-backend/tests/unit/agent_runtime/` — this cluster maps to `agent/` (15 files: factory async/BYOK/capability-tools/skill-cards, deep-agent builder, contracts, dependencies, errors, observability, web-subagent suffix), `execution/` (9 files: depth, tool-error sanitizer/policy, citation pipeline, suggested-connectors prompt, workspace write gate, file-store checkpointer, openai-compat, user-policy kwargs), `delegation/` (atlas task-tool signature, fs permissions), `subagents/` (async lifecycle), `memory/` (context management, policy user-toggle, prompt injection, subagent trace), `budgets/` (period, estimator, enforcer+charger), `pricing/` (7 files incl. refresh loop, compare CLI, seed script), `deployment/` (profile), plus `test_provider_kwargs.py` and `test_import_boundaries.py`.

Well covered: contract validation, depth scaling, tool-error sanitization, pricing math/ingestion (byte-identical rounding asserted), budget preflight/charge, deployment profile matrix, factory assembly permutations (BYOK, skill cards, capability tools). Gaps: (a) everything runs against fakes — no test constructs a real LangChain model or deepagents graph, so integration breakage surfaces only in live smoke (a known past incident per project memory); (b) `runtime_config`'s trace-metadata behavior for root runs is unasserted (F6); (c) the dead async-subagent lifecycle has the _most_ elaborate tests in the delegation area despite zero production use (F2); (d) no test ties the prompt's tool-cap number to the enforced ToolBudgetRecord (F1); (e) `test_import_boundaries.py` only bans _legacy_ module paths — it does not enforce the documented domain→api layering direction (F3).

## Health Assessment

**Strengths.** This is a genuinely well-engineered domain core: frozen contract-first Pydantic everywhere, real dependency inversion at the ports, single-funnel model construction with a CI guard, careful byte-identical-image gating for desktop-only features, integer-only money math with matched rounding on both ingest and charge sides, and unusually good inline documentation of _why_ (checkpoint-wording regression, monkey-patch rationale, cache scoping). The pricing subsystem (vendored source + seeds + overrides + sanity-bounded refresh + idempotent planner) is production-grade.

**Weaknesses.** The cluster carries a visible layer of shipped-but-never-wired machinery: the async subagent lifecycle (~550 LOC + contracts), `ConfiguredRuntimeGraph`, `FeatureFlag`, the depth-scaled `tool_call_budget` that nothing reads, and doc-stubs like `BudgetReservationManager`'s promised-but-absent methods. Prompt-vs-enforcement facts (tool-call cap, checkpoint discipline) have drifted into contradiction because the prompt text is baked at import time while enforcement values live in settings/DB rows. The "pure domain" story is compromised by `runtime_api.schemas` imports from domain modules (subagent_trace here; a wider pattern in capabilities), which the boundary test doesn't catch. Compliance-flavored knobs (Anthropic/Gemini training opt-out) are weaker than their code comments imply.

**Risks.** Medium overall. The dead code is inert but misleads readers and auditors; the SSOT drift on the tool-cap actively sends the model a number different from the enforced one; the training-opt-out gap is a real compliance-claims risk for regulated buyers; the module-level checkpointer singleton + env reads make the domain's behavior depend on process env in a way that is easy to misconfigure in the desktop supervisor.

## Findings

F1. **[ssot-violation | high | high]** Per-tool call cap has 3+ divergent sources and a dead scaling path — the prompt suffix is baked at import with `_DEFAULT_TOOL_CALL_BUDGET = 5` (`deep_agent_builder.py:44,104,168-188`), `RuntimeSettings.execution.tool_call_budget` defaults to **6** (`settings.py:117`) and is copied onto `ModelConfig` then depth-scaled (3/6/12) by `DepthBudgetTable.apply` (`models.py:117-122`, `depth.py:127-130`), but **no production code reads `ModelConfig.tool_call_budget`**; actual enforcement is `ToolBudgetRecord.max_calls_per_run=6` seed rows in the adapters (`runtime_adapters/in_memory/runtime_api_store.py:167-175`) via `ToolBudgetMiddleware`.
