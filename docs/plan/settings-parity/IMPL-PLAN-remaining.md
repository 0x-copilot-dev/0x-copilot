## D1 — reasoning-depth vocabulary unification + persisted-default consumption at run-create (settings-parity)

**Current state:** Three distinct vocabularies encode one concept. (1) WIRED per-run: `ReasoningDepth = fast|balanced|deep` — Python StrEnum in services/ai-backend/src/agent_runtime/execution/depth.py:31, api-types `ReasoningDepth` (packages/api-types/src/index.ts:1399) + `AgentReasoningDepth` (agents.ts:99) + chat-surface composer `ThinkingDepth`/`Depth` (composer/depth.ts:23, ModelPicker.tsx:25). It is consumed exactly once via DepthBudgetTable.apply inside ModelConfigResolver.resolve (execution/models.py:141), fed by CreateRunRequest.reasoning_depth (runs.py:223) threaded at run_coordinator.py:511. (2) STORED-but-UNCONSUMED: `ReasoningEffort = low|medium|high` on `WorkspaceBehaviorOverrides.default_reasoning_effort` (workspace_defaults.py:78 + api-types index.ts:681/697). It persists to the org-scoped behavior_overrides JSONB and rides into AgentRuntimeContext.workspace_behavior_overrides, but the only execution consumer (execution/provider_kwargs.py:92) reads training_data_opt_out ONLY — nothing maps default_reasoning_effort to a depth or budget. (3) LOCAL-ONLY UI: `ReasoningDepth = auto|quick|standard|deep` — chat-surface ModelBehaviorPage.tsx (name COLLIDES with api-types ReasoningDepth; re-exported at chat-surface index.ts:976 + settings/index.ts:247). It is never persisted: desktop SettingsMount.tsx:844 mounts ModelBehaviorPage but persists only patch.defaultModel; reasoningDepth stays in local useState (DEFAULT_MODEL_BEHAVIOR.reasoningDepth="auto", SettingsMount.tsx:136). Worse, the two hosts are NOT lockstep: web SettingsBinder.tsx:354 renders the LEGACY features/settings/sections/ModelAndBehavior.tsx (which DOES persist default_reasoning_effort low/med/high), while desktop renders the chat-surface ModelBehaviorPage — different component, vocabulary, and persistence. Separately, the interactive composer ALWAYS sends a concrete reasoning_depth: ChatScreen.tsx resolves KV chats.default_depth → per-user → hardcoded DEFAULT_THINKING_DEPTH "balanced" (chatDepthKv.ts readDepthOrDefault, never null), so today the workspace default would never be consulted for composer-originated runs even if it were wired.

**Plan:**

- CANONICAL: adopt the runtime's fast|balanced|deep as the ONE wire vocabulary (already the SSOT in execution/depth.py + api-types ReasoningDepth + AgentReasoningDepth + composer ThinkingDepth). 'Auto' is NOT a fourth enum value — it is the null/absent sentinel meaning 'no persisted default → runtime baseline'. The design's Auto/Quick/Standard/Deep become presentation LABELS over values null/fast/balanced/deep. This retires low|medium|high (ReasoningEffort) and the ModelBehaviorPage private auto|quick|standard|deep union entirely, leaving depth.py as the single source of truth.
- MAPPING (single table, owned by chat-surface for UI + api-types for wire): Auto↔null (field omitted; run-create leaves reasoning_depth None → DepthBudgetTable no-op → runtime baseline); Quick↔fast (0.5×/0.6×/0.5×); Standard↔balanced (1.0× baseline, attaches reasoning_depth); Deep↔deep (2.0×/1.5×/2.0×). Legacy read-translation low→fast, medium→balanced, high→deep.
- PERSISTENCE = workspace_defaults (recommended over backend me_preferences): the record already lives in ai-backend and is already fetched at run-create, so consumption needs no new cross-service HTTP hop; me_preferences lives in the `backend` service and holds appearance/notification prefs, not run-execution policy. Rename the JSONB field default_reasoning_effort → default_reasoning_depth typed ReasoningDepth|null (null==Auto). JSONB blob → no column migration.
- BACKWARD-COMPAT (sharp): WorkspaceBehaviorOverrides has model_config extra='forbid', so existing rows carrying the old key default_reasoning_effort would FAIL read-validation after the rename. Add a model-level `before` validator on WorkspaceBehaviorOverrides that, when it sees a legacy default_reasoning_effort (low/medium/high), translates it into default_reasoning_depth and drops the old key — so old JSONB rows read cleanly with zero downtime and no data migration required.
- RUN-CREATE SEAM: in run_coordinator.create_run, consolidate the two existing get_record calls (line 400 in \_apply_workspace_default_model + line 461 in \_resolve_workspace_behavior_overrides) into ONE fetch threaded through, then add \_apply_workspace_default_reasoning_depth: if request.reasoning_depth is None AND record.behavior_overrides.default_reasoning_depth is a concrete depth, set request.reasoning_depth to it; else leave None. Precedence: explicit per-turn pick > workspace persisted default > runtime baseline (Auto/None). Agents/routines carry their own concrete reasoning_depth so the None-guard never clobbers them.
- chat-surface ModelBehaviorPage: delete the private `export type ReasoningDepth = auto|quick|standard|deep`, import ReasoningDepth from api-types, model the field as `reasoningDepth: ReasoningDepth | null` (null==Auto). REASONING_DEPTHS becomes [{value:null,label:'Auto'},{value:'fast',label:'Quick'},{value:'balanced',label:'Standard'},{value:'deep',label:'Deep'}]; the Select maps null↔'' for the DOM exactly like the existing defaultModel field does.
- BOTH-HOST LOCKSTEP (the actual web/desktop mismatch this worktree is named for): switch web SettingsBinder.tsx model-behavior slot from the legacy ModelAndBehavior to the chat-surface ModelBehaviorPage, and retire features/settings/sections/ModelAndBehavior.tsx + its ReasoningEffort pills. In BOTH SettingsMount.tsx and SettingsBinder.tsx, initialize reasoningDepth from workspaceDefaults.behavior_overrides.default_reasoning_depth ?? null and, on patch.reasoningDepth, persist via a read-merge-PUT of behavior_overrides.default_reasoning_depth — mirroring the existing persistDefaultLocalModel/persistDefaultModel read-merge pattern so both binders stay behaviourally identical.
- OPTIONAL follow-up (makes the setting observable in interactive chat): seed the composer's per-user default from the workspace default — when KV chats.default_depth is unset, fall back to workspace default_reasoning_depth instead of hardcoded 'balanced' in chatDepthKv.readDepthOrDefault/ChatScreen. Without this, the run-create seam only governs runs that OMIT reasoning_depth (programmatic/API/routine), because the composer always sends a concrete value.

Preflight runs BEFORE status→RUNNING (run.py:268); the real message list is only assembled later at run.py:384 via `_messages_for_run`, which already produces token_counter-ready `{role, content:str}` dicts. `BudgetEnforcer.preflight` takes a concrete BudgetEstimate, short-circuits when no ACTIVE budgets (enforcer.py:99), and uses `est_tokens = input+output` (enforcer.py:108) — so the estimate is computed on EVERY run even with no budgets.
