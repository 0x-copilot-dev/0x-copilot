Scores are 0–10 per axis. "Migration cost" is scored so that **higher = cheaper/lower cost** (a better score), to keep the table directionally consistent (higher is always better).

| **P2 — Port-first, one RunDestination / two hosts** | 9           | 10                  | 8                       | 9                 | 7                          | **8.6**        |
| **P3 — Risk-gated, reversible-first, delete-last**  | 9           | 9                   | 10                      | 9                 | 8                          | **9.0**        |

**Notes on the deltas that matter:**

- **P1** is correct and lean; its one weakness is modelling the MCP-OAuth seam as a bare `onConnectMcp` callback. That works, but it under-specifies the desktop-vs-web divergence and makes the desktop binder's contract implicit. Boundary score docked one point for that ambiguity.
- **P2** has the cleanest boundary reasoning: it names the _one_ genuinely substrate-divergent capability (MCP-OAuth launcher) and gives it a typed `McpAuthPort`, while explicitly _refusing_ to invent ports for cancel / conversation-nav (zero substrate divergence). This is the correct "port only where substrates differ" discipline and earns the 10. Its migration cost is marginally higher because the port type + web/desktop binders are more upfront ceremony than P1's callback.
- **P3** wins on sequencing. It is functionally P2 plus a _risk-gated ordering_: Phase-9 first (improves the still-live ChatScreen, flag-independent), the E1/E2 OAuth split so an intermediate landing can never touch host OAuth code, and an explicit re-audit → bake → delete discipline that never removes ChatScreen before parity re-passes. That is the single most important property for an irreversible cutover.

**Recommendation: synthesize P3's risk-gated sequencing + delete-last discipline with P2's typed `McpAuthPort` boundary and P1/P2's cockpit-owned cancel.** The keystone (web `RunComposer`) is Phase 1; Phase-9 pagination ships early and non-breaking; ChatScreen/`chatRunState`/flag deletion is the last, explicitly-gated phase.

## 2. Problem statement

Under the `runCockpitWeb` flag, `apps/frontend` mounts the shared `@0x-copilot/chat-surface` `RunDestination` cockpit, but the **host never wires the seams the cockpit already exposes**, so the web cockpit is not at parity with the legacy `ChatScreen` (`apps/frontend/src/features/chat/ChatScreen.tsx`, ~2755 lines):

1. **Turn-N composer is inert (keystone).** `RunRoute` mounts `RunDestination` with `renderEmptyComposer` but **no `renderComposer`** (`apps/frontend/src/features/run/RunRoute.tsx:212-219`), and there is no web `RunComposer`. A second in-chat message cannot start/bind a run.
2. **Reopen / switch is broken.** `App.openRun` discards the id every caller passes (`apps/frontend/src/app/App.tsx:814-816`); `RunRoute` is mounted without a `conversationId` (`App.tsx:994-996`), so the already-added `RunRoute.conversationId` groundwork prop is inert.
3. **New-chat URL identity is lost.** `RunRoute` swallows the lazily-created `conversation_id` internally (`RunRoute.tsx:177-180`); there is no run-destination URL slot (`routes.ts`/`HashRouter.ts` special-case run → `/`), so reload/deep-link loses the thread.
4. **Cancel/Stop is unwired end-to-end** — even on desktop. `AssistantComposer` supports `running`/`onCancel` (`packages/chat-surface/src/composer/AssistantComposer.tsx:130-135,580-591`) but the `renderComposer` ctx never supplies them and `RunDestination` never derives run-running state or POSTs cancel.
5. **Mid-run MCP-OAuth resume is unreachable.** `completedMcpAuthAction`/`oauthStatus` are minted by App's `/mcp/oauth/callback` effect (`App.tsx:632-729`) and threaded **only into ChatScreen** (`App.tsx:1005`), never into `RunRoute`; the OAuth return navigates with no conversation id.
6. **Optimistic user echo is a deferred follow-up** in `useRunTranscript` (`packages/chat-surface/src/destinations/run/useRunTranscript.ts:19-22`).
7. **Should-fixes** (in-chat citation chips, run-phase pulse, Sources tool-citation merge) have ready-but-unwired chat-surface seams.
8. **Phase-9 latent bug:** `list_messages` is oldest-N-ASC (`services/ai-backend/src/runtime_adapters/postgres/runtime_api_store.py:1096-1106`), silently dropping the newest turns past the limit; `next_cursor` is declared but never populated. This truncates transcripts for **both** the live ChatScreen and the cockpit.

The gaps are **host wiring + a small set of additive chat-surface props/selectors + one backend pagination fix** — not missing facade routes (all required routes already exist on `:8200`).

## 3. Goals

- **G1.** Bring the shared web `RunDestination` cockpit to full ChatScreen parity on the six MUST-FIX flows (turn-N send, reopen/switch, new-chat URL, cancel, mid-run MCP-OAuth resume, optimistic echo) plus the citation/pulse/Sources should-fixes.
- **G2.** Flip `runCockpitWeb` on and retire `ChatScreen` (and `chatRunState`, dead neighbors, the flag) **only after** a re-run parity audit passes and a bake window completes.
- **G3.** Converge web and desktop onto the **identical** `RunDestination` cockpit; every capability lands once in chat-surface (both substrates inherit it) or once per host binder where the substrate genuinely differs.
- **G4.** Fix Phase-9 `list_messages` truncation for **both** consumers without changing either consumer's code (response stays ASC).
- **G5.** Every intermediate state is flag-scoped and reversible; nothing can regress the flag-OFF ChatScreen path until the final gated deletion.

---

## 4. Non-goals

- **NG1.** Porting ChatScreen's freeze/thaw multi-conversation background-run store (`useBackgroundChatStreams`). The cockpit's single `boundRunId` + HEAD-resolution is the deliberate simplification and replaces it.
- **NG2.** Loading skills into the web run composer's `/`-menu (web `RunEmptyComposer` passes `skills:[]`). Matching the current web empty composer is in scope; matching desktop's skills load is a tracked follow-up (Open Question OQ-3).
- **NG3.** Full Tools nav parity (`onShowConnectors` / `onOpenSkillsSettings` for custom-MCP / skills deep-links) — requires App to thread new nav callbacks into `RunRoute`; tracked as a follow-up, not a MUST-FIX.
- **NG4.** `loadMore` / infinite-scroll consumer wiring for older pages. Phase-9 populates `next_cursor`; consuming it is a separate opt-in follow-up.
- **NG5.** Any change to how the desktop host binds the cockpit beyond inheriting shared chat-surface additions (cancel, echo, citations).

## 5. Functional requirements

- **FR-1 (keystone).** A web `RunComposer` (`apps/frontend/src/features/run/RunComposer.tsx`) mounts the shared `AssistantComposer` and routes send through the injected `ctx.dispatch` (a `RunStartRequest`) — **never its own POST** — so turn-1 and turn-N share the one bind sink (`RunDestination.handleStartRun → session.bindRun`).
- **FR-2 (reopen/switch).** `App.openRun(id)` → `openConversation(id)` (else `openNewRun()`); `RunRoute` receives `conversationId={activeConversationId}` and re-keys the cockpit; `useRunSession` HEAD-resolves and binds the run.
- **FR-3 (new-chat URL identity).** `RunRoute.onConversationCreated(id)` bubbles the lazily-created id to `App.openConversation`; a run-destination URL slot (`/run/<conversationId>`) persists it so reload/deep-link reopens the thread.
- **FR-4 (cancel/Stop).** The `renderComposer` ctx carries `running:boolean` + `onCancel:()=>Promise<void>`; `RunDestination` derives `running` from `session.runStatus`, and `onCancel` optimistically settles pending interaction parts, clears `boundRunId` in the same tick, then best-effort POSTs `/v1/agent/runs/{boundRunId}/cancel` via the injected Transport port. The Stop button must **not** re-arm after cancel.
- **FR-5 (mid-run MCP-OAuth).** The cockpit recognises `approval_kind: 'mcp_auth'`, renders a Connect card, and invokes a host-supplied `McpAuthPort.beginAuth(serverId)`; `mcp_discovery` cards resolve **locally** (never POST). On OAuth return the host resolves run→conversation via `GET /v1/agent/runs/{run_id}` and calls `openConversation`, after which cursor-based SSE resume rebinds the stream. Connect must **not** overload `resolveApproval`'s `/decision` POST.
- **FR-6 (optimistic echo).** `useRunTranscript` accepts a `pendingUserMessage` fed at dispatch time; it merges into `history ⊕ live` and dedupes against the run-start `/messages` re-seed (mirror the assistant `alreadyPersisted` dedupe). No rollback on send failure.
- **FR-7 (citation chips).** `RunDestinationProps.markdownComponents` threads to `TcChat`; a pure `projectCitations` selector over `session.events` feeds a host `CitationsProvider`; in-chat `#cite-ord:`/`#cite:` anchors render `OrdinalCitationChip`.
- **FR-8 (run-phase pulse).** `RunHeader.status` is supplied a ● working/queued/cancelling chip derived from `session.runStatus`.
- **FR-9 (Sources tool-citation merge).** `applySourceEvent` folds `citation_made`/tool-call citations (today only `source_ingested`/`sources_ingested`); `RunWorkspaceRail`'s `onSelectSource`/`onJumpToChatSource`/`SourceRowComponent` seams are threaded from `RunDestination.rightRail`.
- **FR-10 (Phase-9 pagination).** `list_messages` returns the **most-recent window** (fetch DESC LIMIT, reverse → return **ASC**) and populates a real keyset `next_cursor`. Default (no cursor) is non-breaking for both consumers.
- **FR-11 (parity gate + retirement).** After all MUST-FIX + should-fix land, re-run the design-parity harness; flip `runCockpitWeb`; bake; then delete `ChatScreen`, `chatRunState`, dead neighbors, and the flag.

## 6. Non-functional requirements

- **NFR-1 (substrate-agnostic chat-surface).** No `window`/`fetch`/`localStorage`/`EventSource`, no `apps/*` imports anywhere in `packages/chat-surface`. Enforced by eslint `no-restricted-globals` (`packages/chat-surface/CLAUDE.md`). All new cockpit capability is a pure selector over `session.events` or a call through the injected Transport port / a host-supplied port.
- **NFR-2 (one event projection).** No new SSE subscription and no second `useEventProjector`. Every new consumer (citations, pulse, echo) is a pure selector over the single `session.events`.
- **NFR-3 (facade-only).** Apps call `backend-facade:8200` only, always through the injected `useTransport` port; never `backend:8100` / `ai-backend:8000`, never a raw `fetch`.
- **NFR-4 (no `apps/*→apps/*`).** The web `RunComposer`/`McpAuthPort` binder/conversation-nav **duplicate** desktop's pure logic and share only chat-surface component contracts + `@0x-copilot/api-types`. Zero import from `apps/desktop`.
- **NFR-5 (host-owned browser flows behind ports).** OAuth full-page redirect, sessionStorage stash, `/mcp/oauth/callback` route detection, and router/URL nav stay in `apps/frontend`, surfaced to the cockpit only as the `McpAuthPort` + `completedMcpAuthAction` prop + `conversationId`-in / `onConversationCreated`-out.
- **NFR-6 (flag-scoped reversibility).** Every intermediate landing executes only under `runCockpitWeb`; the flag-OFF ChatScreen path is provably untouched until the final gated deletion.
- **NFR-7 (ASC ordering contract).** The `/messages` response array stays ASC. Both consumers read `res.messages` in array order and ignore `next_cursor`/`has_more`; a DESC-in-response change is forbidden.
- **NFR-8 (Phase-9 is product persistence).** Pagination lives in ai-backend persistence + query shaping and a thin facade pass-through — never AI orchestration in the facade, never in chat-surface.

## 7. Architecture decisions

Each decision names its **boundary side**: `chat-surface-port` (substrate-agnostic package), `app-host` (apps/frontend), `facade`, or `ai-backend`.

- **AD-1 — Keystone routes turn-N through `ctx.dispatch`, not its own POST.** _(app-host + chat-surface-port seam)_ The web `RunComposer` `await dispatch(request)`s; `RunDestination` injects `dispatch = handleStartRun` which binds via `session.bindRun`. **Rationale:** a direct POST reintroduces the dropped-run-id / unbound-2nd-message bug and breaks the single-bind-sink + ensure-conversation-on-run invariants from the run-identity program.

- **AD-2 — Web `RunComposer` reuses the exact web helpers `RunEmptyComposer` already binds.** _(app-host)_ Onboarding attachment adapter / filePicker / plusMenu, `useOnboardingComposerModels` + `modelSelectionForId` (**live** `GET /v1/agent/models`, not desktop's static `desktopModelCatalog`), `ChatToolsTrigger`, `createComposerConnectorsPort(identity)`, `createFirstRunProviderKeysPort()`, `toReadableRunAttachments`; mount the **bare** `AssistantComposer` (not `OnboardingComposer`). **Rationale:** prevents drift between the two web composers; a wrong-module import would ship the static desktop catalog on web.

- **AD-3 — Extract `useWebRunComposerTools`.** _(app-host)_ Own `webSearchEnabled`/`activeConnectorIds → connectorScopes` in one shared hook consumed by `RunComposer` + `RunEmptyComposer`. **Rationale:** the web analog of desktop's single `useRunComposerBindings`; the two web composers cannot silently diverge on tools/model/connectors.

- **AD-4 — Cancel is cockpit-owned; **no** dedicated port or host callback.** _(chat-surface-port)_ `RunDestination` derives `running` from `session.runStatus`, extends the `renderComposer`/`TcChat.renderComposer` ctx with `running`+`onCancel`, and POSTs cancel through the injected Transport port (best-effort, mirroring `resolveApproval`). **Rationale:** cancel touches no `window`/`localStorage`/navigation, the facade route already stamps `requested_by_user_id` from the verified identity, and the cockpit binds from HEAD (no ChatScreen-style items scan) — a port here would be abstraction-for-its-own-sake. This one change lights up **both** substrates.

- **AD-5 — Cancel preserves ChatScreen's load-bearing ordering, re-expressed for the single-`boundRunId` model.** _(chat-surface-port)_ `onCancel` optimistically settles pending interaction parts **and** clears `boundRunId` in the same tick; head-resolution binds only when prev is null, so the run cannot re-bind and Stop cannot resurrect. **Rationale:** ChatScreen's comment (`ChatScreen.tsx:1180-1195`) proves omitting the settle → auto-rebind → Stop reappears.

- **AD-6 — MCP-OAuth launcher is a **typed `McpAuthPort`**, not a bare callback.** _(chat-surface-port type + app-host impl)_ chat-surface defines the port TYPE (`beginAuth(serverId)`, `skipAuth(serverId)`, `installFromCatalog(slug)`); the host implements it over `createComposerConnectorsPort` (web) / Electron IPC (desktop). **Rationale (adopting P2 over P1):** this is the _one_ genuinely substrate-divergent capability (full-page redirect + sessionStorage stash + callback route). A typed port makes the web/desktop contract explicit and symmetric, where a bare `onConnectMcp` callback leaves the desktop side implicit. The redirect/stash/callback never enter the package.

- **AD-7 — `mcp_auth` recognition + Connect card, distinct from `resolveApproval`.** _(chat-surface-port)_ Carry `approvalKind` through the `RunApproval → TcChatApproval` boundary; render an `mcp_auth` Connect variant; `mcp_discovery` cards resolve **locally** (a POST 404s). Connect invokes the `McpAuthPort`, never the `/decision` POST (mcp_auth resolves via a separate `mcp_auth_resolved` decision after OAuth returns). **Rationale:** overloading `/decision` would 404 on discovery and mis-resolve the auth gate.

- **AD-8 — MCP-OAuth resume needs **no** new chat-surface resume code.** _(app-host trigger)_ `useRunSession` already resumes cursor-based (`?after_sequence=<cursor>`, dedupes redelivered boundary events) and `bindRun` rebinds any run id. The host reuses App's callback effect unchanged, resolves run→conversation via `GET /v1/agent/runs/{run_id}` (the stash stores runId only), then `openConversation`. **Rationale:** the stream self-resumes; the substrate-specific parts (redirect/stash/route/nav) stay host-owned.

- **AD-9 — Optimistic echo lives in `useRunTranscript`, not `chatProjection`.** _(chat-surface-port)_ `chatProjection` deliberately never emits user turns; `useRunTranscript`'s own header names the echo the deferred follow-up. Feed `pendingUserMessage` at dispatch, dedupe against the re-seed, no rollback on failure. **Rationale:** matches ChatScreen semantics; pure, no substrate.

- **AD-10 — Conversation-nav is host-owned; `RunDestination` stays `conversationId`-in / `onConversationCreated`-out.** _(app-host)_ App owns `activeConversationId` (seeded from URL) + `openConversation`/`openNewRun`, mirroring `apps/desktop/renderer/bootstrap.tsx:212-265`; a run-destination URL slot round-trips `/run/<conversationId>`. **Rationale:** router/URL is substrate; the cockpit already resets on `conversationId` change — no chat-surface change beyond `RunRoute` wiring. No port invented (zero substrate divergence beyond the URL, which is app-local).

- **AD-11 — Citations: pure `projectCitations` selector + `markdownComponents` prop; host owns the chip node + provider.** _(chat-surface-port selector + app-host provider)_ Add `projectCitations` (peer of `projectSubagents`/`projectApprovals`) feeding a host `CitationsProvider`; thread `markdownComponents` into `TcChat`. **Rationale:** keeps the one-event-projection invariant; the chip renderer node + registry data source are nav-aware and host-owned.

- **AD-12 — Phase-9 returns the most-recent window **ASC** + keyset `next_cursor`.** _(ai-backend + facade)_ Persistence port gains a `before`-cursor param; all three adapters fetch DESC LIMIT then reverse → return ASC; query service populates `next_cursor`; facade `/messages` accepts + forwards the cursor. **Rationale (AD-12 is the load-bearing non-breaking decision):** both consumers read `res.messages` in array order and ignore the cursor, so returning the tail ASC is a strict improvement (you now see the latest turns) with zero consumer code change; DESC-in-response would silently reverse both transcripts.

- **AD-13 — Delete last, behind a re-audit + bake.** _(app-host)_ ChatScreen, `chatRunState`, dead neighbors, and the flag are removed only after the parity harness re-passes and the flag-ON bake completes, keeping the host OAuth pieces the cockpit's `McpAuthPort` now depends on and the shared pure reducers (`chatModel/*`). **Rationale:** deletion is irreversible; ChatScreen is the instant rollback until the bake signs off.

## 9. Risks

- **R1 — Cancel re-bind race (gating).** If `onCancel` omits the optimistic settle + `boundRunId` clear, head-resolution re-binds the run and Stop resurrects (a no-op click). _Mitigation:_ explicit test asserting no rebind after cancel (AD-5); time-boxed `cancelling` overlay reconciled by the `run_cancelled` SSE frame.
- **R2 — MCP-OAuth resume assumes run→conversation resolution (gating).** The sessionStorage stash carries `runId` only; `GET /v1/agent/runs/{run_id}` must map it, and if a backend restart lost the approval row the flow must degrade to landing on the conversation. _Mitigation:_ reuse ChatScreen's tolerant semantics; test the lost-row path; if the run already terminated during redirect, land on the completed transcript, not a hung stream.
- **R3 — Phase-9 ordering flip.** Returning DESC-in-response instead of most-recent-window-reversed-to-ASC silently reverses **both** transcripts. _Mitigation:_ explicit ASC-ordering test on all three adapters; changes which window is returned, never the response shape (AD-12, NFR-7). Handle tail windows referencing a `parentId` outside the window — `messagesToChatItems` already tolerates orphan parents (degrades to flat).
- **R4 — Web/desktop composer drift.** Two substrates duplicate the composer. _Mitigation:_ `useWebRunComposerTools` + reusing the exact web ports `RunEmptyComposer` binds (live model catalog, not desktop's static one) is mandatory (AD-2/AD-3).
