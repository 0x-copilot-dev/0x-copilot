# Safe-signing security review (3)

**Verdict:** needs-changes

### [high] HIDDEN CALLDATA / DELEGATECALL behind a benign card (defeats informed consent; fund-loss). RequestWalletSignatureInput (§3.1) accepts fully agent-controlled `data: str` and `operation: int`, but the card only surfaces To/Amount/Token/Chain/Safe/Nonce (§5), and SafeApprovalRecogniser is specced to project only to/value/token/chain/nonce — NOT `data`, NOT `operation`. operation=1 is DELEGATECALL = arbitrary code execution in the Safe's context (drains everything). A malicious agent sets summary='Send 250 USDC to treasury', token='USDC', value='250' while `data` is an ERC20 approve(attacker,MAX)/transfer(attacker,all) or `operation=1` delegatecall. The card reads benign; the wallet popup shows only raw EIP-712 SafeTx typed-data (to/value/data-hex/operation) which a normal user cannot interpret (contrast walletProof.ts, which today only ever personal_signs a human-readable SIWE string). The human clicks 'sign' on a lie. This is the primary fund-loss path and it requires no bypass of the wallet gate at all.

**Fix:** Make the card display fields authoritative from the DECODED/simulated transaction, not from agent-claimed fields. Always render `operation` prominently and hard-block or force a distinct, explicit high-friction confirmation for operation=DELEGATECALL and for any non-empty `data` to a non-allowlisted target. Decode `data` (ERC20/known-ABI) and show the real method+recipient+amount; when calldata cannot be decoded, treat it as unverified and fail closed (see next finding). Add the decoded-calldata rows to SafeApprovalRecogniser's params projection.

### [high] SIMULATION FAIL-OPEN. Risk §8 states: on simulate failure/uncertain, 'show a warning and still require the same two human confirmations — never auto-approve.' Because the card's to/value/token/summary are agent-authored (see AC#4 violation below) and the wallet popup is uninterpretable EIP-712, the renderer simulation asset-diff is the ONLY independent source of truth about what the tx does. Making it advisory means an attacker who can make simulation fail or be wrong — trivial: craft calldata that reverts under eth_call simulation but succeeds on execution, or exploit Tenderly/RPC downtime — gets the human to sign blind on a benign-looking card. Fail-open on the sole integrity control for an irreversible, value-bearing action is not acceptable for a signing gate.

**Fix:** Fail CLOSED on simulation failure for any tx that is value-bearing, has non-empty `data`, or uses operation=DELEGATECALL: disable the sign button, do not open the wallet popup. Only allow sign-on-unsimulated for provably zero-value, empty-data, operation=0 txs (and even then show it as unverified). Never present an agent-authored summary as the tx description when simulation is unavailable.

### [high] AC#4 NOT MET — card text is agent-authored. AC#4 claims 'The card shows the renderer's simulation, not agent-authored text,' but SafeProposalSummary/RequestWalletSignatureInput `to`, `value`, `token`, `summary` all originate from the builtin tool input = model output (modeled on ask_a_question.py L118-137 where the entire payload is agent-constructed). Only the asset-diff is renderer-derived. So the card TITLE and every params row can lie (token='USDC' while data moves a different asset; to=treasury display while the real recipient is in calldata). The design ships an agent-controlled review surface for an irreversible action.

**Fix:** Derive every displayed field (recipient, asset, amount, direction) from the renderer's own simulation/decoded calldata. The agent-supplied summary/token/value may only be shown labelled as 'agent claims', side-by-side with the decoded truth, and any disagreement between claimed and decoded must block signing or force explicit acknowledgement.

### [high] BOUND-SAFE NOT ENFORCED AT THE SIGN GATE (AC#5 gap). AC#5 says 'the agent cannot target an unbound Safe,' and the binding is enforced on the MCP read/propose tools via call_tool.py's McpPermissionPolicy.is_server_card_authorized re-check (L62,L83) + paused_connectors. But request_wallet_signature is a NATIVE builtin assembled directly in factory.py (L395, next to AskAQuestionTool) — native builtins do NOT pass through call_tool.py authorization at all. `safe_address` is agent-supplied on the SIGN gate, and the design's only sign-time checks (§4 step 7) are: re-derive hash, chain ∈ allowlist, ownerAddress is a Safe owner — NONE of which verify safe_address ∈ the user's bound Safes. A malicious agent can raise a wallet_signature for any Safe the user co-owns (or any Safe at all, then rely on ownerAddress matching), moving the human's signature onto a Safe/tx they never intended to operate.

**Fix:** Enforce `safe_address ∈ {this user's explicitly bound Safes}` SERVER-SIDE in the facade /v1/wallet/safe/{safe}/simulate and .../transactions endpoints (and re-check chain), not in the renderer and not via the MCP layer that the builtin bypasses. Persist bindings per-user and reject the submit if the target isn't bound.

### [medium] AC#2 'two independent human confirmations' is really ONE cryptographic gate. The signature is produced out-of-band in resolveApproval (§4 step 7) BEFORE the approval decision is POSTed (step 8), and the approval-decision endpoint (approval_coordinator.record_approval_decision) never verifies that a signature occurred. So the langgraph interrupt/approval card gates the AGENT'S RESUME, not the signature — the only thing actually gating the signature is the wallet popup. The claim 'gated per-call by the approval interrupt' overstates the guarantee: approve the card and sign are glued only by client code, and the card confirmation is a UI step, not a cryptographic one. Defense-in-depth collapses to the wallet popup, whose EIP-712 display is uninterpretable (findings above).

**Fix:** Stop describing the approval card as a security gate for the signature. Bind the two: the facade submit endpoint should require a server-issued, single-use challenge that is only minted when the approval row is PENDING for this exact (approval_id, safe_tx_hash), so a signature cannot be submitted for a tx that isn't currently sitting at a live per-call gate. Record the signature-submitted fact server-side and only allow decision=approved after it.

### [medium] safeTxHash re-derivation is presented as a security control but is tautological against a malicious agent. AC#4/§4-step-7 lean on 'renderer re-derives safeTxHash from canonical fields and refuses (hash_mismatch).' But the renderer re-derives from the SAME agent-supplied fields (to/value/data/operation/nonce). A consistent malicious agent supplies matching fields+hash and the check passes; it only catches transport corruption or a buggy/inconsistent agent — never a malicious one. It provides false assurance for the fields that actually matter (`data`,`operation`).

**Fix:** Document the hash-match as an integrity/consistency check only, not a safety control, and do not count it toward the safety invariant. The real defenses must be decoded-calldata display + fail-closed simulation + bound-Safe enforcement (above).

### [medium] Signature-in-`answer` threading is guarded only by convention. record_approval_decision writes client-supplied `request.answer` into the persisted ApprovalDecisionRecord (L338-349) AND into RuntimeApprovalResolvedCommand.answer (L382-391), which flows into graph state and is read back by the tool's \_resume_result (modeled on ask_a_question.py L173-181, agent-visible on resume). `answer` is free-form and unbounded. Risk §8 says 'never thread the raw signature,' but nothing ENFORCES it — a buggy/compromised/future renderer could place the signature (or any secret) in `answer` and it enters graph state + the persisted decision record. AC#1 ('signature never enters agent/graph/worker/approval-decision state') is therefore not enforced, only requested.

**Fix:** For approval_kind=wallet_signature, have the coordinator (or the decision-request validator) REJECT any `answer` not in a fixed status enum {signed,rejected,chain_not_allowed,not_owner,hash_mismatch,nonce_changed} and cap length. Enforce server-side so no free-form string can ride the resume for a signing approval.

### [medium] propose is `session`-approved and shares one client module with the submit path. desktop_profiles.yaml specs all Safe tools (incl. propose) as approval:session, so after first consent every subsequent propose runs SILENTLY with no per-call gate. §2 lists safe_txservice_client.py as doing BOTH read-queue AND 'create-multisig-tx-with-signature.' The only thing separating side-effect-free propose from queue-writing submit is code discipline inside one module reachable under session auth. A coding error (or prompt-injected argument that flips a flag) could let a session-approved propose write to the Safe queue. AC#3 ('propose has ZERO side effects') is an assertion about not-yet-written code with no architectural boundary behind it.

**Fix:** Physically split the read/propose client from the submit client into separate modules/credentials. Make the create-multisig-tx path reachable ONLY from the facade endpoint that requires a signature body, never importable/callable from any MCP tool. Add the promised no-broadcast/no-submit grep+runtime assertion as a CI gate on the propose module.

### [medium] Desktop safe-sign.html spoofed-deeplink / URL-passed tx. The design models safe-sign.html on wallet-login.ts, which passes handoff/state via URL query (wallet-login.ts L95-98). If safe-sign.html accepts the safeTxHash/EIP-712 typed-data via URL params (as the login page accepts handoff), a malicious local process or deep link could present a spoofed tx to the user's system-browser wallet, outside the agent/approval flow entirely.

**Fix:** safe-sign.html must NEVER accept caller-supplied typed-data/safeTxHash. It must take only an opaque proposal_ref + loopback state, FETCH the canonical EIP-712 doc from the facade itself (authenticated), verify the loopback state round-trips, and refuse non-loopback handoff targets exactly as wallet.html does. Add a test that URL-injected typed-data is ignored.

### [low] Kind-integrity depends on dispatch ordering. native_interrupt_payloads (stream_events.py L633-662) tries auth → ask_a_question → then FALLS THROUGH to native_tool_approval_payloads which stamps approval_kind='mcp_tool' (L831-832). If \_native_wallet_signature_payload is not inserted before that fallthrough, or the recogniser mis-tags the kind, a wallet_signature interrupt could be projected as mcp_tool — which IS forwardable (APPROVAL_FORWARDABLE_KINDS) and routes through the generic tool-approval resume, losing the per-call/non-forwardable/reversible=no guarantees.

**Fix:** Add \_native_wallet_signature_payload BEFORE the tool-approval fallthrough (as specced) AND add regression tests: (a) a wallet_signature interrupt never yields approval_kind=mcp_tool, (b) a forward decision on a wallet_signature approval is rejected 422, (c) reversible=no so no undo window is offered.

### [info] Confirmed-holding controls (do not regress these). No server-held wallet key exists (only AuditChainSigner + webhook signers). Auto-resume paths never approve: sweeper enqueues REJECTED only (approval_expiry_sweeper.py L209-224); the two approving callers are identity-scoped to the assigned human (approval_coordinator.py L310). wallet_signature is correctly outside APPROVAL_FORWARDABLE_KINDS (L70-75). The literal invariant 'no signature without a human wallet action' is not bypassable by model/tool output as designed — the failures above are deceptive-consent and integrity gaps, not a keyless-signing bypass.

**Fix:** Keep the no-server-key and rejection-only-sweeper properties under test. Add a CI assertion that no wallet private key or eth_signTypedData/eth_sign path is ever introduced server-side, and that no new enqueue_approval_resolved caller can emit decision=approved without the identity-scope check.
