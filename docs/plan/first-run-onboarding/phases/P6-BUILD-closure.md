## 2. The `[low]` classifier fix, done right (cross-cutting — lands once)

**File:** `services/ai-backend/src/runtime_worker/stream_events.py`, `_connector_action_is_read_only` (**L909–918** at HEAD). Owned here so it lands **exactly once**, in **P6-BUILD-Sheets.md PR-S2** (Sheets ships first, §4). The Safe track must **not** re-edit this function.

### 2.1 The bug, precisely

Verified at HEAD:

```python
# services/ai-backend/src/runtime_worker/stream_events.py:909-918
@classmethod
def _connector_action_is_read_only(cls, tool_name: str) -> bool:
    """Return ``True`` when the tool name contains no write-operation terms."""
    normalized = tool_name.lower()
    if any(
        term in normalized
        for term in ("create", "post", "send", "update", "delete", "write")
    ):
        return False
    return True
```

It is a **denylist that defaults to read**: any name not containing one of six write-verbs is labeled read-only. Consequences confirmed at HEAD:

- `copy_file` (`desktop_profiles.yaml:114-117`, gdrive, `product_scope: write`) contains no write-verb ⇒ `_connector_action_is_read_only("copy_file") == True` ⇒ **a write tool renders a read consent label today.**
- The Sheets writes `write_append_values`/`clear_values` only classify correctly because PRD-P6b §3.2 pre-named them with write-verbs — a naming _belt_, not a fix. A real MCP server that ships `append_values`/`clear_values` (no write-verb) would render a **read** label over a mutating call.

### 2.2 The fix — invert to a write-wins allowlist (fail-safe to write)

A pure allowlist (`read/get/list/… ⇒ read`, else write) is the review's suggestion, but a tool like `get_or_create` would slip through as read. The no-bandaid version is **write-wins**: a tool is read-only iff it names a read op **and no write op**; an unrecognised verb fails safe to write.

```python
# services/ai-backend/src/runtime_worker/stream_events.py — replaces L909-918
# Consent-CLARITY heuristic ONLY. This never gates a tool call: the runtime
# interrupts on EVERY call_mcp_tool, and the real per-call gate is the
# profile's ``approval: per_call`` + graph HITL (PRD-P6b §4.3). This label
# only chooses the consent card's category/risk. Write-terms win, and an
# unrecognised verb fails safe to "write" so a new connector's mutating tool
# can never render a low-risk read consent (P6-plan-review [low]).
_WRITE_ACTION_TERMS: tuple[str, ...] = (
    "create", "post", "send", "update", "delete", "write",
    "append", "clear", "insert", "set", "remove",
    "copy", "move", "patch", "put", "share", "add", "rename",
    "import", "revoke", "grant",
)
_READ_ACTION_TERMS: tuple[str, ...] = (
    "read", "get", "fetch", "list", "search", "find", "view", "describe", "query",
)

@classmethod
def _connector_action_is_read_only(cls, tool_name: str) -> bool:
    """Consent-CLARITY label (NOT the write gate).

    Read-only only when the name clearly reads and never writes; write-terms
    win and unknown verbs fail safe to write. Display/label only —
    enforcement rides the profile ``approval`` policy + graph HITL, which
    interrupts on every ``call_mcp_tool`` regardless of this result.
    """
    normalized = tool_name.lower()
    if any(term in normalized for term in cls._WRITE_ACTION_TERMS):
        return False
    return any(term in normalized for term in cls._READ_ACTION_TERMS)
```

Result matrix (the PR-S2 unit table): `copy_file`→write, `append_values`→write, `clear_values`→write, `update_values`→write, `create_spreadsheet`→write, `batch_update_spreadsheet`→write, `get_values`→read, `batch_get_values`→read, `search_spreadsheets`→read, `get_spreadsheet`→read, and an unknown verb `frobnicate_sheet`→**write** (fail-safe).

### 2.3 Two honest caveats the build docs must carry

1. **Fail-safe-to-write over-warns for verbless read tools.** Under the inversion, a read tool whose name carries **no** read-verb (e.g. a Safe read named `owners`/`threshold`/`nonce`/`balances`/`pending`) flips from read to **write** — a louder consent, never a security miss. Mitigation, and the reason this is coordinated with the Safe track: the adopted read-only Safe MCP (`5ajaki/safe-mcp-server`, Decision C) read tools must be surfaced with **read-verb names** (`get_owners`, `get_threshold`, `get_nonce`, `get_balances`, `list_pending`) so they label as reads. Same discipline for any adopted Sheets MCP. This is a naming contract on the connector profiles, not a second code change.
2. **This is a label, not a gate — say so, once.** Both build docs must keep PRD-P6b §4.3's honest framing: reads currently also prompt (as read-category cards); the profile `approval: session|per_call` is the declared contract + scope gate; the per-call gate is graph-level HITL. Do **not** write "reads never prompt" and do **not** gate any interrupt decision on `read_only`. PR-S2 adds a regression asserting the interrupt fires for a `read_only=true` tool.

### 2.4 Sibling heuristic — leave alone or align

`_connector_action_name` (**L894–907**, same file) is a parallel label producing `search/read/modify/action` and carries the same denylist gap (`create/post/send/update/delete`, no `append/copy/…`). It feeds only a human-readable action word, not the read/write category. Optional: fold it onto the same `_WRITE_ACTION_TERMS`/`_READ_ACTION_TERMS` sets for consistency. **Not required** for P6 — flag it as a one-line follow-up in PR-S2, don't expand PR-S2's scope to chase it.
