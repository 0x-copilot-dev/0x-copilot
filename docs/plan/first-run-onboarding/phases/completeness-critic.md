# Completeness critic — cross-phase

## Gaps

- P3 CSV attach is a HARD BLOCKER, not a 'verify-at-impl'. The web attachment adapter's accept list `OFFICE_ACCEPT` (apps/frontend/src/features/chat/runtime/attachments/file.ts:15-23) is pdf/doc/docx/xls/xlsx/ppt/pptx ONLY — `.csv`/`text/csv` is absent. The 'Explain a CSV' starter chip pre-attaches `airdrop-claims.csv`, which the current adapter will reject at pre-attach. P3 must extend the accept list on BOTH host adapters (web `file.ts` + the desktop `createDesktopAttachmentAdapter` used in apps/desktop/renderer/composer/RunComposer.tsx:40) or route CSV through the text/composite adapter. Not optional.
- P3's `airdrop-claims.csv` fixture does not exist anywhere in the repo (verified: `find` returns nothing). P3 must author the CSV contents/schema AND supply two host resolutions to turn it into a File: web via a bundled-asset URL, desktop via a bundled resource/IPC. Neither the asset nor its two binder resolutions are specified; the chip is inert without them.
- The shared composer's `ModelPicker` (packages/chat-surface/src/composer/ModelPicker.tsx:34-36) is HARDCODED to three Claude models (Opus 4.7 / Sonnet 4.6 / Haiku 4.5). The FTUE design requires a model popover with Local (Qwen 3 4B) + BYOK provider groups sourced from `/v1/agent/models`. No phase (P1/P2/P3) explicitly owns making the composer model list catalog-driven — each assumes the host injects a catalog-backed `models` prop, but if the mounted composer falls back to ModelPicker's internal list it shows the wrong models. Assign an owner (likely P3, or a shared pre-req) and confirm AssistantComposer takes an injected models prop rather than ModelPicker's constant.
- P6a's 'new approval_kind modeled byte-for-byte on ask_a_question, machinery inherited for free' understates the work. `ask_a_question` is special-cased across 5 non-test files (services/ai-backend/src/runtime_worker/stream_tools.py, stream_events.py, handlers/approval.py, runtime_api/schemas/events.py:701-702, runtime_api/schemas/common.py) and the ApprovalKind constant is declared in at least two classes (agent_runtime/api/constants.py:246 and :232) plus schemas/common.py. Adding `wallet_signature` as an interrupt-raising kind is therefore NOT purely additive — the interrupt→approval normalizer/projector/recognizer each hardcode `ask_a_question` and must be taught the new kind or the interrupt won't project to a consent card.
- P0's proposed `AuthService.accountKey(workspaceId)` must resolve the session via `#loadSession` (async; apps/desktop/main/auth/index.ts:498-510), NOT a bare synchronous `#cache` read. The `#cache` is only warm after signIn/getSession runs; a disk-only session (post-restart, cold path) lives in SecretStorage until `#loadSession` populates the cache. If `accountKey` is implemented as a sync cache getter it can return undefined and mis-key (or fail to write) the first-run flag. Also note RendererSession deliberately omits `sub` (index.ts:106-111), so the accessor is genuinely required — that part of P0 is correct.
- P4 (wallet chip + tools-popover connector rows + per-run web-search toggle) has NO PRD in this set, yet P1, P3, P6a, and P6b all depend on it. P1 ships an empty `walletChipSlot`; P3's ack line 'tools — {web search|none}' cannot be truthfully wired because `WebSearchToolRegistry` is always-on today (per README §7.4 / runtime_worker/dependencies.py:53) with no per-run flag; the wallet chip needs `/v1/me/profile` (SIWE address) but STATUS.md notes the server exposes no `truncated_display_address` so the chip must truncate client-side. Schedule P4 as a real phase or fold its pieces into P3/P6.
- The shipped `ToolPicker` (packages/chat-surface/src/composer/ToolPicker.tsx:40-55) renders only a flat `ComposerToolDescriptor` toggle list — it has NO connector-install / 1-click-connect / per-chat-scope / web-search-toggle plumbing. So P6a's claim that 'the FTUE Tools-popover 1-click Safe{Wallet} works with zero new UI plumbing' and P6b's 'Sheets row comes from GET /v1/connectors/desktop/catalog' are false until the connector-aware tools popover is built (that is P4). Both P6 PRDs must list the tools-popover UI (P4) as an explicit dependency.
- P6b's write-scope escalation is correctly located (verified 4 sites: services/backend/.../connectors/oauth_coordinator.py:153 and :298-304 which never adds `write`, desktop_routes.py:91, packages/api-types/src/connectors-desktop.ts:66), and `profile_catalog.py:65,71` already models `write` so the profile can declare it while the request path caps at read|draft. But the PRD omits the ordering constraint: the backend `Literal['read','draft']` in desktop_routes.py:91 must be WIDENED to include 'write' BEFORE the desktop/facade sends `requested_product_scope:'write'`, or Pydantic 422s the request. Backend widening must precede the client change.

Verified anchors: smallest design-system token is `--font-size-2xs: 0.7rem` (11.2px) — there is NO `--font-size-3xs`, so the SPEC's 9/9.5/10/10.5px mono labels must round up to 2xs (P1/P2/P3 open question is real); `--radius-lg: 0.75rem` (12px) matches `.fr-gcard` radius 12; `--color-accent:#5fb2ec` (sky), `--color-accent-contrast:#08131d`, `--color-success:#57c785` (jade) all present in packages/design-system/src/styles.css. - The two-step run-create is shared by P2's deferred-launch seam and P3's immediate path and must be defined once. Verified contract: POST /v1/agent/conversations then POST /v1/agent/runs with CreateRunRequest (packages/api-types/src/index.ts:1392) fields {conversation_id, user_input, model?:ModelSelectionRequest(provider/model_name at :1280), attachments?}. - `SuggestionChips` (P3) and `WalletChip` (P1 slot / P4 fill) are net-new shared components (verified: no `SuggestionChips` anywhere; `walletChipSlot` is only a P1 seam). - The chat-surface substrate eslint boundary governs every onboarding file in every phase. Verified eslint.config.js bans window/document/history/navigator/location/localStorage/sessionStorage/fetch/EventSource/XMLHttpRequest/WebSocket as globals and blocks @0x-copilot/frontend + apps/\* imports. Same rule forces every SSE/fetch/IPC/keychain touch across P0-P3 into a host-injected port impl.
- Identity namespacing is a shared, deliberately-asymmetric seam. Desktop keys the first-run flag by hashed claims.sub derived in main (RendererSession omits sub — index.ts:106-111; cache keyed by workspaceId=DEFAULT_WORKSPACE_ID); web keys by org_id+user_id via CopilotApp's injected identity (apps/frontend/src/app/App.tsx:459 passes identity.orgId/userId; KeyValueStore hoisted at App.tsx:339). The two hosts key by DIFFERENT identity tokens, so the same human across web+desktop gets two independent first-run flags. Acceptable, but document it so the 'returning user bypass' semantics are consistent and P7 tests assert per-host, not cross-host.

## Sequencing

- P0 MUST land first (hard serialize). It owns the canonical `FirstRunStore`/`FirstRunGate` port, the desktop IPC channels (following the verified CAPABILITY_CHANNELS pattern in apps/desktop/main/capabilities/channels.ts + ipc/handlers.ts + preload allowlist), the bootstrap.tsx seam (verified at apps/desktop/renderer/bootstrap.tsx:94-104, between the SignInGate render-prop and ChatShellForSession), and the web CopilotApp binding. P1/P2/P3 all import the FirstRunStore port. Do NOT rely on P1's 'interim FirstRunStore' hedge — land P0 first so P1-P3 import one canonical port.
- P1 serializes after P0, and before P2+P3. P1 creates the FirstRunSurface shell + 3-state machine + Gate + KeyForm + onboarding.css, and injects placeholder slots for the dl/ready composer body (P3) and the local card (P2). P2 (GateLocalCard) and P3 (composer+ack) both consume P1's scaffold + injected-slot contract, so they cannot start until P1's slot interface exists.
- P2 and P3 can PARALLELIZE after P1, but share the `modelReady` seam and must agree its contract up front. P2 owns the local-model download hook that flips modelReady at pct===100 (engine.status==='ready'); P3's useFirstRunLaunch consumes it to fire the deferred two-step create; BYOK-connect (P1) also flips modelReady. Define this signal's interface in P1 (or the first commit of whichever lands first) or the two phases will diverge.
- P2's one-line packaged-supervisor edit is a hard prerequisite and can/should land FIRST (even before P1). Verified: apps/desktop/main/services/service-env.ts ai-backend case (L226-283) sets NO RUNTIME_ENABLE_LOCAL_MODELS; the staging tool tools/desktop-runtime/run-local.mjs:713 sets it but the packaged build does not; ai-backend default is enable_local_models=False (settings.py:141). Without this edit the local card is dead (/status→enabled:false) in real builds and P7's local-journey live test fails. Land it as P2's first commit or a standalone pre-req; keep the Pydantic default False (cloud fail-safe).
- P4 (wallet chip + tools-popover connector rows + per-run web-search toggle) is an UNSCHEDULED prerequisite that must be inserted after P3 (so the composer exists) and before P6 (so the tools popover can host the Safe/Sheets 1-click rows). Verified the shipped ToolPicker has no connector/1-click/scope/web-search plumbing. Without P4, P6a/P6b register connectors in the backend but have no FTUE entry point, and P3's ack tools line + P1's wallet slot stay unwired. Either schedule P4 explicitly or fold the tools-popover connector UI into P6.
- P6a and P6b parallelize on their YAML/profile/UI parts but their ai-backend RUNTIME edits must be serialized. Both add a slug to services/backend/src/backend_app/connectors/catalog.yaml + a profile to desktop_profiles.yaml (verified paths). But P6b changes the shared classifier write-term list at runtime_worker/stream_events.py:904/915 (verified 'append' is absent — the Sheets naming gotcha is real) and P6a adds a wallet_signature approval_kind touching the same stream_events.py + approval projection files. Run in parallel they collide on shared runtime files — coordinate or serialize the two ai-backend runtime edits.
- P6 as a whole stays LAST among build phases and is gated on the security/product sign-offs enumerated in STATUS.md and both PRDs' open questions (Safe MCP host location, signing UX, tx simulation provider, execution ceiling; Sheets endpoint existence, operator OAuth client ownership, preview-vs-stable release stage, reads-session-scoping). Do not start P6 build until those land — it is correctly sequenced last before verification.
- P7 (E2E parity + verification) is strictly last and depends on all prior phases PLUS the P2 supervisor env edit — the local-first journey cannot be verified on the live stack without RUNTIME_ENABLE_LOCAL_MODELS set in the packaged supervisor. Sequence: P0 → P1 → (P2 ∥ P3) → P4 → (P6a ∥ P6b, serialized on ai-backend runtime files, gated on sign-off) → P7.
