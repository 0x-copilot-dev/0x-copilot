# Safe-signing security review (1)

**Verdict:** needs-changes

### [critical] Display/calldata divergence — the card and the wallet popup both show agent-authored intent, never the decoded effect of `data`. RequestWalletSignatureInput (§3.1) carries DISPLAY fields (`to`,`value`,`token`,`summary`) SEPARATE from the signed `data` calldata. For every ERC-20 / contract call (USDC transfer, `approve`, DeFi) the real destination+amount live in `data`, where `to`=token contract and `value`=0. The SafeApprovalRecogniser the design adds projects agent-supplied args verbatim with zero decoding — the existing recognisers only `_stringify` (approval_recognisers.py:52-72, 96-107), stream_events passes the interrupt payload through as `{**payload,...}` untouched (stream_events.py:747-757), and ApprovalCard renders title/params exactly as given (ApprovalCard.tsx:67-93). The wallet popup's 2nd confirmation shows EIP-712 SafeTx typed-data whose `data` field is raw hex — no calldata decoding exists anywhere in the app (grep confirms only walletProof.ts personal_sign). So an injected tool result sets summary='Send 250 USDC to treasury.eth', to=USDC, token='USDC', value='250' while data=transfer(0xATTACKER, 1e12) or approve(0xATTACKER, MAX_UINT). The safeTxHash re-derivation (AC#4) confirms hash==fields but NEVER confirms fields==human-readable claim, so the human approves a benign-looking card and signs a drain. This directly defeats the stated goal ('hidden destination/amount').

**Fix:** The card's To/Amount/Token rows MUST be derived by the renderer/facade from decoded `data` (ABI-decode transfer/approve/known selectors), never from agent display fields; treat `summary`/`token`/display-`to` as untrusted labels shown only in a de-emphasized 'assistant note' zone. Refuse to sign (not warn) any tx whose `data` cannot be decoded to a recognized, displayed effect. Note APPROVAL_MAX_PARAMS=6 (approvals.py:280) is already fully consumed by To/Amount/Token/Chain/Safe/Nonce — there is no row left for a decoded-effect line; the params model must change, not just be reused.

### [high] The only independent truth (simulation asset-diff) is downgraded to advisory. Risks §'Simulation provider trust' states: on simulate failure/uncertain, 'show a warning and still require the same two human confirmations — never auto-approve.' An attacker suppresses the asset-diff trivially — craft calldata that reverts under eth_call, target a token/contract the simulator doesn't support, or grief the provider into a timeout — then the renderer shows 'simulation unavailable' and the human, primed by the agent's persuasive `summary`, signs. Because the card+wallet already fail to decode `data` (see critical), the simulation is the sole line of defense against F1, and the design deliberately makes it non-blocking.

**Fix:** Make a successful, decoded simulation MANDATORY-GATING (disable the sign button) whenever `data` is non-empty, `operation`!=0, or the decoded action is a token approve/transfer. Fail closed on simulation failure for anything but a bare native-value transfer with empty calldata. Only a positive, renderer-rendered asset-diff should enable signing.

### [high] DELEGATECALL (operation=1) is not specially gated. RequestWalletSignatureInput exposes `operation:int` (§3.1) and it flows through as a bare numeric params row. A Safe delegatecall executes arbitrary code in the Safe's context — it can swap owners, drop threshold to 1, or sweep every asset in a single tx, and a naive eth_call asset-diff of the delegatecall can render empty/benign. A human will not understand that 'Operation: 1' means 'hand full control of the multisig to an arbitrary contract.' The design never mentions delegatecall.

**Fix:** Block operation==1 by default; permit only delegatecalls whose target is an explicit, hard-coded allowlist of known Safe libraries (e.g. MultiSendCallOnly), and require a distinct, typed human confirmation with an explicit 'this can change owners/threshold' warning. Never render delegatecall as a plain numeric row.

### [high] The safeTxHash re-derivation is sold as the injection kill-switch but is circular. AC#4 and §4.7 claim the renderer 're-derives safeTxHash from canonical fields and blocks on mismatch (hash_mismatch)'. But the renderer/facade re-derive the hash from the SAME agent-supplied `to/value/data/operation/nonce` (§3.1). Re-hashing the agent's own fields only proves the agent didn't lie about the hash of its inputs — it provides zero protection when malicious intent is inside `data` (F1). The test-list item 'tampered safeTxHash caught by re-derivation' tests a case an attacker would never need to trigger. The design overstates hash-equality as consent integrity.

**Fix:** Reframe: hash re-derivation only guarantees the wallet signs exactly the fields the card projected — it is not the kill-switch. The real integrity guarantees are decoded-calldata display (F1) + mandatory simulation (F2). Update AC#4 and the injection test to assert the decoded effect shown equals the effect simulated, not merely hash equality.

### [medium] Chain identity is not cross-checked across the four sources, and the login allowlist is reused for value signing. The design validates only `chainId ∈ SIWE_ALLOWED_CHAIN_IDS` and displays a chain name, but never specifies that the EIP-712 domain.chainId (built by the facade), the wallet's live eth_chainId, the validated chainId, and the displayed chain are all asserted equal before signing. The existing primitive reads eth_chainId only for SIWE (walletProof.ts:33-39) and has no typed-data domain check. A Safe address commonly exists on multiple allowlisted chains; the human can be shown 'Base' (cheap) while the domain/nonce commit a signature valid on Ethereum where the Safe holds real funds. Worse, SIWE_ALLOWED_CHAIN_IDS (default 1,8453,42161,4663 — Robinhood Chain, siwe.py:85) is a LOGIN allowlist being conflated with a high-value SIGNING allowlist.

**Fix:** Before signing, assert wallet eth_chainId == typed-data domain.chainId == validated chainId == displayed chain; abort otherwise. Introduce a separate, explicitly-scoped signing chain allowlist (do not reuse the SIWE login list) so login posture and treasury-signing posture can diverge.

### [medium] The 'bound Safe' constraint is aspirational — no enforcement exists. AC#5 and §4.2 assert the agent 'cannot target an unbound Safe', but grep finds NO safe_binding store/enforcement anywhere in the repo, and connector auth_mode is McpAuthMode.NONE. RequestWalletSignatureTool takes `safe_address` straight from agent input and 'holds no key, just pauses' — no server-side check that safe_address ∈ this user's bound Safes is specified in the tool or coordinator. The only real constraint is the renderer's `not_owner` check, which merely requires the signer be AN owner. A user who owns both a personal and a treasury Safe can have the agent silently redirect a signature to the wrong owned Safe, or to one whose nonce/queue an attacker pre-seeded.

**Fix:** Add a concrete per-user bound-Safe store, captured at CONNECT, and enforce `safe_address ∈ bound_safes` server-side in the tool (or worker) BEFORE the interrupt is raised — not only owner-membership in the renderer. Add a tenant-isolation test that an unbound safe_address is rejected.

### [medium] WalletSignatureApprovalMetadata inherits extra='allow' (McpApprovalMetadata, approvals.py:294), and the design constructs it from the agent-authored `safe` block. `reversible` is a real field on that model, and the undo gate reads it as a plain metadata flag (`approval.metadata.get('reversible')=='yes'`, approval_coordinator.py:478,520). If the tool/worker builds the metadata via `WalletSignatureApprovalMetadata(**agent_safe_block)`, an agent that injects `reversible:'yes'` (or overrides `category`/`vendor`) flips the intended NO and grants a spurious undo window / mis-frames the card — defeating AC#6 (reversible=no). extra='allow' also silently persists arbitrary agent keys into audit metadata.

**Fix:** Construct the metadata server-side from an allowlisted set of fields only; pin `reversible`, `category`, `reason_code`, `vendor` as constants the agent cannot override; do not spread the raw agent payload into an extra='allow' model. Add a test that an agent-supplied `reversible:'yes'` on a wallet_signature does not produce an undo window.

### [medium] Approval fatigue has no structural brake. The design correctly forbids 'approve-all' (§8) and gives each request its own card, but an injected tool stream can emit unbounded request_wallet_signature calls, each with an attacker-authored persuasive summary ('nonce changed, re-sign', 'gas too low, re-sign'). There is no per-run/per-session cap, no cooldown, and no cumulative-value display, so the user is trained to click through — and the card header title is the agent's summary (ApprovalCard title, SPEC 'Header title = the human summary'), which frames each decision in the attacker's words.

**Fix:** Add a per-run/per-session signature-request cap + cooldown; show a cumulative-value-signed-this-session counter; visually flag repeated identical destinations; and compose the card TITLE from decoded fields (e.g. 'Sign: transfer 1,000,000 USDC → 0xAtt on Ethereum'), relegating the agent summary to a labeled note.

### [low] The Safe Transaction Service response is trusted as the confirmation status relayed back to the agent, and idempotency/nonce-race handling is design-only. §4.9 returns `{status, confirmations}` from the tx-service into agent context; a compromised/spoofed tx-service reply ('2/2, ready to execute') could mislead the agent's narration. Blast radius is limited because on-chain execution is out of scope and the human still executes in the Safe app, but the agent's report is not independently verified. Nonce-race + idempotency are described as 'warn/re-check' with no concrete enforcement.

**Fix:** Treat tx-service confirmation counts as advisory in agent narration ('signature submitted' not 'N/M confirmed' unless independently read back); enforce idempotency on (safe, safeTxHash, signature) at the facade; and re-read the on-chain/tx-service nonce at submit time, aborting on drift rather than signing a stale tx.
