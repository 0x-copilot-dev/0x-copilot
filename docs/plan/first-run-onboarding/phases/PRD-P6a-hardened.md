# P6a (HARDENED) — Safe{Wallet} propose-only signing

**Status: design, BUILD-GATED.** Supersedes the parts of `PRD-P6a.md` the three security reviews (`security-review-safe-{1,2,3}.md`, all `needs-changes`: 1 critical + 8 high + 12 medium/low) refuted. Distilled from the design-hardening agent; expand to full detail at P6 sign-off.

## Corrected framing

The literal keyless-signing invariant already holds and is **not** bypassable by model output (review-3 `[info]`): no server-held wallet key exists, the sweeper enqueues `REJECTED` only, approving callers are identity-scoped. **Every finding is a deceptive-consent or secret-leak gap, not a keyless-signing bypass.** So the safety story moves OFF `safeTxHash` re-derivation (all three reviews called it tautological — it re-hashes the agent's own fields) and ONTO three real controls:

- **A. Decoded-calldata authority** — the consent card + wallet-review rows are built from server-decoded `data` + simulation, never from agent-authored display fields. A benign-looking card for malicious `data` must be impossible.
- **B. Fail-closed mandatory simulation** — no positive, renderer-rendered asset-diff ⇒ sign button disabled ⇒ wallet popup never opens (except a bare native-value transfer with empty `data` + `operation==0`, shown "unverified").
- **C. Server-side bound-Safe + chain enforcement at the facade** — because `request_wallet_signature` is a NATIVE builtin (`factory.py:487`) that bypasses the MCP auth re-check (`call_tool.py:83-92`) and `paused_connectors`, binding is enforced where signing physically must pass: the facade `/simulate` + `/transactions` endpoints, on server-derived identity, against a per-user bound-Safe store.

## Residual sign-off questions (block P6 build)

1. Safe MCP host location (new `services/safe-mcp` recommended — physically splits read/propose vs submit) vs backend-hosted.
2. Bound-Safe store location (MCP record metadata vs new `safe_bindings` vs connector scope) — enforcement fixed at the facade; storage open.
3. Signing UX / desktop handoff shape (POST-body-to-loopback vs one-time-token-to-facade).
4. Simulation provider (self-hosted `eth_call`/anvil-fork vs Tenderly) — now safety-load-bearing.
5. Separate `SAFE_SIGNING_ALLOWED_CHAIN_IDS` + its default membership.
6. Delegatecall library allowlist set, or block delegatecall entirely in v1.
7. Execution ceiling — agent stops at "signature added to the multisig queue"; on-chain EXECUTION is a separate, later, explicitly-gated action, out of P6a.

CI invariants to keep under test: no server-side wallet key / `eth_sign*` path ever; sweeper stays rejection-only; no new `enqueue_approval_resolved` caller emits `approved` without the identity-scope check.
