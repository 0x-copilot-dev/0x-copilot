# Safe-signing security review (2)

**Verdict:** needs-changes

### [high] Desktop signing path leaks the SafeTx signature into the system browser's history and the loopback request line. The design says to build safe-sign.ts/safe-sign.html by 'modeling main/auth/wallet-login.ts' and to have the page 'run EIP-1193 in the system browser and relay the signature to the loopback.' The existing mechanism it copies relays cryptographic artifacts via the redirect URL query string: awaitLoopbackWalletProof/parseWalletProofRedirect reads the signature from url.searchParams (apps/desktop/main/auth/loopback-server.ts:373), and the sign-in variant relays the bearer token identically (parseHandoffRedirect, loopback-server.ts:317). A SafeTx owner-confirmation signature placed in http://127.0.0.1:<port>/...?signature=0x... is persisted to the OS browser history and appears in any URL-level access log. That violates the design's own AC-1 ('the raw signature ... never enters logs') and §8's 'never thread the raw signature' rule. Confirmed: the code the design points at as the template puts the signature (and, for sign-in, the bearer) in the URL.

**Fix:** Never place the signature (or a bearer) in a loopback redirect URL. Have safe-sign.html deliver the signature to the loopback via an HTTP POST body (extend the loopback server to read the request body instead of only url.searchParams), OR have main mint a one-time, single-use, short-TTL handoff token that the page uses to POST the signature directly to the facade so it never transits the desktop browser's URL bar. Add a desktop test asserting no signature/bearer ever appears in a redirect URL or navigation history.

### [medium] Provider-token / API-key leak via upstream-error passthrough in the model-visible propose/simulate result. The design's simulation.py and safe_txservice_client.py return simulation asset-diffs and a revertReason to the agent as a tool result, and §8 says 'on simulate failure/uncertain, show a warning.' RPC and Tenderly-style providers routinely embed the secret API key in the request URL (e.g. https://eth-mainnet.g.alchemy.com/v2/<KEY>); an unsanitized 401/5xx body or exception string surfaced verbatim as the tool result leaks that provider token into model output and persisted run state. The design does not require sanitizing upstream errors, contradicting the ai-backend rule 'convert broad exceptions into typed domain errors with safe public messages — never leak internal detail to model output' (services/ai-backend/CLAUDE.md).

**Fix:** Mandate in the Safe surface (simulation.py / safe_txservice_client.py) that every upstream RPC/tx-service/Tenderly error is caught and mapped to a fixed typed domain error with a safe public message BEFORE it becomes a tool result — never pass through raw URLs, response bodies, or exception strings. Keep the provider key only server-side (env/secret), never in a returned field. Add a test that a simulated upstream 401 whose URL contains a fake key yields a tool result with the key absent.

### [medium] The server accepts a signature-shaped approval 'answer', so a client bug or a compromised renderer would leak the raw signature into persisted graph state and model context. ApprovalDecisionRequest.answer is an unbounded str|None (services/ai-backend/src/runtime_api/schemas/approvals.py:65) whose normalizer only strips whitespace (agent_runtime/validation.py:57-61), and record_approval_decision threads answer straight into RuntimeApprovalResolvedCommand -> worker resume -> LangGraph checkpoint state and the request_wallet_signature tool result (agent_runtime/api/approval_coordinator.py:382-391). A 132-char hex signature passes unimpeded. The design's whole no-leak guarantee for the SIGN gate rests solely on client discipline in RunDestination.resolveApproval; there is no server-side guard, and the value bypasses the redaction that only covers the audit row/resolved event.

**Fix:** Add a server-side guard for approval_kind=='wallet_signature': reject (422) any decision whose answer/reason/edited_payload matches a signature/hex-blob shape (e.g. ^0x[0-9a-fA-F]{100,}$) or exceeds a small status-string length cap, and constrain the accepted answer to a fixed status enum ('signed','rejected','hash_mismatch','chain_not_allowed','not_owner'). Enforce this in ApprovalDecisionRequest validation, not just in the client. Add the coordinator test the design lists, but asserting a signature-shaped answer is refused, not merely that the happy-path status round-trips.

### [medium] The bearer-vs-signature exposure tradeoff on desktop is unspecified, so the design leaves the door open to leaking a session secret instead. §7 step 9 says the port submits the signature to the facade, but on desktop the authenticated facade call requires the session bearer, which lives only in the main process's SecretStorage — safe-sign.html in the system browser has no bearer. The two unstated resolutions each leak a secret: (a) relay the bearer into the browser so the page can POST directly (new session-secret exposure), or (b) relay the signature back through the loopback URL (finding 1). The design picks neither explicitly and its file-list wording implies (b).

**Fix:** State explicitly that the bearer never leaves main and is never relayed to safe-sign.html, and that the signature returns to main via a POST body (or one-time scoped token) rather than a URL. Document the chosen handoff shape in the PRD and pin it with a test that asserts neither the bearer nor the signature appears in any browser-visible URL and that main is the only holder of the session bearer for the /transactions submit.

### [low] The propose/simulation asset-diff and safeTxHash the design pushes into the consent card and tool result are agent-influenced text unless the card strictly renders the renderer's own re-derivation. AC-4 requires the card to show the renderer's independent simulation 'not agent-authored text,' but ApprovalParam values are capped only at 128 chars with no structural typing for To/Amount/Token/Chain/Nonce (approvals.py:274-307). A malicious model could still shape the human-readable summary/params to socially-engineer the human into approving, indirectly aiding exfiltration of value even though no key leaks. This is a phishing/consent-integrity risk adjacent to the secret-exposure lens, not a direct secret leak.

**Fix:** Require the card's params and simulation block to be built exclusively from the renderer's re-derived canonical fields (facade /simulate response), never from the agent-supplied summary/params; treat request_wallet_signature.summary as advisory-only and visibly distinguish it from the renderer-computed destination/amount. Add a chat-surface test that the rendered To/Amount/Chain/Nonce come from the re-derivation, and that a mismatch between agent-claimed and re-derived safeTxHash blocks signing (hash_mismatch) before the wallet popup.
