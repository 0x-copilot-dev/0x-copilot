# P6 eng-plan adversarial review

**Verdict:** needs-changes

### [high] Control C (bound-Safe) enforcement checks a store of UNDEFINED provenance. The plan repeatedly says bindings are "captured at CONNECT" (§1.4, §5 PR-W2/W0), but the Safe connector is seeded as auth_mode=McpAuthMode.NONE (mcp_catalog.py:32/39/83; verified no OAuth capture step). A NONE-auth connector has no CONNECT-time flow that captures a user's Safe address, and grep confirms NO existing safe_binding/bound_safe/treasury store or capture anywhere in services/backend, ai-backend, or backend-facade. So the facade check `safe ∈ bound_safes(user)` (§1.4.2) enforces against a table nothing is specified to populate: it is either always-empty (fail-closed but the feature never works) or populated by an unspecified path. If that path ever ingests the agent-supplied safe_address (e.g. seeded at first propose), Control C becomes circular — the agent supplies the binding it is later checked against — which is exactly review-1[med]/review-3[high AC#5]. "Server-derived identity" (the user) is not the same as server-verified Safe ownership. The invariant 'bound-Safe enforced server-side' is therefore not provably held.

**Fix:** Specify a concrete, human-driven, ownership-PROVEN binding capture that is independent of agent input: e.g. an explicit Settings "add treasury Safe" action gated by a SIWE-style Safe-owner signature (prove the connecting wallet is an owner of the Safe on the target chain via the tx-service owners read), writing (org_id,user_id)->{safe_address,chain_id,added_at} only on verified proof. Fail closed: empty store => every /simulate and /transactions 403s. Add a CI/unit test that an agent-supplied safe_address can never seed or mutate the binding store, and a tenant-isolation test that another user's binding is invisible. Do not describe capture as a byproduct of the NONE-auth CONNECT.

### [medium] M2 metadata closure is under-enforced and the plan's own instructions conflict. create_approval_request persists the approval record metadata as `metadata = dict(payload)` (stream_events.py:394), and the undo gate + card read those raw keys (`approval.metadata.get("reversible") != "yes"` at approval_coordinator.py:478 and 520). Row 5 tells the builder to model `_native_wallet_signature_payload` on `_native_ask_a_question_payload` (stream_events.py:730-764), which builds its payload with `{**payload, ...}` (L750) — i.e. it SPREADS the interrupt payload. That directly conflicts with row 5's other clause "never spread agent fields" and with row 8's WalletSignatureApprovalMetadata(extra="forbid"). Because the persisted metadata is dict(payload) and the pydantic model is never instantiated on the persistence path, an extra="forbid" schema is DECORATIVE: a build that follows "mirror the ask payload" will `{**payload}` and let an agent-injected reversible:"yes"/category/vendor ride into approval.metadata, opening an undo window (M2) exactly as in the reviews.

**Fix:** State explicitly that `_native_wallet_signature_payload` must NOT use `{**payload}`. Construct the payload from a fixed server-side allowlist only (approval_id, action_id, batch_id, batch_index, native_interrupt_id, approval_kind='wallet_signature', server-computed `safe` block, `signable`, and a de-emphasised `agent_claims` sub-object) with reversible=no/category=ACTION/vendor=SAFE pinned as constants — stripping any agent-supplied reversible/category/vendor from the interrupt value before projection. Keep the extra="forbid" model but validate/re-serialize the payload THROUGH it at projection time so the closed shape is what reaches dict(payload). Test the persisted approval.metadata (not just the schema) has pinned reversible=no regardless of agent input, and that request_approval_undo 422s for a wallet_signature.

### [medium] The wallet_signature decision surface is incompletely enumerated: `_decide_suggest_edit` (approval_coordinator.py:606) is gated ONLY by status==PENDING (L627) — it has no approval_kind gate, unlike FORWARDED (blocked by APPROVAL_FORWARDABLE_KINDS in _guard_forwardable L1150) and APPROVE_WITH_EDITS (blocked by _EDIT_CAPABLE_COMMIT_KINDS={'draft_send'} L782/L820). So a client/compromised-renderer can POST {decision:'suggest_edit', edited_payload:{...}} on a PENDING wallet_signature; it records the decision, threads request.reason (L657), and creates a child approval carrying edited_payload in metadata (L658/L668) — an agent-visible state channel. Separately, the row-4 answer-guard anchor is imprecise: it says insert "after scope check L316, before record write L338", but SUGGEST_EDIT/APPROVE_WITH_EDITS/FORWARDED all early-return at L317-331 BEFORE L338, so a guard placed literally just above L338 never runs for those branches, leaving reason/edited_payload/edits (a hex signature could nest inside edited_payload as JSON) unfiltered on the SUGGEST_EDIT path. Fund-redirection is neutralised by Control A (proposal_ref authority ignores edited_payload), but the M4 no-leak invariant depends on exact placement.

**Fix:** For approval_kind=='wallet_signature', reject decision not in {approved, rejected} with 422 at the TOP of record_approval_decision (immediately after the scope check at L316, BEFORE the L317 branch dispatch) so SUGGEST_EDIT/APPROVE_WITH_EDITS/FORWARDED are all closed for signing approvals. Place the hex-blob/status-enum answer-guard at the same pre-dispatch point (not "before L338") and make the hex-blob scan RECURSE into edited_payload/edits JSON, not just top-level strings. Add a test that suggest_edit/forward/approve_with_edits on a wallet_signature all 422 and that a nested hex blob in edited_payload is rejected.

### [medium] The desktop canonical-doc fetch credential path is internally inconsistent (§1.6 / row 20). The plan says safe-sign.html "accepts only an opaque proposal_ref + loopback state in the URL" AND that it "FETCHes the canonical EIP-712 doc from the facade (GET /v1/wallet/safe/proposal/{ref}) authenticated by a one-time scoped token minted by main." But safe-sign.html runs out-of-process in the system browser; the only main->page channel is the URL main opens. For the page to send a main-minted token, that token must be in the URL — contradicting "only proposal_ref + state" and re-introducing a (scoped, weaker) credential into browser history, the very class of leak H5 fixes. The H5 win (signature + session bearer never in the URL) is real and correctly designed, but this sub-path is unresolved.

**Fix:** Resolve the mechanism explicitly. Preferred: make proposal_ref ITSELF the read-only, unguessable, single-use, short-TTL capability the facade accepts for GET /proposal/{ref} (no separate token), and state that leaking proposal_ref via history exposes only the already-human-reviewed canonical doc — never the signature or bearer, and grants no submit. Drop the "one-time scoped token minted by main" wording, or if kept, acknowledge it lives in the URL and bound its blast radius (read-only, single-use, seconds-TTL). Keep the test: no signature/bearer in any URL/history; add: proposal_ref confers only canonical-doc read, not submit.

### [low] The "durable" classifier fix (PR-S2) remains an incomplete denylist and is correctly display-only, but the plan should not imply completeness. It adds append/clear/insert/set/remove to _connector_action_is_read_only (stream_events.py:910-918), yet copy/move/patch/put/share/add/rename/import/revoke/grant stay unclassified — and gdrive's existing `copy_file` (product_scope: write, desktop_profiles.yaml) is ALREADY misclassified read_only today. This is display/risk-label only (the interrupt fires for every call_mcp_tool regardless of read_only; the real per_call gate is the profile's approval:per_call + graph HITL, which the plan states honestly), so it is not an enforcement bypass — but the heuristic will always be incomplete.

**Fix:** Keep the plan's honest framing that this classifier is a consent-CLARITY label, never the write gate, and that enforcement rides the profile approval:per_call. Optionally invert to an allowlist (read/get/list/search/fetch/view => read_only, else write) so unknown verbs fail safe to 'write/medium' rather than 'read/low'. If left as a denylist, add copy/move/patch/put/share and note gdrive copy_file is fixed in passing. Do not gate any interrupt decision on read_only.
