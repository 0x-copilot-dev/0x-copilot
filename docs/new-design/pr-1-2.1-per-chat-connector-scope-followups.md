## 1 · PRD

### Problem

PR 1.2 shipped per-chat connector scope override but cleanly missed two real gaps documented in the post-merge audit:

1. **Workspace-admin override path.** The PRD §2.5 promised: "Conversation owner ✅ · Workspace admin ✅ ('support paused this for a member after a security incident') · Other workspace member ❌ 403." The shipped route uses `_conversation_for_scope(org_id, user_id, conversation_id)` which scopes by `user_id`, so an admin attempting to pause Slack on a member's chat gets 404. The "support paused this for a member after a security incident" workflow is therefore unreachable.

2. **Multi-tab `visibilitychange` refetch.** PRD §3.6 promised: "FE refetches on `visibilitychange` if `updated_at` skew is detected (cheap, no protocol change)." Not implemented in [`useConversationConnectors`](../../apps/frontend/src/features/connectors/useConversationConnectors.ts). Last-write-wins still applies on the server, but a stale tab will hold a stale view of the popover indefinitely until the user refreshes.

### Goals

1. A workspace admin can PATCH any conversation in the same org, with the override visible in the audit row.
2. The refetch is silent: no flicker, no extra round-trip when the tab was visible the whole time, and no race against an in-flight PATCH.
3. Both gaps land without changing the wire contract or the SSE handshake — the agent harness stays untouched (snapshot semantics still rule).
4. No new dependency. Reuse the existing scope catalog and the existing `getConversation` API.

### Non-goals

- A new "admin runtime" permission scope. Reusing `admin:users` (the existing workspace-user-admin scope) covers "admin acting on a member's behalf" without coordinating a fresh entry in the auth catalog. A dedicated `admin:runtime` scope can land later if the surface grows beyond connector overrides.
- Server-pushed scope-change notifications (e.g. WebSocket / SSE for non-run state). Out of scope; multi-tab refetch via `visibilitychange` is sufficient for the design's "calm, silent" UX.
- Optimistic-concurrency tokens (ETag / If-Match). Last-write-wins remains correct: the user is sequential per chat; the audit row carries the diff.

### Success criteria

- ✅ A user holding `admin:users` PATCHes a conversation owned by another member; response is 200; audit row has `override_by_admin=true` and the actor's user_id; conversation owner sees the change on their next reload (or on the next `visibilitychange`).
- ✅ A user holding NO admin scope PATCHing another member's conversation gets **404** (same opacity as today — never reveals existence).
- ✅ A FE tab hidden for >5s, then re-shown, fetches the conversation once and silently reconciles `enabled_connectors` if the server's `connectors_updated_at` is newer than what the tab last saw.
- ✅ The visibility-driven refetch is gated by an in-flight PATCH guard so it never clobbers an optimistic flip.
- ✅ Both branches covered by unit tests.

### User stories

| As…                      | I want…                                                                     | So that…                                                 |
| ------------------------ | --------------------------------------------------------------------------- | -------------------------------------------------------- |
| Marcus (workspace admin) | to pause Slack on a member's chat after a security incident                 | I can isolate a connector without waiting on the member  |
| Sarah (end user)         | the popover I had open in another tab to update silently when I switch back | I don't act on stale state                               |
| Compliance               | every admin override to be flagged distinctly in the audit log              | I can answer "did the admin act on user data, and when?" |

## 2 · Spec

### 2.1 Admin override — wire change

**No wire change.** Same `PATCH /v1/agent/conversations/{id}/connectors` route, same body shape, same response. Authorization is widened on the server side; the response shape is identical, including the `connectors_updated_at` field that drives the FE refetch.

The audit row gains one boolean inside `metadata_json_redacted`:

```jsonc
{
  "action": "conversation.connectors.update",
  "actor_user_id": "<admin user>",
  "metadata_json_redacted": {
    "before": { "slack": ["read"] },
    "after": { "slack": null },
    "diff_keys": ["slack"],
    "override_by_admin": true, // ← new; absent / false for owner self-PATCH
    "conversation_owner_user_id": "<member>", // ← present only on admin overrides
  },
}
```

### 2.2 Admin override — authorization

The route already loads `RuntimeServiceAuthenticator.trusted_identity_from_request(request)` via `scoped_identity`. Reuse `permission_scopes` to detect the admin path:

```python
# pseudo, in routes.py
identity = RuntimeServiceAuthenticator.trusted_identity_from_request(request)
is_admin = ADMIN_USERS in identity.permission_scopes
return await cls.service(request).update_conversation_connectors(
    org_id=org_id,
    user_id=identity.user_id,
    conversation_id=conversation_id,
    request=payload,
    allow_admin_override=is_admin,
)
```

The service replaces the strict owner lookup with a new helper:

```python
# Loads the conversation by org only. Returns None unless the caller is
# either the owner OR holds an admin scope (caller declares which via
# allow_admin_override). Same 404 semantics either way — never leaks
# existence to non-owners-non-admins.
async def _conversation_for_owner_or_admin(
    self,
    *,
    org_id: str,
    actor_user_id: str,
    conversation_id: str,
    allow_admin_override: bool,
) -> tuple[ConversationRecord, bool]:  # (record, is_admin_override)
```

`update_conversation_connectors` calls it instead of `_conversation_for_scope`, and threads the boolean into the audit metadata.

The persistence layer is unchanged — `get_conversation` still scopes by `(org_id, user_id, conversation_id)` for the existing single-user routes (read, list, context, run-create); a new `get_conversation_for_org` reads by `(org_id, conversation_id)`. The new method is consumed only by `_conversation_for_owner_or_admin`.

### 2.3 Visibility refetch — wire change

**No wire change.** The existing `GET /v1/agent/conversations/{id}` already returns `enabled_connectors` and `connectors_updated_at` (PR 1.2). The FE just calls it again at the right moments.

### 2.4 Visibility refetch — hook contract

`useConversationConnectors(conversation, identity)` keeps its existing public surface — the new behavior is internal:

- Mount a `document.visibilitychange` listener. On every transition to `document.visibilityState === "visible"`:
  1. If `conversation === null` or `identity === null` → no-op.
  2. If `loading === true` (an in-flight PATCH is racing) → no-op; the PATCH response will overwrite anyway.
  3. Else fetch `getConversation(conversation.conversation_id, identity)`.
  4. If the response's `connectors_updated_at` is **strictly newer** than the local `connectors_updated_at`, replace `scopes` with `response.enabled_connectors ?? {}`.
  5. Swallow errors silently into `error` state — the user already sees "connectivity lost" via other affordances.

The listener is removed on unmount. Switching conversations re-mounts the listener with the new conversation id (already covered by the existing dependency array).

### 2.5 Permissions matrix (final)

| Caller                                                | Outcome | Audit `override_by_admin`                        |
| ----------------------------------------------------- | ------- | ------------------------------------------------ |
| Conversation owner                                    | 200     | `false` (omitted)                                |
| Workspace admin (`admin:users`), not the owner        | 200     | `true`                                           |
| Workspace member without `admin:users`, not the owner | 404     | (no row written)                                 |
| Wrong-org caller                                      | 404     | (no row written)                                 |
| Caller without `runtime:use`                          | 403     | (router-level dependency, before this code runs) |

### 2.6 Error semantics

Unchanged from PR 1.2. The admin path adds **no** new error codes — same 404 for unauthorized callers (opacity preserved), same 422 / 400 for malformed bodies.

## 3 · Architecture

### 3.1 Where this lives

```
   FE  ──── PATCH .../connectors ──────────────────────► facade ──► ai-backend
                                                                       │
                                                                       ▼
                                                       _conversation_for_owner_or_admin()
                                                                       │
                                          ┌────────────────────────────┴────────────┐
                                          ▼                                         ▼
                                ConversationStore.get_conversation_for_org()  audit.write_audit_log(
                                (new method, org-scoped read)                    action="conversation.connectors.update",
                                                                                  metadata={..., override_by_admin: true })

   FE  ──── GET .../{id}    ◄───── (no change to GET path)
       on document.visibilitychange → "visible"
            └── if updated_at strictly newer → replace local scopes silently
```

### 3.2 Why reuse `admin:users` instead of a new scope

Adding a new permission scope is a coordinated change: roles catalog seed (`services/backend/migrations/...`), per-tenant role assignment, and an entry in [`docs/roadmap/29-a10-rbac-enforcement`](../roadmap/29-a10-rbac-enforcement.md). The semantic fit for `admin:users` ("manage users, sessions, lockouts, role assignments") covers "admin acts on a member's data" naturally — pausing a connector on a member's chat is a user-impacting admin action. The audit row carries the explicit `override_by_admin` flag for SIEM differentiation. If a dedicated `admin:runtime` scope becomes warranted later (e.g. for forced cancellations, conversation deletion, etc.), it lands as a single addition without re-keying this PR.

### 3.3 Why `_conversation_for_owner_or_admin` returns `(record, is_admin_override)`

The boolean is computed inside the helper because that's where ownership is checked. Bubbling it up via the return tuple lets the caller — `update_conversation_connectors` — stamp the audit metadata without re-querying ownership or re-scoping the user check. Single source of truth, zero re-evaluation.

### 3.4 Why `visibilitychange` and not polling / SSE

Three reasons:

1. **No wire change.** Polling adds bandwidth. Server-push for non-run state would force a new connection model (the existing SSE is per-run, frozen at `runtime_context_json`).
2. **Mirrors the user's mental model.** A hidden tab is "frozen"; it makes sense for it to reconcile when it becomes active. This matches how Gmail/Slack behave for similar metadata.
3. **Cheap.** The `getConversation` round-trip is already part of the chat-load path; reusing it costs one query at the moment a tab regains focus. Rate-limiting the listener (debounce or "in-flight guard") is sufficient to handle pathological tab-flip scenarios.

### 3.5 Why the in-flight guard

The optimistic update in `patch()` calls `setScopes` synchronously. If a tab refocuses _during_ an in-flight PATCH, the GET response could land before the PATCH response and overwrite the optimistic flip. The guard (`loading === true → no-op`) prevents that race entirely. The PATCH response is the canonical write; the GET is a reconciliation that defers when a write is pending.

### 3.6 DRY — what we reuse vs. what we add

| Concern                     | Reuse                                                    | Add                                                                 |
| --------------------------- | -------------------------------------------------------- | ------------------------------------------------------------------- |
| Identity / RBAC             | `RuntimeServiceAuthenticator`, `permission_scopes`       | —                                                                   |
| Admin scope                 | `ADMIN_USERS` from `enterprise_service_contracts.scopes` | —                                                                   |
| Audit chain                 | `runtime_audit_log` writer                               | one boolean key in metadata                                         |
| Conversation read for admin | `_conversation_record` decoder                           | one new method on the persistence port (`get_conversation_for_org`) |
| Refetch endpoint            | existing `GET /v1/agent/conversations/{id}`              | —                                                                   |
| FE visibility listener      | `document.visibilitychange` (browser API)                | small `useEffect` inside the existing hook                          |
| FE refetch call             | existing `getConversation` from `agentApi.ts`            | —                                                                   |

**Net new code:**

- 1 new persistence port method (`get_conversation_for_org`) + adapter implementations (~20 LOC across in-memory + postgres + async wrapper)
- 1 service helper (`_conversation_for_owner_or_admin`, ~25 LOC)
- 1 service-method signature parameter (`allow_admin_override: bool`)
- 1 route signature parameter
- ~20 LOC `useEffect` block in `useConversationConnectors`
- 2 ai-backend tests (admin override happy + non-admin denied)
- 2 FE hook tests (refetch on visible + no-op when in-flight)

Target: **~150 net LOC**, ~60 of which is tests.

### 3.7 Sequence — admin overrides Slack on a member's chat

```
Admin Marcus           FE                   facade               ai-backend           Postgres
     │                  │                     │                     │                     │
     │  open chat owned by Sarah (no read access today; overlay handles)
     │                  │  PATCH .../connectors {scopes:{slack:null}}
     │                  │ ─────────────────► │ ─────────────────► │  permission_scopes ⊇ {admin:users}
     │                  │                     │                     │  is_admin = True
     │                  │                     │                     │  _conversation_for_owner_or_admin
     │                  │                     │                     │   ↪ org-scoped lookup
     │                  │                     │                     │   ↪ owner mismatch but admin → allow
     │                  │                     │                     │  store.update_conversation_connectors(...)
     │                  │                     │                     │ ───────────────────► UPDATE
     │                  │                     │                     │  audit.write({
     │                  │                     │                     │    override_by_admin: True,
     │                  │                     │                     │    conversation_owner_user_id: <Sarah>
     │                  │                     │                     │  })
     │                  │                     │ ◄──────────────────│  200 + new scope map
     │                  │ ◄────────────────── │
     │                  │  popover updates                                                  │
                                                                                            │
     [Sarah's tab is hidden in another window]
     [Sarah un-minimizes the chat — tab becomes visible]
                                                  Sarah's FE
                                                       │  document.visibilitychange → "visible"
                                                       │  GET .../conversations/{id}
                                                       │  response.connectors_updated_at > local
                                                       │  setScopes(response.enabled_connectors)
                                                       │  popover silently reflects the admin pause
```

### 3.8 Edge cases

| Case                                                   | Behaviour                                                                                                                                       |
| ------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| Admin patches their own conversation                   | `override_by_admin=false` (owner-self path); helper returns `(record, False)`.                                                                  |
| Admin scope present but caller IS the owner            | Owner path takes precedence; no override flag.                                                                                                  |
| Conversation deleted between scope check and UPDATE    | UPDATE returns `None`; service raises 404 (existing behaviour).                                                                                 |
| Tab refocus while a PATCH is in flight                 | `loading === true` → refetch skipped; PATCH response replaces state.                                                                            |
| Tab refocus and `connectors_updated_at` equal to local | `setScopes` skipped; no React re-render.                                                                                                        |
| Tab refocus with no conversation loaded                | No-op; listener early-returns.                                                                                                                  |
| Hook unmount while listener is pending                 | `removeEventListener` in the cleanup function; no leak.                                                                                         |
| Multiple rapid focus toggles                           | Each call awaits the previous; `loading` guard prevents stacking. Browser already coalesces visibilitychange to one event per state transition. |

### 3.9 Test plan

**ai-backend** (`tests/unit/runtime_api/test_conversation_connector_scope_route.py` — extend existing):

- `test_admin_override_succeeds_for_other_org_user` — caller has `admin:users`, body 200, audit row has `override_by_admin=true`.
- `test_non_admin_other_user_404s` — caller lacks `admin:users`, target conversation not theirs → 404 (no row written).
- `test_admin_owner_path_does_not_flag_override` — admin patching their own chat → `override_by_admin=false`.

**Frontend** (`apps/frontend/src/features/connectors/useConversationConnectors.test.tsx` — extend existing):

- `refetches on visibility transition and reconciles when server is newer`
- `is a no-op while a PATCH is in flight`
- `is a no-op when the server's updated_at matches the local one`

### 3.10 Rollout

- Flag-free. The admin path uses an existing scope; if no role in the org carries `admin:users`, behaviour is unchanged.
- No migration. Audit metadata is JSONB — adding a key is forward-compatible.
- Backout: revert the route + service helper. Audit rows already written keep the new key (harmless extra metadata).
